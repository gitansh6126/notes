K-means Clustering — Lloyd's Algorithm

The exact "best possible" way to group data into K clusters is computationally out of reach. Lloyd's algorithm is the fast, iterative compromise almost everyone actually uses instead.

Problem · Partition n points into K clusters (NP-hard, exactly) Solution · Lloyd's algorithm — an iterative heuristic Guarantee · Converges, but not necessarily to the global optimum
1

Overview: A Heuristic for an Impossible Search

THE SETUP

K-means is NP-hard. Lloyd's algorithm is how we cope.

The K-means problem asks for the partition of n data points into K clusters that minimizes the total squared distance from each point to its cluster's center. Checking every possible way to partition n points into K groups is combinatorially explosive — the problem is NP-hard, so no algorithm can guarantee the exact best answer in reasonable time as data grows.

Lloyd's algorithm — what almost everyone means when they say "K-means" — doesn't try to solve that problem exactly. It's a fast, iterative heuristic that reliably lands on a good, if not provably optimal, partition.

Running analogy: the pizza chain problem

Imagine you're opening K pizza stores in a city and want every customer's trip to their nearest store to be as short as possible, on average. Checking every conceivable way to divide the city among K stores would take longer than the universe has existed. Lloyd's algorithm is what a sensible operations team actually does instead: guess some starting locations, then repeatedly adjust — and stop once nothing improves. We'll follow this pizza chain through the rest of the lecture.

The Two-Step Loop

MECHANICS

Compute means, reassign points, repeat

After an initial (often random) assignment of every point to one of K clusters, Lloyd's algorithm alternates between two simple steps until nothing changes:

One full iteration of Lloyd's algorithm
Start: every point already has some cluster label z
📍

Compute Means

Set each centroid μk to the average of the points currently in cluster k

→
🔁

Reassign Points

Move each point to whichever centroid it's now closest to

↻ repeat both steps until no point's assignment changes
μk  =  mean{ xi : zi = k }       zi  ←  argmink ‖xi − μk‖ left: recompute each store's location · right: send each customer to their nearest store
Running analogy: the pizza chain problem

Compute Means is moving day: each store relocates to sit at the geographic center of the customers it currently serves. Reassign Points is the customers reacting: anyone now closer to a different store switches allegiance. A store that used to be closest but just moved away can lose customers; one that moved closer can gain them. The cycle repeats until nobody wants to switch anymore. 

1# X: (n, d) data points   z: (n,) cluster labels, init randomly
2
3while not converged:
4    # --- Compute Means step ---
5    centroids = [X[z == k].mean(axis=0) for k in range(K)]
6
7    # --- Reassign step ---
8    dists = cdist(X, centroids)              # (n, K) distance table
9    new_z = dists.argmin(axis=1)          # nearest centroid per point
10
11    converged = np.all(new_z == z)
12    z = new_z
  • Each centroid is nothing but an average — line 5 is literally "Compute Means."
  • Line 9 is "Reassign Points": every point independently picks its closest current centroid.
  • Convergence is checked by comparing old and new labels directly — no distance threshold needed.

Question 1 — Convergence

DOES IT STOP?

Yes — every point eventually settles

The algorithm has converged when a full reassignment step changes nobody's label — equivalently, every point is already strictly closer to its own cluster's mean than to any other. Lloyd's algorithm is guaranteed to converge: each iteration can only decrease (or leave unchanged) the total squared distance from points to their centroids, and that quantity can't drop forever.

Customers switching stores, iteration by iteration
iter 1iter 2iter 3iter 4iter 5iter 6

Fewer customers switch stores each round. Convergence is the round where that bar finally hits zero — nobody has any reason left to move.

Key insight

Convergence is guaranteed, but the destination is not: Lloyd's algorithm reliably comes to a stop, but because K-means itself is NP-hard, that stopping point is only a local optimum — not necessarily the single best partition that exists.

Running analogy: the pizza chain problem

The chain stops relocating stores once every customer is already buying from their closest option. That's a stable, "good enough" arrangement — but it says nothing about whether some completely different set of store locations, far from where this run started, might have served the city even better.

Question 2 — What Shape Are the Clusters?

CLUSTER GEOMETRY

Straight-line boundaries, every time

Because every point simply joins whichever centroid is nearest, the boundary between any two clusters is always exactly the straight line of points equidistant from both centroids. Repeated across all K centroids, this carves the whole space into a Voronoi diagram — flat-sided, convex regions, regardless of how the data itself is actually shaped.

Three stores' delivery territories
Store A territory Store B territory Store C territory

Each dot is a customer, colored by whichever store (ringed marker) is nearest. The boundaries between colors are perfectly straight — that's not a simplification, it's mathematically guaranteed by "nearest centroid wins."

Worth noting

This is also K-means' key limitation: real clusters that are long, curved, or nested inside one another can't be captured by straight-sided territories, no matter how many iterations you run. K-means assumes roughly round, similarly-sized blobs.

Running analogy: the pizza chain problem

This is just a school-district map for pizza: each neighborhood is assigned to whichever store is closest as the crow flies, so the district lines are always straight — even if the "natural" neighborhoods in the city are winding streets or an oddly shaped river district that no straight line could ever fairly represent.

Question 3 — Initialization

WHERE YOU START MATTERS

A bad first guess can get permanently stuck

Because Lloyd's algorithm only ever improves on its starting point, a poor initial placement of centroids can converge to a distinctly worse local optimum — sometimes with an entire natural cluster split awkwardly between two centroids, or two real clusters merged under one.

Same data, two different starting guesses
⚠ Bad initialization

All three stores dropped into the same corner — one whole district is left unclaimed at the far end.

✓ Good initialization

Stores spread across the city from the start — each converges to its own natural district.

Watch out

Plain random initialization can easily place two or more centroids close together by chance. In practice, most implementations use K-means++: pick the first centroid randomly, then pick each next one with probability proportional to its squared distance from the centroids already chosen — spreading them out on purpose. Running the whole algorithm several times from different random starts and keeping the best result is a common complementary safeguard.

Running analogy: the pizza chain problem

Dropping all three new stores in the same downtown block means they spend the whole simulation squabbling over the same few blocks while the suburbs go completely unserved — no amount of later shuffling fixes that. Spreading the initial guesses across the city gives each store its own territory to grow into from day one.

Question 4 — Choosing K

HOW MANY CLUSTERS?

Look for the point of diminishing returns

K-means is unsupervised — there's no ground truth to check the cluster count against. A common approach is to run the algorithm for several values of K and plot the total within-cluster squared distance (often called inertia) against K.

The elbow method

Inertia always falls as K grows — with enough stores, everyone lives next door to one. The "elbow," where the curve stops dropping sharply and starts to flatten, marks the point past which adding another cluster buys little.

Running analogy: the pizza chain problem

One store serving the whole city means long average trips for everyone. Fifty stores make every trip trivially short — but at the cost of a store on every block, which is absurd overkill. The elbow is where opening one more store stops meaningfully shortening anyone's commute — the sweet spot between "underserved" and "wasteful."

Key Takeaways

1

The K-means problem is NP-hard; Lloyd's algorithm is the practical heuristic — alternating "compute means" and "reassign points" until nothing changes.

2

Convergence is guaranteed (total squared distance never increases), but the result is only a local optimum, not necessarily the global best.

3

Because points join their nearest centroid, clusters always come out as convex, straight-boundaried (Voronoi) regions — a real limitation for oddly-shaped data.

4

Initialization matters a lot (K-means++ helps); choosing K usually comes down to the elbow method, silhouette scores, or domain knowledge.
