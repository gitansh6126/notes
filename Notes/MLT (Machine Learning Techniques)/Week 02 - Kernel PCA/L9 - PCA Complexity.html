PCA's hidden cost:
when features outnumber data

PCA's textbook recipe quietly assumes you can afford to decompose a d×d matrix. The moment your features (d) vastly outnumber your data points (n) — genomics, images, text — that assumption becomes the whole bottleneck.

§1 The bottleneck §2 The key trick §3 The algorithm §4 Feel the speedup §5 Toward kernels
01

The bottleneck: a matrix too big to decompose

PCA finds principal components by eigen-decomposing the covariance matrix of the data. That matrix's size is dictated entirely by the number of features — not the number of data points.

Real-life analogy

Imagine describing 5 people using 20,000 medical measurements each (blood markers, genetic variants, scan readings). You'd build a giant 20,000 × 20,000 "how do these measurements relate to each other" matrix — but there are only 5 people's worth of actual information in the room. Most of that giant matrix is structurally redundant before you even start solving it.

the covariance matrix
C = (1/n) X XT    —   X is d × n, so C is d × d
C — d × d
eigen-decomposition: O(d³)
K — n × n
the size we'd rather work with
!
Why this hurts

Eigenvalue decomposition of a d×d matrix costs O(d³). With d in the tens of thousands — genes, pixels, vocabulary words — this becomes computationally impossible long before it becomes merely slow.

02

The key trick: recycle the data itself

The escape route relies on one elegant fact: every principal direction we're searching for is secretly already buildable from the data points we have.

Real-life analogy

Think of mixing paint. You don't need to invent a brand-new pigment to get a new principal direction — you just need to blend the paint samples (data points) you already own, in different proportions. With only n samples, there are only n independent ways to blend, no matter how many color channels (features) each sample has.

α₁ · x₁ + α₂ · x₂ + … + α_n · x_n = w_k

Every eigenvector w_k of the covariance matrix is just a weighted combination of the n data points — written compactly as w_k = Xα_k, with α_k ∈ ℝⁿ.

substituting into the eigenvalue equation
C w_k = λ_k w_k   →   K α_k = (n λ_k) α_k
where K = XTX  is only n × n
✓
The payoff

We've traded a d × d eigen-problem for an equivalent n × n one. When d ≫ n, that's the difference between "impossible" and "instant."

03

The algorithm

Four steps take you from raw data to the same principal components PCA would have found the expensive way.

1

Compute K

Build the n × n matrix K = XᵀX instead of the d × d covariance matrix.

2

Eigen-decompose K

Find K's eigenvectors β_k and eigenvalues λ̃_k — an O(n³) operation.

3

Normalize

K's eigenvectors aren't quite the α_k we need — rescale them.

α_k = β_k / √(n λ_k)
4

Reconstruct

Map back to the original feature space to recover the actual principal components.

w_k = X α_k
i
Why this is valid

A standard linear-algebra fact underwrites the whole trick: XXᵀ and XᵀX share the same non-zero eigenvalues. We never lose any real structure — we just choose to compute it in whichever matrix is smaller.

05

One more observation — the door to Kernel PCA

Look closely at what K = XᵀX actually contains: entry (i, j) is xᵢᵀxⱼ — the dot product between data points i and j.

Real-life analogy

It's like judging how similar two songs are just by how they make you feel side by side, rather than comparing every individual audio frequency in each track. K is nothing but a table of pairwise similarities between data points — PCA never actually needs the raw high-dimensional features themselves, only how each point relates to every other point.

why_this_matters.txt
// K only ever needs pairwise dot products:
K[i][j] = x_i . x_j

// swap the dot product for any valid "similarity" function,
// and the whole algorithm still works — even for data where
// no straight-line relationship exists at all
K[i][j] = kernel(x_i, x_j)
→
This is the seed of Kernel PCA

Once PCA is rewritten purely in terms of pairwise similarities, replacing the dot product with a kernel function lets PCA uncover non-linear structure — without ever explicitly building the giant feature vectors.
