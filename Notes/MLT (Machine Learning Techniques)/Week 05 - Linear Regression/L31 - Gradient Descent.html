0The Closed Form Has a Scaling Problem

An exact answer that becomes unusable

Lectures 29–30 gave us an exact, closed-form answer: W* = (XXᵀ)⁻¹XY. That's the correct solution — but "correct" and "computable" are different things once the problem gets big.

BottleneckWhy it breaks down
Matrix inversion, ~O(D³)D = number of features. Millions of features (common in text, genomics, ad-tech data) make cubing D computationally impossible
Forming XXᵀ itselfN = number of data points. Even before inverting anything, just building this matrix requires touching every one of N (possibly billions) data points
The reframe this lecture makes Give up on landing exactly at w* in one shot. Instead, accept an iterative process that gets closer and closer to w* using only cheap, local information — the gradient — at each step.
RMS / Cubedekho lens If Cubedekho ever tracked customer behavior across millions of product-page interactions (D huge) or logged every visitor event across years (N huge), the closed-form Normal Equation stops being a option — not because the math is wrong, but because no machine can invert that matrix in a reasonable time.
Stage 2 → 7 · Engineering Thinking, Formula Construction, Algorithm

2Gradient Descent — Walking Downhill

The idea in one sentence

At your current guess, the gradient points in the direction of steepest increase. So step in the exact opposite direction, by some small amount, and you're guaranteed to be a little lower than before — repeat until you stop improving.

The update rule
Wt+1 = Wt − ηt ∇f(Wt)
Eq. 34.1
Wt — the weight vector at iteration t
ηt — the step size / learning rate at iteration t
∇f(Wt) — the gradient of the objective, evaluated at the current W

Initialize

Start with an initial guess W₀ (often all zeros, or random)

Compute the gradient

Evaluate ∇f(Wₜ) — for sum-of-squared-errors, this is 2XXᵀWₜ − 2XY

Step downhill

Wₜ₊₁ = Wₜ − ηₜ∇f(Wₜ)

Repeat

Until the gradient is ≈ 0, or a fixed number of iterations passes

🖼Descending a Convex Bowl

Reading the diagram The rings are level sets of the objective — every point on one ring has the same error value. Because the sum-of-squared-errors surface is convex (a single smooth bowl, no other dips or bumps), a step "downhill" always moves toward the one and only minimum. There's no risk of getting trapped in a false valley.
Stage 6 · Why the Convergence Guarantee Holds

6Convexity Is Doing All the Work

The key property, stated precisely

The sum-of-squared-errors objective for linear regression is a quadratic, convex function of W. Convexity means: any local minimum is automatically the global minimum — there is no other valley to get stuck in.

The one condition attached to the guarantee "Gradient descent converges to the global minimum" comes with a fine-print condition stated directly in the lecture: the step size ηₜ must be chosen correctly. Too large a step can overshoot the minimum and diverge; too small a step converges, just very slowly.
Beyond what's stated — the technical reason, for context The precise condition (not spelled out in this lecture, but the standard theory behind the claim) is that ηₜ needs to be smaller than 2 divided by the largest eigenvalue of the Hessian (2XXᵀ). This is why learning rate is usually tuned rather than fixed arbitrarily.
Stage 2 → 7 · Engineering Thinking, Algorithm

3Stochastic Gradient Descent — Estimate, Don't Compute Exactly

The core trade-off

Full-batch gradient descent still requires touching all N data points at every single step — that's the N-scaling problem from the very start of this lecture, unsolved. SGD's answer: don't use all N points per step. Use a small random sample instead, and accept that each step's gradient is noisy rather than exact.

StepWhat happens
1. SampleDraw K points uniformly at random to form X̃ and Ỹ (K = batch size)
2. PretendTreat this small sample as if it were the entire dataset
3. UpdateCompute the gradient using only these K points, and take a normal GD step
SGD update rule — same shape as Eq. 34.1, different gradient source
Wt+1 = Wt − ηt ∇fK(Wt)
Eq. 34.2
∇fK — gradient computed from only the K sampled points, a noisy stand-in for the true full-dataset gradient
Handling the noise: averaging Because each step's gradient is noisy, the final Wₜ after T iterations can bounce around near the optimum rather than settling exactly on it. The fix stated in the lecture: instead of reporting the last Wₜ, report the average of every W encountered during training — this smooths out the sampling noise.

Why this actually solves the scaling problem

PropertyBenefit
Decoupled from DNo D×D matrix inversion anywhere in the loop
Decoupled from NEach step only touches K points, not all N — K is fixed and chosen by you, independent of dataset size
Many cheap stepsFaster overall progress than one enormous, exact-but-expensive computation
Stage 9 · Edge Cases

9Where GD/SGD Can Misbehave

Step size too large

Overshoots the minimum on every step — the "path" bounces between the walls of the bowl and can diverge outward instead of converging.

Step size too small

Converges eventually, but each step makes almost no progress — you trade divergence risk for painfully slow training.

K = 1 (pure SGD)

The noisiest possible gradient estimate — cheapest per step, but needs the most steps and benefits the most from weight-averaging.

K = N (full-batch GD)

SGD collapses back into ordinary gradient descent — no noise, but back to touching every point per step.

Stage 18 · Examiner Trap Library

18Common Exam Traps

⚠ Trap 1 — "convex" alone isn't the whole convergence guarantee

Convexity guarantees no bad local minima exist — it does NOT by itself guarantee convergence regardless of step size. A question that says "the function is convex, so any step size works" is testing whether you remember the step-size condition is separate and required.

⚠ Trap 2 — treating SGD's per-step gradient as exact

∇f_K(Wₜ) is an estimate of the true gradient, built from K randomly sampled points — not the true gradient itself. Any claim that a single SGD step moves in "the" gradient direction is imprecise; it moves in a noisy estimate of it.

⚠ Trap 3 — forgetting why averaging W helps

The final Wₜ after SGD is not necessarily the best estimate — because of per-step noise, it may be off in some random direction on that particular iteration. Averaging over all iterations cancels out that noise; using only Wₜ ignores this correction.

Stage 20 · Industry Implementation

20GD and SGD in Practice

# Plain gradient descent for linear regression (small-scale, illustrative)
import numpy as np

W = np.zeros(D)
eta = 0.01
for t in range(1000):
    grad = 2 * X @ X.T @ W - 2 * X @ Y   # full-dataset gradient, O(N) per step
    W = W - eta * grad

# SGD: sample K points per step instead
W = np.zeros(D)
W_history = []
for t in range(1000):
    idx = np.random.choice(N, size=K, replace=False)
    X_tilde, Y_tilde = X[:, idx], Y[idx]
    grad = 2 * X_tilde @ X_tilde.T @ W - 2 * X_tilde @ Y_tilde
    W = W - eta * grad
    W_history.append(W.copy())

W_avg = np.mean(W_history, axis=0)   # the averaged, noise-smoothed estimate

# In practice: sklearn.linear_model.SGDRegressor implements this directly,
# with batch size, learning-rate schedules, and averaging options built in.

Stage 22 · One-Page Brain

22Everything on One Page

Core Intuition

  • Closed-form W* is exact but O(D³) to invert, O(N·) to even build XXᵀ
  • GD trades exactness for cheap iterative steps downhill
  • Convexity of squared error ⟹ no bad local minima
  • SGD replaces the full-dataset gradient with a noisy K-sample estimate

Key Formulas

  • GD: Wₜ₊₁ = Wₜ − ηₜ∇f(Wₜ)
  • SGD: Wₜ₊₁ = Wₜ − ηₜ∇f_K(Wₜ), K-point sample
  • Final estimate: average of all Wₜ, not just W_T

5 Triggers

  • "closed form too expensive"
  • "iterative optimization" / "step size"
  • "convex ⟹ global minimum" (with a step-size caveat)
  • "batch size K" / "mini-batch"
  • "noisy gradient" / "average the weights"

5 Traps

  • Convexity ≠ automatic convergence — step size still matters
  • SGD's gradient is an estimate, not the true gradient
  • Final Wₜ can be noisy; averaging is the fix, not optional
  • K=1 → pure SGD; K=N → collapses back to full GD
  • SGD fixes the N-scaling problem; it doesn't touch the D³ inversion problem, which GD avoids differently (never forming/inverting XXᵀ at all)
