0Why This Topic Exists

The problem unsupervised learning can't solve

Everything before this lecture — K-means, GMM, EM — worked on data with no answer key. You handed the algorithm points and asked it to find its own structure: clusters, densities, latent groupings. That's powerful, but it can't answer a very ordinary business question: "is this specific thing X actually a Y?"

Supervised learning exists because most real decisions need a known correct answer to learn from. You don't want a model to invent its own idea of what "spam" means — you want it to learn your definition of spam from thousands of emails you've already labeled.

RMS / Cubedekho lens If you were running lead-scoring for RMS, unsupervised clustering might discover "there seem to be 3 kinds of visitors." That's interesting but not actionable. Supervised learning answers the question you actually care about: "will this specific lead convert, yes or no?" — because you can label your past leads with what actually happened.

Engineering requirements before any math

Before writing a single formula, any supervised system has to satisfy three practical constraints:

RequirementWhy it matters
Must generalize to unseen inputsMemorizing the training emails is useless — new spam arrives every day
Must handle the right kind of outputA yes/no answer needs different machinery than a rainfall amount in mm
Must be learnable from a finite labeled sampleYou'll never have every possible email or every possible rainfall scenario
Stage 3 → 5 · Mathematical Translation, Formula Construction, Anatomy

3Setting Up the Formal Problem

Turn the engineering requirements above into notation, one piece at a time.

Step 1 — every input is a point living in a d-dimensional space (its features):

Feature space
X1, X2, …, Xn  ∈  ℝd
Eq. 27.1

Step 2 — every input is paired with a label, the "correct answer" a teacher supplies:

Labeled dataset
D  =  { (X1, Y1), (X2, Y2), …, (Xn, Yn) }
Eq. 27.2
X — feature vector (also called an attribute or predictor)
Y — label (also called supervision)
d — number of features describing one input
n — number of labeled examples available

Step 3 — the goal is a function that maps any feature vector to a predicted label, including ones it has never seen:

Learning objective
f : ℝd → 𝒴    such that    f(Xnew) ≈ Ynew
Eq. 27.3
f — the model / hypothesis we are trying to learn
𝒴 — the label space; what 𝒴 looks like is what defines the whole problem type below
The one idea that matters most Everything in this lecture reduces to a single design choice: what does 𝒴 look like? Two values, several values, or the entire real number line — and each answer spins up an entirely different branch of machine learning.
Stage 7, 11, 12 · Algorithm shape, Pattern Recognition, Formula-Shape Recognition

7The Three Shapes of 𝒴

Three problem families fall out of the single question above — nothing more exotic than "how many values can the label take, and are they ordered?"

𝒴 has 2 values

Binary Classification

𝒴 = { +1, −1 } or { 0, 1 }

One of exactly two buckets. No in-between.

Example: Spam Detection — an email's features go in, "spam" / "non-spam" comes out.

𝒴 has K+1 values

Multi-class Classification

𝒴 = { 0, 1, …, K }

A finite set of unordered categories, more than two.

Example: Digit Recognition — an image goes in, one digit 0–9 comes out.

𝒴 is continuous

Regression

𝒴 = ℝ = (−∞, +∞)

Any real number — infinitely many possible answers, with deep statistical roots.

Example: Rainfall Prediction — temperature, humidity, pressure go in, exact mm of rainfall comes out.

Mapped onto your businesses Binary → "will this Cubedekho cart abandon or convert?"  ·  Multi-class → "which cube category is this product photo — 3×3, pyramid, or megaminx?"  ·  Regression → "how much revenue will this RMS batch generate?" Same three shapes, three completely different businesses.

🖼Visual Intuition: Same Pipeline, Different Output Shape

💻The Same Idea, in Code

None of these problems needs a specific algorithm yet — this course hasn't built one. But the shape of the output already tells you what a scikit-learn call for each would look like:

Binary — spam detection

# X_email = feature vector built from an email (word counts, sender, links...)
# Y ∈ {spam, non-spam}  → exactly 2 possible outputs
from sklearn.linear_model import LogisticRegression

model = LogisticRegression()
model.fit(X_train, y_train)          # y_train is a 0/1 array
prediction = model.predict(X_new)    # returns 0 (not spam) or 1 (spam)

Multi-class — digit recognition

# X_image = pixel values of a handwritten digit image
# Y ∈ {0, 1, 2, ..., 9}  → 10 possible outputs
from sklearn.svm import SVC

model = SVC(decision_function_shape='ovr')  # one-vs-rest under the hood
model.fit(X_train, y_train)                 # y_train has values 0–9
prediction = model.predict(X_new)           # returns a single digit

Regression — rainfall prediction

# X_weather = [temperature, humidity, pressure]
# Y ∈ ℝ  → any real number, e.g. 14.7 (mm)
from sklearn.linear_model import LinearRegression

model = LinearRegression()
model.fit(X_train, y_train)          # y_train is continuous, e.g. [0.0, 14.7, 32.1, ...]
prediction = model.predict(X_new)    # returns a real number, not a category
Notice The import line changes with the shape of 𝒴 — LogisticRegression/SVC for discrete outputs, LinearRegression for continuous ones. Using a classifier's .predict() on a regression target (or vice-versa) is a category error, not just a suboptimal choice.

Stage 16 · Invariant Library

16Always / Sometimes / Never True

Always True

Every supervised problem has a feature space ℝᵈ and a label space 𝒴 defined before training starts.

The label space 𝒴 alone determines whether the problem is classification or regression.

Sometimes True

Binary labels are sometimes encoded {0,1} and sometimes {+1,−1} — the encoding is a convention chosen per algorithm, not a rule of the problem itself.

Never True

Regression labels are never a finite fixed set — the moment 𝒴 is finite and discrete, it is classification, not regression, no matter how many values K has.

Stage 18 · Examiner Trap Library

18Common Exam Traps

⚠ Trap 1 — "Rating 1 to 5" looks continuous but isn't automatically regression

A 1–5 star rating has a finite, discrete set of values. Whether it's treated as multi-class classification or as regression is a modeling choice, not something the label space forces on you — questions love testing whether you notice this ambiguity.

⚠ Trap 2 — confusing "two classes" with "binary" when order matters

Binary classification assumes no meaningful order between the two labels. If the two outcomes have an implicit order (e.g., "pass/fail" as a proxy for a score), that ordering information is thrown away by treating it as plain binary classification.

⚠ Trap 3 — mixing up d and n

d is the number of features per example (dimensionality), n is the number of examples in the dataset. Swapping them in a formula is one of the most common silent errors in this course.

Stage 22 · One-Page Brain

22Everything on One Page

Core Intuition

  • Supervised = learn from (input, correct-answer) pairs
  • Goal: predict correctly on unseen inputs
  • Problem type = shape of the label space 𝒴

Key Formulas

  • Xᵢ ∈ ℝᵈ — feature vectors
  • D = {(Xᵢ, Yᵢ)} — labeled dataset
  • f : ℝᵈ → 𝒴 — the model
  • Binary: 𝒴 = {+1, −1}
  • Multi-class: 𝒴 = {0, …, K}

5 Triggers

  • "labeled data" / "training pairs"
  • "spam or not" → binary
  • "which digit / category" → multi-class
  • "predict the amount" → regression
  • units attached to the answer (mm, ₹, kg) → regression

5 Traps

  • Discrete-looking ratings ≠ automatic regression
  • Two classes with hidden order ≠ plain binary
  • d (features) vs n (examples) mix-up
  • {0,1} vs {+1,−1} is convention, not a rule
  • Finite label set is always classification, never regression
