Vue Form Validation

Every form your app collects has to pass through two checkpoints — a friendly one that helps the user, and a strict one that protects the server. This is how Vue builds both.

🟦 Client Gate
Fast · friendly · for UX
→Required fields filled?
→Email matches a pattern?
→Numbers add up correctly?
→Show errors instantly, no page reload
→
🟪 Server Gate
Strict · mandatory · for security
→Re-checks everything, trusts nothing
→Blocks malicious or malformed input
→Protects the database
→The only gate that truly matters for security
The Two Gates — client validation is a courtesy, server validation is the law

2Core Intuition · WHY

Why this exists

🛂
Think of a nightclub bouncer and an airport security scanner. The bouncer (client-side validation) gives you fast, friendly feedback at the door — "no, you forgot your ID" — so you're not stuck in a long line for nothing. But a determined person can walk around the bouncer. Airport security (server-side validation) is the checkpoint that actually can't be skipped, because it's the one guarding what really matters.

Client-side validation exists purely to make the experience faster and friendlier. It is never a security boundary — anyone can open DevTools, disable JavaScript, or send a raw request straight to your server, bypassing every client-side check entirely.

5Deep Fundamentals

The four questions

1. Why was this feature created?

Forms are the main channel through which users send data into an application. Left unchecked, bad input can break business logic, corrupt data, or open security holes — so both browsers and frameworks evolved tools to catch problems as early as possible.

2. What problem does it solve?

Two separate problems, actually: a UX problem (don't make users wait on a round trip to discover they forgot the "@" in their email) and a security problem (never trust anything a client claims, because the client can be manipulated).

3. What happens internally?

Vue's v-model keeps a form field's value in sync with reactive data via a getter/setter pair. On submit, the browser's default action (a real HTTP POST + page navigation) is intercepted so custom JavaScript can run first.

4. How does Vue/JavaScript execute it?

event.preventDefault() — written as .prevent in Vue's template shorthand — synchronously cancels the browser's default submission behavior the instant the handler runs, before any validation logic even executes.

7Syntax Anatomy

Every keyword, explained

<form novalidate @submit.prevent="checkForm">
novalidate tells the browser: skip your built-in popups, I'm handling validation myself
@submit Vue's shorthand for listening to the form's submit event
.prevent Vue's event modifier — automatically calls event.preventDefault() for you
checkForm the method that runs the actual validation logic
<p v-if="errors.length">
  <li v-for="e in errors">{{ e }}</li>
</p>
v-if only renders this block into the DOM when errors exist — nothing rendered at all when the array is empty
errors.length truthy check — zero is falsy, any count above zero is truthy
v-for loops over each error message to list them individually
8Code Walkthrough

A tiny validated form

✗ Please enter a valid email address
✗ Your items must add up to exactly 1000
export default {
  data() {
    return {
      email: '',
      budgetItems: [400, 400],
      errors: []
    }
  },
  methods: {
    checkForm() {
      this.errors = []   // reset before every check

      const emailPattern = /^[^\s@]+@[^\s@]+\.[^\s@]+$/
      if (!emailPattern.test(this.email)) {
        this.errors.push('Please enter a valid email address')
      }

      const total = this.budgetItems.reduce((a,b) => a+b, 0)
      if (total !== 1000) {
        this.errors.push('Your items must add up to exactly 1000')
      }

      if (this.errors.length) { return }   // stop here, .prevent already blocked the POST

      // otherwise, safe to send onward to the server
      fetch('/api/submit', { method: 'POST', body: JSON.stringify(this.$data) })
    }
  }
}
9Behind the Scenes

What Vue & the browser are really doing

event.preventDefault()

Every DOM event carries a default action — for a form's submit event, that default is "send an HTTP POST and navigate away." Calling preventDefault() flags the event so the browser skips that default action, while everything else about the event still fires normally.

Async validation

A fetch-based check (e.g. "does this city exist?") is asynchronous — the function has to pause with await, often flipping a loading flag so the UI can show a spinner while it waits for the network response.

⚠️
Because JavaScript can be disabled, edited in DevTools, or skipped entirely by a direct API call, every one of these client-side checks must be assumed absent by the server. The server re-runs its own version of every rule, independently.
10Real Project Usage

Where you've already used this

Google Sign-up

Password strength and "passwords match" checks appear instantly as you type — pure client-side UX, no server round trip needed yet.

Username availability

Gmail/Instagram-style sign-ups fire an async request to the server as you type a username, since only the server's database actually knows what's taken.

Checkout card fields

Amazon/Stripe checkouts validate card number format and expiry client-side before ever hitting the payment network, saving a wasted request.

Weather app city search

Before saving a city, the app calls the weather API to confirm the city actually exists — a validation call that can only be answered by the server.

11Common Mistakes

Wrong vs. Right

❌ WrongTrusting client-side validation as the only line of defense
✅ CorrectAlways re-validate every field again on the server, independently
❌ WrongForgetting .prevent on @submit
✅ Correct@submit.prevent="checkForm" — stops the page reloading before validation even runs
❌ WrongUsing v-if for something toggled on every keystroke
✅ CorrectUse v-show for frequently-toggled elements — cheaper than repeatedly destroying/recreating DOM nodes
❌ WrongAppending to the errors array on every check without resetting first
✅ Correctthis.errors = [] at the start of every validation run

14Interview Questions

Three tiers

Easy
What's the difference between v-if and v-show?
v-if adds/removes the element from the DOM entirely; v-show keeps it in the DOM and just toggles CSS display.
Medium
Why is client-side validation not enough on its own?
Client-side code runs in an environment the user fully controls — it can be disabled, edited, or bypassed by sending requests directly to the server, so the server must independently verify every field.
Advanced
How would you implement an async validation check without blocking the UI?
Mark the field as "checking" with a loading flag, fire the fetch call, await its response, then update the errors array and clear the loading flag — the UI reflects each state (idle/checking/valid/invalid) reactively throughout.
