2Core Intuition · WHY

Why this exists

💡
A Vue app's data is like writing on a whiteboard. The moment you refresh the page, someone wipes the whiteboard clean and hands you a fresh one — Vue re-initializes from scratch. Persistent storage is simply a filing cabinet next to the whiteboard: before the board gets wiped, you copy what matters onto paper and file it away, so next time you can pull it back out instead of starting over.

Browsers give you four different "cabinets" for this — cookies, sessionStorage, localStorage, and IndexedDB — each with a different lifespan, size limit, and level of structure. Choosing the right one is the whole skill.

5Deep Fundamentals

The four questions

1. Why was this feature created?

HTTP is stateless — a server forgets you the instant a response is sent. Cookies (1994) were invented so a server could stamp an ID on your browser and recognize you next request. Later, as apps became more client-heavy (SPAs, offline apps), browsers added Web Storage (2009) for pure client-side key-value data, and IndexedDB for anything that outgrew a simple key-value pair.

2. What problem does it solve?

Without it, every reload = total amnesia. A Vue app re-runs its data() function from scratch on every load, so anything not saved outside the JS runtime disappears — preferences, cart contents, draft text, game progress, all gone.

3. What happens internally?

The browser maintains a storage area partitioned by origin (protocol + domain + port). Each origin gets its own isolated cookie jar, sessionStorage, localStorage, and IndexedDB database — your site can never read another site's storage.

4. How does Vue/JavaScript execute it?

Web Storage (local/session) is synchronous — a plain string-only key-value dictionary you read/write directly. IndexedDB is asynchronous and transactional — you open a connection, start a transaction, and get results via events or promises.

7Syntax Anatomy

Every keyword, explained

localStorage.setItem('cities', JSON.stringify(this.cities))
localStorage the browser's global persistent key-value store for this origin
.setItem() writes a value under a key — always takes two strings
'cities' the key you'll use later to retrieve this exact value
JSON.stringify() converts the array/object into a string, because storage only accepts strings
this.cities the reactive Vue data property being saved
mounted() {
  this.cities = JSON.parse(localStorage.getItem('cities')) || []
}
mounted the lifecycle hook Vue calls once the component is on the real DOM — safe place to read storage
JSON.parse turns the saved string back into a real array/object
|| [] fallback for the very first visit, when nothing has been saved yet
8Code Walkthrough

A tiny persisted city list

data()
→
mounted()
→
load from storage
→
watch()
→
save to storage
export default {
  data() {
    return {
      cities: []       // starts empty every time the app boots
    }
  },
  mounted() {
    // 1. pull any previously saved cities back in
    this.cities = JSON.parse(localStorage.getItem('cities')) || []
  },
  watch: {
    cities(newVal) {
      // 2. any time the list changes, re-save it automatically
      localStorage.setItem('cities', JSON.stringify(newVal))
    },
    deep: true
  }
}
✅
Add a city, refresh the page, and it's still there — the watcher wrote it to localStorage before the reload wiped Vue's memory.
9Behind the Scenes

What the browser is really doing

setItem() called
→
browser writes to an on-disk store, per origin
→
main thread blocks briefly (sync API)
→
data available to any tab on the same origin

localStorage / sessionStorage

Synchronous, string-only, capped around 5–10 MB per origin. Because it's synchronous, large reads/writes can briefly freeze the UI — one reason to keep what you store small.

IndexedDB

Asynchronous and transactional, built like a lightweight NoSQL database with "object stores" and indexes. Much larger capacity, doesn't block the main thread — the trade-off is a more verbose API.

10Real Project Usage

Where you've already used this

Gmail — offline mode

Recent emails are cached in IndexedDB so you can open and read messages on a flight with no signal, syncing back up once you reconnect.

Spotify Web Player

Volume level, last queue, and UI preferences are saved to localStorage so the player looks and sounds the same every time you come back.

Amazon guest cart

A cookie identifies your session so the server can match your cart to you across page loads, even before you log in.

Multi-step checkout forms

Form progress is often kept in sessionStorage — meant to last only for this checkout attempt, cleared once the tab closes.

🌦️
A weather app remembering your three saved cities, or a browser game remembering your level progress — both textbook cases of "small amount of state, doesn't need a server round-trip, just needs to survive a refresh."
11Common Mistakes

Wrong vs. Right

❌ WronglocalStorage.setItem('cities', this.cities) — stores the literal text "[object Object]"
✅ CorrectlocalStorage.setItem('cities', JSON.stringify(this.cities))
❌ WrongAssuming localStorage syncs your data to your phone or another browser
✅ CorrectIt's local to one browser on one machine — nothing syncs across devices automatically
❌ WrongStoring auth tokens or sensitive personal data in localStorage
✅ CorrectKeep sensitive data server-side / in secure cookies — anything in localStorage is readable by any script on the page
❌ WrongTreating local storage as the "real" copy of important data
✅ CorrectThe server remains the master copy; local storage only supplements performance and UX

14Interview Questions

Three tiers

Easy
What's the difference between localStorage and sessionStorage?
localStorage persists indefinitely per origin; sessionStorage is cleared as soon as the tab/window closes.
Medium
Why do you need JSON.stringify/parse with localStorage?
Web Storage only accepts and returns strings, so any object or array must be serialized to a string to store it and deserialized to use it again.
Advanced
When would you choose IndexedDB over localStorage, and what does that cost you?
When data is large, structured, or needs querying/indexing — IndexedDB scales far beyond Web Storage's ~5–10MB. The cost is a more complex, asynchronous, transaction-based API instead of simple get/set calls.
