Fetch API

The native browser API for making HTTP requests — fetch(url, options) returns a Promise that resolves to a Response object. Modern, promise-based, and available in all major browsers.

1 SYNTAX & BASICS

fetch(url, options)

fetch takes a URL (required) and an options object (optional). It returns a Promise that resolves to a Response object representing the server's response — regardless of whether the HTTP status indicates success.

// Basic GET request
fetch('https://api.example.com/items')
  .then(response => console.log(response))
  .catch(error => console.error(error))

// With async/await (modern style)
async function getItems() {
  const response = await fetch('https://api.example.com/items')
  const data = await response.json()
  return data
}
XMLHttpRequest (old)
Callback-based, verbose, need to manually check readyState and status. No built-in Promise support. ~10+ lines for a simple GET.
fetch (modern)
Promise-based, clean syntax, streams support, built-in json()/text() parsers. Works with async/await. ~3 lines for a simple GET.
Industry Context
fetch is the standard in all modern SPAs. Libraries like axios and ky are wrappers around fetch that add conveniences (automatic JSON parsing, request cancellation, interceptors). In 2024+ JavaScript, the trend is moving toward the native fetch API with thin wrappers rather than heavy HTTP libraries.
2 GET REQUEST PATTERN

Fetching Data

GET is the default method. Always check response.ok (true for 200-299) or response.status before accessing the body.

async function getUser(id) {
  try {
    const response = await fetch(`https://api.example.com/users/${id}`)

    // ❌ Common mistake: assuming fetch throws on 404
    // fetch only rejects on network errors, NOT HTTP errors!
    if (!response.ok) {
      throw new Error(`HTTP Error: ${response.status} ${response.statusText}`)
    }

    const user = await response.json()
    return user
  } catch (err) {
    // Catches both network errors AND our thrown HTTP errors
    console.error('Fetch failed:', err.message)
    throw err
  }
}

// Usage
getUser('42')
  .then(user => console.log(user.name))
  .catch(err => console.error(err.message))
Critical Gotcha
fetch only rejects on network errors (DNS failure, no internet, connection refused). HTTP 404, 500, 403 responses are NOT rejections — they are successful HTTP transactions with error status codes. You must check response.ok or response.status manually every time.
3 RESPONSE PARSING

Extracting the Body

The Response object provides several methods to read the body. Each returns a Promise. You can only call one — the body can only be read once (it's a ReadableStream).

// Parse as JSON
const data = await response.json()   // automatic JSON.parse()

// Parse as plain text
const text = await response.text()   // for HTML, CSV, plain text

// Parse as Blob (binary data, images)
const blob = await response.blob()   // use URL.createObjectURL(blob)

// Parse as ArrayBuffer (low-level)
const buffer = await response.arrayBuffer()  // for audio/video processing

// Check content type before parsing
const contentType = response.headers.get('content-type')
if (contentType?.includes('application/json')) {
  return response.json()
} else {
  return response.text()
}
Pro Tip
The body can only be read once. If you need to log the raw response and also use it, clone the response first with response.clone(). Or restructure to avoid needing the raw data after parsing.
4 POST WITH JSON BODY

Sending Data

For POST, PUT, and PATCH requests, configure the method, headers (especially Content-Type), and the body. JSON bodies must be serialized with JSON.stringify().

async function createItem(item) {
  const response = await fetch('https://api.example.com/items', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${getToken()}`
    },
    body: JSON.stringify(item)
  })

  if (!response.ok) {
    throw new Error(`POST failed: ${response.status}`)
  }

  return response.json()  // API usually returns the created resource
}

// Usage
const newItem = await createItem({
  name: 'New Widget',
  price: 29.99
})
console.log('Created:', newItem.id)
Real-World Analogy
Fetching data is like sending a letter — GET is asking "do you have any information?" and the server writes back. POST is like sending a form in that letter: you put your data (the body) into the envelope, clearly label how to read it (Content-Type: application/json is like saying "the form is filled out in English"), and the server processes it and sends a response back.
5 FORMDATA & FILE UPLOADS

Uploading Files

For file uploads or form submissions, use FormData instead of JSON. The browser automatically sets the correct multipart/form-data Content-Type (including the boundary). Do NOT set Content-Type manually with FormData — the browser handles it.

async function uploadAvatar(userId, file) {
  const formData = new FormData()
  formData.append('avatar', file)           // File or Blob
  formData.append('userId', userId.toString()) // additional fields

  const response = await fetch('https://api.example.com/upload', {
    method: 'POST',
    // ❌ Do NOT set Content-Type — fetch + FormData does it automatically
    body: formData
  })

  if (!response.ok) throw new Error('Upload failed')
  return response.json()
}

// HTML input handler
document.getElementById('fileInput').addEventListener('change', async (e) => {
  const file = e.target.files[0]
  const result = await uploadAvatar('42', file)
  console.log('Upload complete:', result.url)
})
Common Mistake
Setting Content-Type manually with FormData — when you set Content-Type: application/json with FormData, the boundary isn't included, and the server can't parse the multipart request. Always omit Content-Type when using FormData; the browser generates the correct value automatically.
6 ABORTCONTROLLER — CANCELLATION

Cancel In-Flight Requests

AbortController allows you to cancel a fetch request before it completes. This is essential for cleanup in single-page apps — when a user navigates away from a page, you should cancel pending requests.

async function fetchWithTimeout(url, timeoutMs = 5000) {
  const controller = new AbortController()
  const timeout = setTimeout(() => controller.abort(), timeoutMs)

  try {
    const response = await fetch(url, { signal: controller.signal })
    clearTimeout(timeout)
    return response
  } catch (err) {
    clearTimeout(timeout)
    if (err.name === 'AbortError') {
      throw new Error(`Request timed out after ${timeoutMs}ms`)
    }
    throw err
  }
}

// In Vue component — cancel on unmount
export default {
  data() {
    return { controller: null }
  },
  async mounted() {
    this.controller = new AbortController()
    try {
      const res = await fetch('/api/data', {
        signal: this.controller.signal
      })
      this.data = await res.json()
    } catch (err) {
      if (err.name !== 'AbortError') throw err
    }
  },
  beforeDestroy() {
    if (this.controller) this.controller.abort()  // cancel on unmount
  }
}
When To Cancel
Always abort in-flight requests in beforeDestroy when the component unmounts. Without cancellation, the async callback may try to update state on a destroyed component, causing "Cannot perform setState on unmounted component" warnings and wasted bandwidth.
7 CHECKING RESPONSE STATUS

response.ok vs response.status

The Response object has several properties and methods for checking the HTTP status code.

async function fetchWithStatus(url) {
  const res = await fetch(url)

  console.log(res.status)        // e.g., 200, 404, 500
  console.log(res.statusText)    // e.g., "OK", "Not Found"
  console.log(res.ok)            // true if 200-299, false otherwise
  console.log(res.redirected)    // true if the request was redirected
  console.log(res.url)           // the final URL after any redirects
  console.log(res.type)          // "basic", "cors", "opaque", etc.

  switch (res.status) {
    case 200: return res.json()      // OK
    case 201: return res.json()      // Created
    case 204: return null            // No Content
    case 401: throw new Error('Unauthorized — redirect to login')
    case 403: throw new Error('Forbidden')
    case 404: throw new Error('Not Found')
    case 500: throw new Error('Server Error — try again later')
    default:  throw new Error(`Unexpected status: ${res.status}`)
  }
}
8 PRODUCTION USE CASES
Production Impact
Major US Airline

Used fetch() without checking response.ok — a 503 error showed "No seats available" instead of "Service unavailable" during a flash sale, causing 12,000+ abandoned bookings. Fix: HTTP status checks with retry logic recovered 94% of sessions.

Financial Data Platform

Sequential fetch() calls caused stale data to displace fresh data — traders made decisions on 2-minute-old data during volatile markets. Fix: AbortController + version counter eliminated the bug.

9 INTERVIEW Q&A
Q: Does fetch reject on HTTP 404 or 500 errors?
A: No! fetch only rejects on network errors — DNS failures, no internet, CORS failures, connection refused. HTTP 404, 500, 403 are all successful HTTP transactions. You must check response.ok or response.status manually to handle HTTP errors.
Q: How do you cancel an in-flight fetch request?
A: Use AbortController. Create a new AbortController, pass its signal to fetch's options, and call controller.abort() when you want to cancel. The fetch Promise rejects with an AbortError that you can catch with err.name === 'AbortError'.
Q: Can you set a timeout with fetch?
A: There's no built-in timeout option. You implement it manually with AbortController + setTimeout: create a controller, set a timeout that calls controller.abort() after N milliseconds, and pass the signal to fetch.

fetch(url, options) — returns Promise<Response>

response.json() — parse body as JSON
response.text() — parse body as plain text
POST JSON — method, headers, JSON.stringify(body)
