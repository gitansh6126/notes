02Core Intuition

Why Fetch Exists

Calling fetch() is like handing a courier an address and walking away — you get a tracking number (a Promise) immediately, long before the courier has even left the building. That tracking number only tells you the courier reached the destination and got something back. It says nothing about whether the "something" was the package you wanted, a "sorry, out of stock" note, or an empty box. Reading the actual contents is a separate step you do afterward.

05Deep Fundamentals

Question the Feature, Not Just the Syntax

Why was it created?

Its predecessor, XMLHttpRequest, needed nested callbacks and verbose event listeners for every request — hard to read, harder to chain.

What problem does it solve?

It gives every network call a single, Promise-shaped return value, so requests can be chained with .then() or written linearly with async/await.

What happens internally?

The browser opens a connection, sends the request, and resolves the first Promise the moment response headers arrive — the body may still be streaming in.

How does JavaScript execute it?

fetch() is non-blocking: it fires the request and immediately returns control to your code. The event loop keeps running; the Promise settles later via the microtask queue once the network layer reports back.

07Syntax Anatomy

Take the Call Apart

syntaxfetch(url, init)
fetch(url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(data)
})
  • fetchGlobal browser function — a built-in Web API, not a language keyword.
  • urlFirst argument: the resource address, as a string (or a Request object).
  • { }Second argument: the optional init config object.
  • methodThe HTTP verb — GET, POST, PUT, DELETE. Defaults to GET if omitted.
  • headersKey–value metadata about the request, e.g. telling the server the body is JSON.
  • bodyThe payload for POST/PUT — must be a string, FormData, or Blob, never a raw object.
  • returnsA Promise<Response> — resolved once headers come back.
08Code Walkthrough

Five Implementations, Line by Line

1 · Basic GET request

basic-get.js2 promises, 2 awaits
async function getUsers() {
  // Promise #1: resolves once headers arrive
  const response = await fetch('/api/users');
  // Promise #2: resolves once the body is parsed
  const data = await response.json();
  return data;
}

Two separate promises are awaited: one for the trip, one for unwrapping the parcel.

2 · Handling HTTP errors

check-status.jsresponse.ok is manual
async function getUsers() {
  const response = await fetch('/api/users');
  if (!response.ok) {
    // fetch() never rejects for a 404 — you must throw it yourself
    throw new Error(`HTTP ` + response.status);
  }
  return await response.json();
}

Without this check, a 404 page silently flows through your code as "successful" data.

3 · Sending JSON (POST)

post-json.jsstringify + header, always paired
async function createUser(user) {
  const response = await fetch('/api/users', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(user)
  });
  return await response.json();
}

The header and the stringify call are a matched pair — the header tells the server how to read the body you just built.

4 · Form & file upload

form-upload.jsFormData sets its own headers
async function uploadAvatar(formEl) {
  const data = new FormData(formEl); // reads <input type="file"> too
  const response = await fetch('/api/avatar', {
    method: 'POST',
    body: data // no Content-Type header — the browser sets it
  });
  return response.ok;
}

FormData reads every field in the form, including binary file input, and packages it as multipart data automatically.

5 · Async/await version, end to end

full-flow.jstry/catch wraps both awaits
async function loadUser(id) {
  try {
    const response = await fetch(`/api/users/` + id);
    if (!response.ok) throw new Error('Not found');
    const user = await response.json();
    return user;
  } catch (err) {
    // catches BOTH network failures and thrown HTTP errors
    console.error('Load failed:', err);
  }
}
09Behind the Scenes

What the Browser Is Actually Doing

fetch() called
→
Socket opened, TCP/TLS handshake
→
HTTP request sent
Server replies with headers first
→
Response object created (body still streaming)
→
Promise #1 resolves
.json() reads & parses the stream
→
Promise #2 resolves

This is why the two promises exist: the connection can succeed (Promise #1) long before the browser has finished downloading and parsing the actual body (Promise #2).

10Real Project Usage

Where This Runs Right Now

Netflix

Fetches thumbnail metadata and "next episode" info as you scroll, so rows load in the background without freezing the UI.

Instagram

Pull-to-refresh fires a fetch for new feed posts; the app checks response.ok before swapping in new content.

Swiggy / Zomato

Fetches nearby restaurants based on location, then POSTs your cart as JSON when you place an order.

Gmail

Background fetches poll for new mail; attachments are retrieved with .blob() for binary file data.

11Common Mistakes

Wrong vs. Correct

❌ WrongAssuming fetch() rejects on a 404 or 500 response.
✅ CorrectAlways check response.ok (or response.status) and throw manually if it's false.
❌ WrongUsing the raw response as if it were your data.
✅ Correctawait response.json() to unwrap the actual payload — that's a second promise.
❌ WrongPassing a plain object directly as body.
✅ CorrectJSON.stringify(obj) plus a matching Content-Type header.
❌ WrongLeaving network failures uncaught.
✅ CorrectWrap awaited calls in try/catch, or chain .catch().
12Debugging Mindset

Problem → Observe → Find Stage → Fix

TypeError: Failed to fetch

Observe: no status code shows in the Network tab at all. Find stage: the request never reached the server — DNS, offline, or CORS. Fix: check the URL, connectivity, and CORS headers.

data is undefined after the call

Observe: you logged the raw response, not the parsed body. Find stage: the parsing step. Fix: await response.json() before using the result.

Silent failure on a 404 page

Observe: code runs "successfully" but the UI shows garbage. Find stage: missing status check. Fix: add the response.ok guard.

13IITM Exam Pattern

How This Shows Up on Paper

Trigger words"asynchronous", "resolves", "network error", "status code"
Trap wordsAny option claiming "fetch() throws on a 404" — false, a classic trap.
MCQ trickDistinguish a network error (promise rejects) from an HTTP error status (promise resolves, response.ok is false).
PYQ pattern"What does fetch() return?" → a Promise that resolves to a Response object, regardless of HTTP status.

Elimination technique

Cross out any option claiming fetch auto-parses JSON, or auto-rejects on 4xx/5xx — both are the most common wrong-answer bait.
