Module 02 · Core Intuition

Why This Exists

💡

Imagine one waiter running an entire restaurant. If they had to stand at the kitchen window until every dish finished cooking, the whole restaurant would freeze — no new orders, no refills, nothing.

Instead, the waiter drops off the order and moves on. When the kitchen rings the bell, the waiter comes back to deliver the food. JavaScript's single thread works the same way: async/await lets it fire off slow work (network calls, timers) and keep serving everyone else, then come back exactly when the result is ready.

Module 05 · Deep Fundamentals

Understanding, Not Memorizing

Why was it created?

Chaining many .then() calls for sequential async steps produced deeply nested, hard-to-read "callback hell." Async/await gives that same behavior a flat, linear shape.

What problem does it solve?

It lets JavaScript start slow operations (network, timers, I/O) without blocking the single main thread that also handles clicks, scrolls, and rendering.

What happens internally?

An async function always returns a Promise. Hitting await pauses that function only — not the whole program — until the awaited Promise settles.

How does JS execute it?

The rest of the program keeps running past the async call. Once the awaited value is ready, the paused function's remaining code resumes as if it were synchronous.

Module 07 · Syntax Anatomy

Every Piece, Explained

anatomy.js
async function getUser() {
  const response = await fetch('/api/user');
  return response;
}
  • async Marks this function as asynchronous — it will now always return a Promise, even if you write a plain return statement.
  • function getUser() A normal function declaration — the name and parentheses work exactly like any other function.
  • await Pauses only this function until the Promise on its right settles. Legal only inside a function marked async.
  • fetch('/api/user') Returns a Promise immediately — await is what unwraps it into the actual response.
  • return response Because the function is async, this value is automatically wrapped in a resolved Promise for whoever calls getUser().

  try {

    const orders = await fetchOrders(); // then pause until orders arrive

Module 09 · Behind the Scenes

What the Engine Is Actually Doing

Promise created → state: pending
→
Async operation runs off the main thread (browser/OS level)
→
Operation finishes → resolve() or reject() called
→
State becomes fulfilled or rejected
→
await unwraps the value / try-catch traps the error
⚙️

A Promise can only ever transition once: pending → fulfilled, or pending → rejected. It never flips back, and it never settles twice — that's what makes it a reliable placeholder for a future value.

Module 11 · Common Mistakes

Wrong vs. Correct

❌

Using await in a plain function — throws SyntaxError: await is only valid in async function.

✅

Mark the enclosing function async before using await inside it.

❌

Writing an await with no try/catch — a rejection surfaces as an unhandled promise rejection and silently breaks the flow.

✅

Wrap awaited calls in try { } catch (err) { } so failures are handled gracefully.

❌

await-ing unrelated promises one after another — each one blocks the next, even though they don't depend on each other.

✅

Use Promise.all([...]) to run independent async calls in parallel.
