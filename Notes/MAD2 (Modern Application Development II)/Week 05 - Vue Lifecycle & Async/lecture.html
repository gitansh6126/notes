How a frontend and backend agree to talk to each other — and why that conversation has to be asynchronous.

Backend (data) ⇄ JSON over URL ⇄ Frontend (UI)
1Foundation

Separation of Concerns

Two jobs, deliberately kept apart:

⚙️ Backend

Manages data models — storage, business logic, validation. Has no idea what any screen looks like.

🖥️ Frontend

Manages the UI — layout, interaction, presentation. Has no idea how data is stored.

Analogy A restaurant kitchen (backend) never plates food based on which table's decor it's going to — it just sends out a dish. The waiter (frontend) decides how to present it: which tray, which table, which order to serve. If the kitchen started deciding table layouts, the whole restaurant breaks the moment you redecorate.
📌
Required, by design:
• Backend must never know what the UI looks like
• No direct calls to HTML template rendering from backend logic
• Data output in a neutral format — JSON is the modern default, but not the only option
• Data input via form data or URLs

This split needs two mechanisms to actually work:

Fetch mechanism + Rendering mechanism

(retrieve data from backend)   +   (server-rendered & pushed, or browser pulls & renders)

2Core Problem

Fetch Has to Be Asynchronous

Fetching data depends on things the server (and browser) don't control:

📡 Network latency

Distance, congestion, routing — all unpredictable.

⚠️ Load & disruptions

The server might be slow, or the connection might drop entirely.

🚫
What must never happen: the browser hangs waiting for data that isn't there yet. JavaScript is single-threaded — one blocking call freezes the whole page.
Analogy Ordering coffee: you don't stand frozen at the counter, unable to blink, until your name is called. You order, step aside, and do other things (check your phone, chat) — then react when your name is called. That "step aside and react later" pattern is asynchronous programming.

Asynchronous operation, concretely: start the fetch in the background → keep the page responsive → wait for the result → update the UI when it arrives.

3Mechanism

Events & Callbacks

The most primitive async tool: hand a function to something slow, and say "call me back when you're done."

🐢
let result = doSomething() — blocks the entire JS interpreter until result exists. Nothing else on the page can run. This is the failure mode callbacks solve.
callback-pattern.js
doSomething(successCB, failureCB) {
  let result = doLongComputation();
  if (result) successCB();   // called as a function, later
  else failureCB();
}
Higher-order function: doSomething takes other functions as arguments and decides when to call them, based on a runtime condition — not called imperatively in sequence.
Analogy A DOM onclick handler is a callback you never call yourself. You just tell the browser: "when this click event happens, run this function." It's declarative, not imperative — you describe the trigger, not the execution order.

5Mechanism

Promises

Same idea as callbacks, cleaner syntax — and stronger guarantees.

callback style
doSomething(successCB, failureCB) {
  ...
}
promise style
doSomething()
  .then(successCB, failureCB);
✅
Promises are more than syntax sugar — they add real guarantees: a promise settles exactly once (success or failure, never both, never zero), and .then() calls chain cleanly instead of nesting ("callback hell"). See: MDN — Using Promises
6Concept

Concurrency vs. Parallelism

🔀 Concurrent

Multiple operations are in process at once — but maybe only time-multiplexed, not literally simultaneous.

⚡ Parallel

Multiple operations are physically executing at the same instant, on separate cores/threads.

🔑
Parallel requires concurrent — not vice versa. Async operations bring concurrency; whether the runtime happens to implement that as parallel or time-multiplexed is up to the environment (e.g. web workers and timers can run in parallel).
Analogy One chef switching rapidly between four pans is concurrent (all four dishes are "in progress"). Four chefs, each at their own stove, is parallel. From the diner's seat both can look similar — but only one actually uses more hands.
7Tool

The fetch() API

The built-in, promise-based way to make a URL request async, since ES6.

fetch-example.js
fetch("/api/students")
  .then(response => response.json())
  .then(data => {
    // data is now a real JS object — safe to render
    renderUI(data);
  })
  .catch(err => console.error(err));
Implemented internally using Promise. Built into most modern browsers; polyfills exist for older ones.

10Solve Questions

Quick Self-Check

Q1 · Concept checkseparation of concerns
Why should backend code never call HTML template rendering directly?
Reveal answer
Because it couples the data layer to one specific presentation — the moment the UI changes (new frontend, mobile app, different framework) the backend would need to change too. Neutral data output (e.g. JSON) keeps them independent.
Q2 · Trapconcurrency vs parallelism
True or false: "This code is async, so it must be running in parallel."
Reveal answer
False. Async guarantees concurrency (multiple things "in flight"), not parallelism (multiple things executing at the same physical instant). A single-threaded event loop can be concurrent without ever being parallel.
Q3 · Code readingpromises
doSomething().then(successCB, failureCB) — what guarantee does this have that a plain callback pair doesn't?
Reveal answer
The promise is guaranteed to settle exactly once — either successCB or failureCB fires, never both, never neither — and multiple .then() calls can be chained predictably.

🧠 One-Page Brain — Using APIs

Core Intuition
  • Backend owns data, frontend owns UI — connected only by a neutral data contract (usually JSON).
  • Fetching is unreliable in timing, so it must never block the single JS thread.
Trigger Words
  • "async" → callbacks / promises / fetch
  • "single-threaded" → event loop, blocking
  • "backward compatible" → axios over raw fetch
Common Traps
  • Assuming async = parallel (it only guarantees concurrency)
  • Backend rendering HTML directly — breaks separation of concerns
  • Forgetting fetch() needs an explicit .json() step; axios does it for you
Dependencies
  • Promises underpin fetch()
  • Event loop underpins all of JS's async model
