Vue Lifecycle Hooks

Every Vue component goes through a predictable set of stages — creation, mounting, updating, and destruction. Lifecycle hooks let you run code at each stage, giving you fine-grained control over component behavior.

1 CREATION — beforeCreate & created

The Blueprint Phase

When a component is first initialized, Vue sets up data reactivity, computed properties, watchers, and methods. The creation hooks fire before the component is added to the DOM.

beforeCreate

Called after the component instance is constructed, but before reactive data and events are set up. You cannot access data, computed, or methods yet.

// beforeCreate — nothing is available yet
export default {
  beforeCreate() {
    console.log('data:', this.$data)       // undefined
    console.log('message:', this.message)  // undefined
    this.fetchData()                      // TypeError
  }
}

created

Called after reactive data, computed properties, methods, and watchers are fully initialized. The component is not yet mounted (no DOM access). This is a common place to fetch initial data for non-DOM operations.

export default {
  data() {
    return { message: 'Hello' }
  },
  created() {
    console.log(this.message)              // 'Hello'  ✅ data is available
    this.fetchInitialConfig()              // ✅ methods are ready
    this.$el                             // undefined ❌ no DOM yet
  }
}
Real-World Analogy
beforeCreate is like when you've hired an architect but the blueprints haven't been drawn yet — ask about the kitchen layout and you'll get a blank stare. created is when the blueprints are finalized: all the rooms, plumbing, and wiring are planned on paper, but no construction has started on the lot.
Industry Context
Enterprise apps often use created to initialize analytics trackers, restore user preferences from localStorage, or set up non-DOM integrations (WebSocket connections, logging services). Never use it for DOM-dependent operations.
2 MOUNTING — beforeMount & mounted

Moving Into the DOM

Mounting hooks fire when the component's template is being rendered into the DOM for the first time.

beforeMount

Called right before the component's template is compiled and mounted to the DOM. The render function is about to be called. The virtual DOM exists but the real DOM hasn't been updated yet.

mounted

Called after the component is added to the DOM. All child components are also mounted. You can now access the DOM via this.$el, use $refs, and perform DOM operations. This is the recommended hook for fetching data from an API.

export default {
  data() {
    return { items: [], loading: true }
  },
  async mounted() {
    try {
      const res = await fetch('/api/items')
      this.items = await res.json()
      this.$refs.container.scrollIntoView()  // ✅ DOM is ready
    } finally {
      this.loading = false
    }
  }
}
Real-World Analogy
beforeMount is the furniture delivery truck pulling up to your address — the house is built off-site and about to be placed on the lot. mounted is move-in day: the doors are open, the keys are in your hand, and you can finally arrange the furniture (DOM manipulation) and plug in appliances (start timers, fetch data).
3 UPDATING — beforeUpdate & updated

Renovation Mode

When a reactive property changes, Vue re-renders the component. Update hooks fire before and after the DOM is patched.

beforeUpdate

Called when reactive data has changed but the DOM hasn't been re-rendered yet. Use this to read the current DOM state before Vue patches it, or to manually tweak values.

updated

Called after the DOM has been re-rendered following a data change. Be careful — do not mutate data in this hook (it creates infinite loops). Use $nextTick if you need to wait for the specific DOM update.

export default {
  data() {
    return { search: '', results: [] }
  },
  beforeUpdate() {
    console.log('Scroll position:', this.$el.scrollTop)  // old DOM state
  },
  updated() {
    // this.search = 'new' — DON'T do this! Infinite loop!
    this.$nextTick(() => {
      highlightMatches(this.$el)  // safe after render
    })
  }
}
Common Mistake
Mutating data inside updated hook — this triggers another update, which calls updated again, creating an infinite loop. Always use computed properties or watchers for reactive data transformations instead.
4 DESTRUCTION — beforeDestroy & destroyed

Demolition Day

When a component is removed from the DOM (v-if toggled off, route changes, etc.), destruction hooks fire so you can clean up resources.

beforeDestroy

Called right before the component instance is destroyed. The component is still fully functional — this is your last chance to clean up: remove event listeners, clear timers, unsubscribe from stores, cancel pending API requests.

destroyed

Called after the component instance has been destroyed. All directives, watchers, and child components are unbound and removed. The component is gone from the DOM and memory.

export default {
  data() {
    return { timer: null }
  },
  mounted() {
    this.timer = setInterval(() => {
      console.log('tick')
    }, 1000)
  },
  beforeDestroy() {
    clearInterval(this.timer)          // ✅ crucial cleanup!
    this.cleanupSubscriptions()         // unsubscribe from stores
    this.$refs.video?.pause()          // clean up media
  }
}
Common Mistake
Forgetting cleanup in beforeDestroy — setInterval, addEventListener, WebSocket connections, and RxJS subscriptions all leak memory if not cleaned up. This is the #1 cause of memory leaks in Vue SPAs. Always pair your creation with destruction.
5 KEEP-ALIVE HOOKS — activated & deactivated

Cached Components

When a component is wrapped in <keep-alive>, it's not destroyed when toggled off — it's deactivated and cached. Two additional hooks become available.

<keep-alive>
  <component :is="currentTab" />
</keep-alive>

activated — fires when the component is reactivated from cache. Useful for refreshing data. deactivated — fires when the component is cached (not destroyed). Clean up here instead of beforeDestroy if using keep-alive.

export default {
  activated() {
    this.refreshData()  // refresh when tab becomes active
  },
  deactivated() {
    clearInterval(this.timer)  // cleanup but component stays alive
  }
}
6 ASYNC IN LIFECYCLE HOOKS

Async/Await in Hooks

Lifecycle hooks can be async, but Vue doesn't wait for the promise to resolve before continuing the lifecycle. This is usually fine — your data will arrive and trigger a re-render when it resolves.

export default {
  async created() {
    const config = await fetch('/api/config')
    Object.assign(this, config)  // will trigger re-render when resolved
    // Vue has already mounted by the time this finishes — that's okay!
  }
}
Pro Tip
Use mounted (not created) for async data fetching in browser-rendered apps. In SSR contexts, created is the only hook that runs on the server — so you might fetch there for universal data, but always guard against DOM-dependent code.
7 COMMON MISTAKES

Arrow Functions in Hooks

// ❌ BAD — arrow function doesn't bind to component instance
export default {
  data: () => ({
    count: 0
  }),
  created: () => {
    console.log(this.count)  // undefined — 'this' is window (or undefined in strict mode)
  }
}

// ✅ GOOD — method shorthand binds to the component
export default {
  data() {
    return { count: 0 }
  },
  created() {
    console.log(this.count)  // 0 ✅
  }
}
Arrow Function
this inherits from parent scope (likely window/undefined). Methods and data are inaccessible.
Method Shorthand
this is the Vue component instance. All data, computed, and methods are available.

Forgetting to Clean Up

// ❌ BAD — setInterval lives on after component is destroyed
mounted() { this.timer = setInterval(() => {}, 1000) }
// beforeDestroy never defined — memory leak!

// ✅ GOOD — cleanup in beforeDestroy
beforeDestroy() { clearInterval(this.timer) }
8 LIFECYCLE DIAGRAM
9 INTERVIEW Q&A
Q: What's the difference between created and mounted? When would you use each?
A: created runs after reactive data is initialized but before DOM insertion — use for non-DOM setup (API calls in SSR, init non-DOM libraries). mounted runs after the DOM is rendered — use for DOM-dependent operations, third-party DOM libraries, and fetching data in client-side apps.
Q: Why should you avoid arrow functions in Vue lifecycle hooks?
A: Arrow functions don't have their own this binding — they inherit it from the enclosing lexical scope. In Vue options, arrow functions would bind to the module scope (window or undefined) instead of the component instance, making this.data, this.methods, and this.$el inaccessible.
Q: What happens if you use async on a lifecycle hook?
A: It works fine — the async function returns a promise, but Vue doesn't await it before continuing the lifecycle. The hook fires, starts the async operation, and the component continues mounting. When the promise resolves, data updates trigger a re-render. Just be careful: the component may have been destroyed by the time the promise resolves.
10 QUICK REVISION
beforeCreate — nothing exists yet, no data/methods
created — data/methods ready, no DOM, init non-DOM work
beforeMount — template compiled, not yet in DOM
mounted — DOM ready, fetch data, DOM libraries
beforeUpdate — data changed, DOM not yet patched
updated — DOM re-rendered, avoid mutating data here
beforeDestroy — cleanup timers, listeners, subscriptions
destroyed — component fully torn down, all watchers removed
activated — keep-alive: component reactivated
deactivated — keep-alive: component cached
