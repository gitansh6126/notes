0Overview

What is "the lifecycle"?

The Vue.js lifecycle is the series of stages an app or component passes through — from initialization and creation, to mounting, to reacting to changes, to eventual destruction and removal. Lifecycle hooks are special functions Vue calls automatically at each of these exact stages, giving you a place to hook in your own code.

Analogy Think of a component like a plant. There's a moment the seed is planted (created), a moment it breaks through the soil and becomes visible (mounted), moments it grows new leaves in response to sunlight (updated), and eventually a moment it's uprooted (destroyed). You can't water a plant before it's planted, and you can't prune it after it's gone — the hook has to match the stage.
1Stage

Creation Stage

Initialization — before anything is drawn on screen.

🌱Creation

The instance is being set up internally. Nothing is on the DOM yet.

beforeCreate

Called synchronously immediately after the instance is initialized. Data observation, event watchers, and injections are not yet set up.

✕ component data ✕ DOM
created

Reactivity, computed properties, methods, and watchers are fully initialized. A common place for initial data fetching.

✓ component data ✕ DOM (el)
2Stage

Mounting Stage

DOM interaction — the component becomes visible.

🌿Mounting

The compiled template gets attached to the real, rendered DOM.

beforeMount

Called after the template is compiled but before the virtual DOM is rendered and attached to the real DOM.

mounted

Virtual DOM has been rendered — the real DOM is now accessible. Preferred spot for DOM manipulation or data fetching that needs the rendered template.

✓ real DOM
3Stage

Updating Stage

Reactivity — runs every time reactive data changes, for as long as the component lives.

🔁Updating

Triggered whenever reactive state changes and the virtual DOM needs to be patched.

beforeUpdate

Called when data changes, before the DOM is re-rendered.

updated

Called after the DOM has been updated to reflect the data change.

4Stage

Destruction Stage

Cleanup — your last chance to tidy up before the component is gone.

🍂Destruction

The component is being torn down.

beforeDestroy

Called before the instance is destroyed. Still fully functional — the last opportunity to clean up event listeners, timers, or child components.

✓ still functional
destroyed

Called after destruction. All event listeners and child components are removed; data is no longer accessible.

✕ data
⚠️
Always use beforeDestroy to remove event listeners or clear timers — skipping this is one of the most common sources of memory leaks in single-page applications (SPAs).
5Best Practices

Getting Hooks Right

🚫
Never use arrow functions for lifecycle hooks.
wrong-vs-right.js
// ✕ WRONG — arrow function binds `this` to the surrounding scope
mounted: () => {
  console.log(this.data);  // `this` is NOT the Vue instance
}

// ✓ RIGHT — standard function binds `this` to the Vue instance
mounted() {
  console.log(this.data);  // `this` correctly refers to the component
}
Why it breaks: arrow functions don't have their own this — they inherit it from the enclosing scope. Inside a hook, that means this never becomes the component, so this.data and this.methods are simply unreachable.
PracticeGuidance
Data fetchingValid in both created and mounted — but mounted is often preferred: it guarantees the component is fully ready and plays better with server-side rendering patterns.
async/await in mounted✓ Safe. It does not block initial rendering, because the component is already mounted by the time the async code runs.
CleanupAlways release listeners/timers in beforeDestroy — otherwise they silently leak in SPAs.
Hook syntax✕ Arrow functions    ✓ Standard function syntax
Analogy created vs mounted for data fetching is like ordering food to go vs. to a set table. You can hand someone their food the moment it's ready (created) — but if the table (DOM) isn't set yet, you can't do anything with it visually until mounted, when the table is ready to receive it.
6Concept

Component Lifecycle & Nesting

Every Vue component follows the same lifecycle as the root app. A nested component runs its own full initialization and mounting process — parent and child each have their own independent set of hooks, executing in a specific order during startup and shutdown.

Parent: beforeCreate → created → beforeMount
↓ mounts child before finishing its own mount ↓
Child: beforeCreate → created → beforeMount → mounted
↓ child is ready, parent finishes ↓
Parent: mounted
💡
On teardown, order reverses: children are destroyed before their parent finishes destroying — mirroring how they were mounted last but must clean up first.
7Solve Questions

Quick Self-Check

Q1 · Trapthis-binding
Why does mounted: () => { this.loadData() } silently fail?
Reveal answer
It's an arrow function, so this is inherited from the surrounding (non-Vue) scope instead of being bound to the component instance — this.loadData is not the method you think it is.
Q2 · Concept checkcreated vs mounted
You need to measure the pixel width of a rendered <div> in your component. Which hook do you use, and why?
Reveal answer
mounted — it's the first hook where the real DOM exists. In created, the template hasn't been compiled or attached yet, so there's nothing to measure.
Q3 · Trapcleanup
A component adds a window.addEventListener('resize', ...) in mounted but never removes it. What breaks, and where should the fix go?
Reveal answer
Memory leak: the listener (and its closure over the component) survives after the component is destroyed, since nothing else ever removes it. Fix: call window.removeEventListener in beforeDestroy, while the instance is still functional.
