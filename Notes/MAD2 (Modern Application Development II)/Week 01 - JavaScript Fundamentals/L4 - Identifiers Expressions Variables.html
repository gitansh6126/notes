Identifiers, Expressions & Variables

A deep look at how JavaScript handles variable declarations (var vs let vs const), hoisting, the Temporal Dead Zone, scoping rules, and the difference between expressions and statements.

1Identifiers

Naming Your Variables

An identifier is the name you give to variables, functions, classes, etc. JavaScript has strict naming rules and strong conventions.

Rules

  • Must start with: letter (a-z, A-Z), underscore (_), or dollar sign ($)
  • Following characters: letters, digits, _, $
  • Case-sensitive: myVar ≠ myvar ≠ MYVAR
  • Cannot use reserved keywords: if, for, class, let, const, function, return, etc.
  • Unicode letters are allowed: const π = 3.14; (valid, though uncommon)

Conventions (not enforced by JS)

// camelCase — standard for variables & functions
const userName = "Alice";
function getUserData() {}

// PascalCase — for classes & constructors
class UserProfile {}

// UPPER_SNAKE_CASE — for constants (true constants)
const MAX_SIZE = 100;
const API_KEY  = "abc123";

// _prefix — "private" convention (not truly private)
const _internalValue = 42;
2Expressions vs Statements

The Two Categories of Code

Every piece of JavaScript code is either an expression (produces a value) or a statement (performs an action).
Expression
Produces a value. Can be assigned to a variable, logged, or passed as argument. 42, "hello", 5 + 3, fn(), a ? b : c
Statement
Performs an action. Cannot be used where a value is expected. if (x) {}, for (;;) {}, return x;, let x = 5;
// Expressions (can go anywhere a value is expected)
5 + 3;              // expression → value 8
Math.random();       // expression → random number
greet("Alice");     // expression → return value

// Statements (do things)
let x = 5;            // declaration statement
if (x > 0) {}         // if statement
for (let i = 0; i < 10; i++) {} // for statement

// Special: expression statements (expressions used as statements)
console.log("hi");  // expression statement
3var vs let vs const

The Three Declaration Keywords

ES6 introduced let and const to fix the design flaws of var. Modern JavaScript strongly prefers const by default, let only when reassignment is needed, and var never.

var x = "old";       // don't use

const arr = [1, 2, 3];

4Hoisting & Temporal Dead Zone

The Most Confusing JS Concept — Explained

Hoisting is JavaScript's behavior of moving declarations to the top of their scope before code execution. But when is the variable initialized? That's the key difference.
undefined Can access before declaration (just gets undefined) let / const Hoisted but NOT initialized → Temporal Dead Zone Access before declaration → ReferenceError! 
// var hoisting
console.log(a);     // undefined (not ReferenceError!)
var a = 5;

// let hoisting + TDZ
console.log(b);     // ❌ ReferenceError: Cannot access 'b' before initialization
let b = 10;

// TDZ is the zone from the start of the block until the declaration
{
  // TDZ starts here for 'c'
  const fn = () => console.log(c);
  // TDZ still active
  let c = 3;
  // TDZ ends — c is now safe to access
  fn();    // ✅ works because c is accessed after declaration
}
Real-World Analogy
Think of var as a reserved table at a restaurant — you know the table exists and it's there (undefined) even before you arrive. let/const are like a table that exists on the reservation list (hoisted) but the restaurant hasn't prepared it yet — if you try to sit before the declaration, the host says "your table isn't ready" (ReferenceError). The TDZ is the time between when you enter the restaurant and when your table is prepared.
5Scoping Deep Dive

Global, Function, Block Scope

Scope determines where a variable is accessible. JavaScript has three levels of scope, and understanding them is critical. 
const global = "I'm everywhere";  // global scope

function outer() {
  const funcScope = "Inside outer only";  // function scope

  if (true) {
    const blockScope = "Inside if block";  // block scope (const/let)
    var leaked = "I escape the block!";    // var is function-scoped!
  }

  console.log(leaked);     // ✅ "I escape the block!" (var leaks)
  console.log(blockScope); // ❌ ReferenceError (block-scoped)
}
Best Practice
Use const by default for every variable. Only use let when you know the variable needs to be reassigned. Never use var. This eliminates an entire category of scoping bugs.

The TC39 committee introduced let and const in ES6 specifically because the function-scoping and hoisting behavior of var was the #1 source of developer confusion and bugs. 7Common Mistakes
Common Mistake
Using const for objects/arrays and thinking they're immutable. const prevents reassignment, not mutation. const arr = [1,2,3]; arr.push(4); works fine. Use Object.freeze() for shallow immutability.
Common Mistake
Redeclaring variables with let/const in the same scope. Unlike var, let/const throw a SyntaxError if redeclared. This is a feature, not a bug — it catches accidental duplicate declarations.

9Interview Q&A
Q: What is the Temporal Dead Zone (TDZ)?
A: The TDZ is the time between entering a scope (where a variable is hoisted) and the variable's declaration. Accessing the variable during TDZ throws a ReferenceError. Only let and const have a TDZ — var does not (it's initialized as undefined).
Q: Can you redeclare a let variable? What about a const?
A: Neither let nor const can be redeclared in the same scope. Both throw a SyntaxError. var can be redeclared silently, which is one reason it's considered dangerous.
Q: What's the difference between an expression and a statement?
A: An expression produces a value (e.g., 5 + 3, fn()). A statement performs an action (e.g., if, for, return). Expressions can be used anywhere a value is expected; statements cannot. A function declaration is a statement; a function expression is an expression.
Q: What happens when you assign to a const property?
A: const prevents reassignment of the binding (variable), not mutation of the value. If the value is an object, you can modify its properties. Use Object.freeze() for shallow immutability or TypeScript's readonly modifier.
