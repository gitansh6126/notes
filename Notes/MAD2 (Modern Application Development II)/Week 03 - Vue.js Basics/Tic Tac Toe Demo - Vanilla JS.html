Overview

This demo builds Tic Tac Toe entirely with vanilla JavaScript (no frameworks). The browser holds the game state, handles clicks, updates the DOM directly, and never talks to a server during gameplay. This is client-side rendering in its purest form.

🎮 How It Works

  1. The HTML has an empty 3×3 grid container and a status display.
  2. JavaScript creates the board in memory as an array (['','','', ...]).
  3. When a player clicks a cell, JS checks if it is empty, updates the array, re-renders that cell.
  4. After every move, JS checks for a winner or draw using the in-memory array.
  5. The DOM is updated in place — no page reloads, no server round-trips.
ℹ
Core Idea

The browser is now a smart client. It stores state, runs logic, and updates its own display. The server is only needed if we want to persist or share the game across devices. Every interaction is instant because it all happens in the same process.

The Document Object Model (DOM)

The DOM is a tree-like representation of an HTML page that JavaScript can read and modify. Every HTML element becomes a node in this tree.

🌳 What is the DOM?

When the browser loads an HTML page, it parses the HTML text and builds a DOM tree in memory:

HTML that produces the treeHTML
<h1>Tic Tac Toe</h1>
<p id="status">Player X's turn</p>
<div class="board">
  <button class="cell" data-index="0"></button>
  <button class="cell" data-index="1"></button>
  <button class="cell" data-index="2"></button>
  ...
</div>
💡
Real-Life Analogy: A Whiteboard

The DOM is like a whiteboard. The HTML is the initial drawing. JavaScript is your hand — it can erase parts, write new text, change colours, all without removing the whiteboard from the wall. DOM manipulation is just reaching out and editing parts of the board.

Event Handling

Events are how JavaScript responds to user interactions. Every click, keypress, or form submission generates an event object that JS can listen for.

🔔 How Events Work in the Browser

↓
Browser creates a click event
↓

↓
Target Phase (the clicked element)
↓
Bubbling Phase (target → root)
↓
Your event handler runs!

📝 Event Handling in Tic Tac Toe

Three ways to handle clicksJavaScript
// Method 1: Add listener to each cell (most common)
document.querySelectorAll('.cell').forEach((cell, i) => {
  cell.addEventListener('click', () => handleMove(i));
});

// Method 2: Event delegation on the board container
document.getElementById('board').addEventListener('click', (e) => {
  const cell = e.target.closest('.cell');
  if (cell) handleMove(+cell.dataset.index);
});

// Method 3: Inline onclick (old way, avoid in production)
// <button onclick="handleMove(0)"></button>

function handleMove(index) {
  console.log('Cell clicked:', index);
}
💡
Event Delegation

Instead of attaching 9 separate listeners, attach one listener to the parent #board div. When any cell is clicked, the event bubbles up to the parent, and we use event.target to determine which cell was clicked. This is more efficient and works for dynamically added cells.

05

Browser Rendering

When your JavaScript changes the DOM, the browser does not instantly repaint. It follows a carefully optimised pipeline.

📈 The Critical Rendering Path

JavaScript modifies DOM
↓
Style Calculation (what CSS applies?)
↓
Layout (reflow) — calculate positions
↓
Paint — fill pixels
↓
Compositing — layer stacking
↓
Screen displays updated pixels

Tic Tac Toe only modifies textContent and adds/removes CSS classes — these are cheap operations that typically only trigger paint (not layout/reflow), because the grid dimensions do not change.

⚠
Performance Trap

Reading element.offsetHeight or getComputedStyle() forces the browser to do an immediate synchronous reflow, flushing the render queue. If you read such values inside a loop that also writes to the DOM, you force repeated reflows and kill performance. Batch your reads separately from writes.

06

JavaScript Game Logic

Every piece of game logic runs in the browser: board state, turn tracking, win detection, draw detection.

🎮 Complete Vanilla JS Implementation

Full game logic (no frameworks)JavaScript
const cells = document.querySelectorAll('.cell');
const status = document.getElementById('status');

// State (no framework!)
let board = ['', '', '', '', '', '', '', '', ''];
let turn = 'X';
let gameOver = false;

// Win patterns: 3 rows, 3 cols, 2 diagonals
const wins = [
  [0,1,2], [3,4,5], [6,7,8], // rows
  [0,3,6], [1,4,7], [2,5,8], // cols
  [0,4,8], [2,4,6]  // diagonals
];

function handleClick(index) {
  if (gameOver || board[index]) return;

  board[index] = turn;                    // Update state
  cells[index].textContent = turn;       // Update DOM
  cells[index].classList.add('taken');

  if (checkWin(turn)) {
    status.textContent = `Player ${turn} wins!`;
    gameOver = true;
  } else if (board.every(c => c)) {
    status.textContent = "It's a draw!";
    gameOver = true;
  } else {
    turn = turn === 'X' ? 'O' : 'X';
    status.textContent = `Player ${turn}'s turn`;
  }
}

function checkWin(player) {
  return wins.some(pattern =>
    pattern.every(i => board[i] === player)
  );
}

// Attach events
cells.forEach((cell, i) =>
  cell.addEventListener('click', () => handleClick(i))
);

↓

↓

↓

↓

State Handling Without Frameworks

In vanilla JS, you manage state manually. There is no reactive system, no watchers, no virtual DOM. You mutate variables and explicitly update the DOM yourself.

📊 How State Works in Vanilla JS

The game state is just a plain JavaScript variable:

State is just dataJavaScript
// This is your entire "state management system"
let board    = ['', '', '', '', '', '', '', '', ''];
let turn     = 'X';
let gameOver = false;

// You update the DOM YOURSELF after every state change
board[i] = turn;
cells[i].textContent = turn;  // <-- manual DOM sync

🧰 State Flow in Vanilla JS vs Frameworks

AspectVanilla JSVue.js
State storagePlain variablesdata() wrapped in reactive proxy
DOM updateYou call .textContent = yourselfAutomatic via reactivity system
Change detectionNone (you must remember to update)Automatic (getter/setter interception)
Re-render scopeYou choose exactly what to changeFramework diffs and patches the DOM
BoilerplateMinimalMore (component structure, directives)
Mental model"I change the DOM when state changes""State changes, DOM follows automatically"
⚠
The Manual Sync Problem

In vanilla JS, you are responsible for keeping the DOM in sync with the state. Forget to update one place? You get a visual bug. This is why frameworks exist — they automate the state → DOM mapping. In a larger app, manual sync becomes error-prone.

Comparison: Vanilla JS vs Flask SSR

The same Tic Tac Toe game built two ways. Understanding the differences reveals why we use frameworks.

📝 Architecture Comparison

✅ Vanilla JS Advantages

  • Instant updates — no network delay, no page reload
  • Offline capable — works without internet after initial load
  • Lower server cost — server just serves static files
  • Rich interactivity — animations, hover effects, real-time feedback
  • Reduced bandwidth — only initial HTML/JS sent once
  • No framework lock-in — you own every line of code

❌ Vanilla JS Limitations

  • Manual state → DOM sync — easy to forget updates
  • No reactivity — re-rendering is your responsibility
  • Harder to scale — complex apps become spaghetti code
  • No SEO — search engines see an empty HTML shell
  • Security — game logic is visible in browser dev tools
  • No state persistence — refreshing the page loses the game

Common Bugs & Mistakes

✖ Bug 1: Forgetting to Update the DOM After State Change

WrongJavaScript
function handleClick(i) {
  board[i] = turn;        // State updated
  turn = turn === 'X' ? 'O' : 'X';
  // DOM not updated! Cell still shows blank.
}

Fix: Always update the corresponding DOM element right after mutating state: cells[i].textContent = board[i].

✖ Bug 2: Using == Instead of === in Win Check

WrongJavaScript
// board[i] is '', which is falsy but == null
if (board[a] == board[b] == board[c]) { ... }  // BAD

Fix: Use strict comparison: board[a] === board[b] && board[b] === board[c] && board[a] !== ''.

✖ Bug 3: Event Listener Inside a Loop (Closure Problem)

WrongJavaScript
for (var i = 0; i < 9; i++) {
  cells[i].addEventListener('click', function() {
    handleClick(i);  // i is always 9 when clicked!
  });
}

Fix: Use let (block scoped) instead of var, or use forEach with its own scope, or use dataset attributes.

✖ Bug 4: Not Preventing Clicks on Occupied Cells

Without the guard if (board[i]) return, players can overwrite existing marks. Always check if (board[i] !== '') before processing a move.

✖ Bug 5: Checking Win Before Updating the Board

Wrong orderJavaScript
if (checkWin(turn)) { // <-- board[i] is still empty!
  console.log('win');
}
board[i] = turn;  // <-- should be BEFORE check

Fix: Always update board[i] = turn before calling checkWin().

Updating the UI

There are multiple ways to update the UI in vanilla JS. Each has different performance and clarity trade-offs.

📝 UI Update Strategies

StrategyCode ExampleBest For
Direct property set el.textContent = 'X' Simple text updates (Tic Tac Toe cells)
Class toggling el.classList.add('taken') Visual state changes (disabled, active, error)
InnerHTML (caution) el.innerHTML = '<span>X</span>' When you need to replace a chunk of HTML
Style property el.style.background = 'green' Inline style changes (animations, highlights)
Re-render entire board Loop over array, rebuild all cells When state changes substantially
Re-render approach (alternative to per-cell update)JavaScript
function renderBoard() {
  // Blindly re-render ALL cells from state
  cells.forEach((cell, i) => {
    cell.textContent = board[i];
    cell.classList.toggle('taken', !!board[i]);
  });
}

// Now handleClick only needs to update state:
function handleClick(i) {
  if (gameOver || board[i]) return;
  board[i] = turn;
  renderBoard();  // <-- One call to sync all DOM
  checkGameState();
}

🎯 Per-Cell vs Full Re-render

ApproachProsCons
Per-cell updateMinimal DOM changes, fastEach handler must know exactly what changed
Full re-renderSimple, consistent, hard to forget somethingUpdates all 9 cells even if only 1 changed

For a 9-cell board, both approaches are instant. For a large list (1000+ items), per-cell updates matter.

Interview & Exam Questions

Q1: What is the DOM and how does JavaScript interact with it?+

The DOM (Document Object Model) is a tree structure representing an HTML document. JavaScript interacts with it via the document object using methods like getElementById, querySelector, createElement, and properties like textContent, innerHTML, and classList. The DOM is live — changes in JS immediately reflect on screen.

Q2: How does the browser rendering pipeline work when you change the DOM?+

1. JavaScript modifies the DOM (text, classes, attributes). 2. Style calculation determines which CSS rules apply. 3. Layout (reflow) computes positions and sizes. 4. Paint fills pixels. 5. Compositing layers them. Changing textContent typically only triggers paint; changing width/height triggers layout which is more expensive.

Q3: What is event delegation and why use it?+

Event delegation means attaching a single event listener to a parent element and using event.target to determine which child was interacted with. Benefits: (1) fewer listeners (one vs nine), (2) works for dynamically added children, (3) less memory. For Tic Tac Toe, you attach one listener to the board div instead of nine to individual cells.

Q4: How does vanilla JavaScript manage state compared to Vue.js?+

Vanilla JS: State is stored in plain variables. You must manually update the DOM whenever state changes. There is no automatic tracking. Vue.js: State lives in data() which Vue wraps in reactive proxies. When state changes, Vue automatically determines which DOM nodes to update and patches them. Vue eliminates the manual sync step.

Q5: What is the difference between textContent and innerHTML?+

textContent sets or gets only the text (HTML tags are escaped). innerHTML parses the string as HTML. Use textContent for text-only updates (it is faster and prevents XSS). Use innerHTML only when you need to insert structured HTML from a trusted source.

Q6: Why does refreshing the page reset the Tic Tac Toe game in Vanilla JS?+

Because the game state is stored in JavaScript variables (board, turn, gameOver) that live in memory. When you refresh, the browser destroys the page's memory and reloads the HTML/JS from scratch. The variables are reinitialised to their starting values. To persist across refreshes, you would need localStorage or a backend server.

Q7: What is the difference between let, const, and var?+

const cannot be reassigned (but objects/arrays can be mutated). let is block-scoped and can be reassigned. var is function-scoped (not block-scoped) and hoisted. In modern JS, prefer const by default and let when you need reassignment.

Q8: How does the some() and every() array methods work in the win check?+

wins.some(pattern => ...) returns true if any winning pattern matches. Inside, pattern.every(i => board[i] === player) returns true if every cell in that pattern belongs to the current player. Together: "Is there at least one pattern where all three cells belong to this player?"

Q9: What is event bubbling and how does it affect the Tic Tac Toe click handler?+

Event bubbling means when an event fires on an element, it first runs handlers on that element, then on its parent, then its grandparent, all the way up to document. In Tic Tac Toe, a click on a cell bubbles up to the board div. If you have a listener on the board, it will catch all cell clicks via event.target. You can stop bubbling with event.stopPropagation() if needed.

Q10: Compare the developer experience of building Tic Tac Toe in Flask vs Vanilla JS.+

Flask: You write Python logic, Jinja2 templates, and handle form submissions. Every feature requires a round-trip. Easier for beginners who know Python. No need to understand the DOM. Vanilla JS: You write JavaScript that directly manipulates the DOM. Instant feedback, but you must understand events, the DOM API, closures, and state management. Steeper initial learning, but more representative of how modern frontend works.

Key Takeaways

  • The DOM is a live tree — every HTML element is a node you can read and modify with JavaScript.
  • DOM manipulation means using methods like querySelector, textContent, and classList to update the page.
  • Events (click, submit, etc.) are the way JavaScript responds to user actions. Event delegation attaches one listener to a parent rather than many to children.
  • Browser rendering follows a pipeline: Style → Layout → Paint → Composite. Different DOM operations trigger different stages.
  • State in vanilla JS is just variables. You must manually update the DOM when state changes — there is no reactivity.
  • Vanilla JS gives instant feedback with no page reloads, but state is lost on refresh unless you use localStorage or a backend.
  • Frameworks (Vue, React) automate the state → DOM sync. Understanding the vanilla way helps you appreciate what frameworks do.
  • Common bugs include: forgetting to update the DOM, incorrect comparison operators, closure issues in loops, and wrong order of operations.
  • The same patterns — select elements, attach listeners, update content — are used in every JavaScript application, from Tic Tac Toe to Google Docs.

Summary

📖 Lecture at a Glance

The Vanilla JavaScript Tic Tac Toe demo is a client-side rendered application where the browser handles everything: state, logic, and display. No server is involved during gameplay.

  • The DOM is a tree of HTML nodes that JavaScript reads and modifies.
  • DOM manipulation uses querySelector, textContent, classList, and style to update the page.
  • Event handling via addEventListener lets JS respond to clicks. Event delegation is the efficient pattern.
  • The browser rendering pipeline (Style → Layout → Paint → Composite) determines how expensive a DOM change is.
  • Game state is held in plain JS variables. You must sync state changes to the DOM manually.
  • Vanilla JS gives instant, offline-capable interactions but lacks reactivity, SEO, and persistence.
  • Compared to Flask SSR, the difference is where logic lives (server vs browser) and how the page updates (reload vs in-place).
  • Frameworks exist to automate the manual sync. Understanding vanilla JS first makes you a better framework developer.

Glossary

DOMDocument Object Model — a tree representation of HTML that JavaScript can read and modify.
DOM ManipulationUsing JavaScript to create, read, update, or delete elements in the DOM tree.
EventA signal fired by the browser (click, keypress, submit) that JavaScript can listen for and respond to.
Event DelegationAttaching a single listener to a parent element and using event.target to handle events from children.
Event BubblingThe phase where an event travels from the target element up to the document root.
Reflow (Layout)The browser recalculates element positions and sizes. Expensive, triggered by layout changes.
RepaintThe browser fills pixels without changing layout. Cheaper than reflow.
StateThe data that determines what the UI displays at any moment (board array, turn, gameOver).
Vanilla JavaScriptPlain JavaScript without any libraries, frameworks, or transpilers.
Client-Side RenderingThe browser runs JavaScript to generate and update the UI, rather than receiving HTML from a server.
NodeListAn array-like collection returned by querySelectorAll. Convert to array with Array.from() if needed.
ClosureA function that retains access to variables from its outer scope even after that scope has exited.
