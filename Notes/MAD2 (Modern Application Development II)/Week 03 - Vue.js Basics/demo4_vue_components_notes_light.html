Core idea — split UI into reusable self-contained pieces

<tictactoe-game> ← root: owns all state (board, turn) └─ <game-board :board="board" @move="move"> ← receives board via prop └─ <game-cell v-for="..." :cell="cell" :index="i"> ← one per square

Props down, events up — the one rule

Parent owns state. Child displays it. Child emits events. Parent decides.

AnalogyRMS head office sets the curriculum (state). Each centre receives it as a handout (props) and delivers it. If a centre wants to change something, they file a request (emit). Head office decides and updates the handout. Centres never rewrite the curriculum themselves.
Real use — CubedekhoA <product-card :product="p"> component. Write once, reuse 30 times via v-for. Each card manages its own hover state but product data comes from the parent. Fix a bug in one card file — fixed everywhere instantly.

Complete code — all 3 components annotated

game-cell — one squaresmallest unit
Vue.component('game-cell', {register globally as custom tag
props: ['cell', 'index'],data IN from parent — read only
template: `
<button
:disabled="cell !== ''"disable if square already taken
@click="$emit('click-cell', index)"send index UP to parent
>{{ cell || '_' }}</button>display prop value reactively
`
})
game-board — grid of 9 cellsmiddle layer
Vue.component('game-board', {
props: ['board'],receives board[] array from root
template: `<div class="board">
<game-celluse child component as tag
v-for="(cell, i) in board"loop 9 times
:key="i" :cell="cell" :index="i"pass props DOWN to cell
@click-cell="$emit('move', $event)"bubble event further UP to root
/></div>
`
})
tictactoe-game — root, owns all stateparent
Vue.component('tictactoe-game', {
data() { return {
board: Array(9).fill(''),ALL state lives here — single truth
turn: 'X'
}},
computed: {
winner() {auto-recalculates when board changes
const wins=[[0,1,2],[3,4,5],[6,7,8],[0,3,6],[1,4,7],[2,5,8],[0,4,8],[2,4,6]]
for (const [a,b,c] of wins) {destructuring — W2_L2 in real use
if (this.board[a] && this.board[a]===this.board[b] && this.board[b]===this.board[c])
return this.board[a]
} return null
}
},
methods: {
move(i) {called when game-board emits 'move'
if (this.winner || this.board[i]) returnguard against invalid moves
this.board.splice(i, 1, this.turn)splice — Vue 2 reactivity requirement
this.turn = this.turn==='X' ? 'O' : 'X'toggle player — Vue re-renders
},
reset() {
this.board = Array(9).fill('')reassign triggers full re-render
this.turn = 'X'
}
},
template: `<div>
<p>{{ winner ? 'Winner: '+winner : 'Turn: '+turn }}</p>computed reactive in template
<game-board :board="board" @move="move"/>props down, events up
<button @click="reset">Reset</button>
</div>`
})

Why this is the right architecture

Components give you isolation, reuse, and debuggability

Fix once → fixed everywhere. Bugs stay local. State has one owner.

ReuseWrite <game-cell> once. It renders in all 9 squares via v-for. Add a hover animation to the cell — all 9 squares get it. In vanilla JS you'd touch 9 elements manually.
DebuggabilityA display bug in a cell is isolated to game-cell. A logic bug in move detection is isolated to tictactoe-game. No component can break another's internal state.
This 4-demo arc is the entire history of frontend in miniature: server rendering → client scripting → reactive framework → component architecture. Every production app today uses the component model from Demo 4.
