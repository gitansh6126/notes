01

Overview — Tic Tac Toe on Flask

This demo builds a multi-player Tic Tac Toe game where every move is handled entirely on the server. The browser sends a move, the server updates the game state, re-renders the entire board as HTML, and sends it back. Every click triggers a full page reload. This is classic server-side rendering (SSR).

🎮 How the Game Works

  1. Two players share the same browser (alternating turns).
  2. A player clicks on an empty cell — the browser submits a POST request to Flask.
  3. Flask receives the request, finds which cell was clicked, updates the board state, checks for a winner.
  4. Flask renders a new HTML page with the updated board and sends it back.
  5. The browser loads the fresh HTML — the page reloads completely every turn.
ℹ
Core Idea

The server owns all the logic — the board state, win detection, turn management. The browser is just a dumb display that sends clicks and renders HTML. Every interaction requires a round-trip to the server.

02

Server-Side Rendering (SSR)

Server-Side Rendering means the server generates the final HTML and sends it to the browser. The browser simply displays what it receives. This is how websites worked before modern JavaScript frameworks.

📡 SSR Explained

In SSR, the server does all the heavy lifting:

  • It stores the application state (board, scores, whose turn it is).
  • It contains all the game logic (validate move, check win, switch turns).
  • It generates HTML (using Jinja2 templates) and sends it in the HTTP response.
  • The browser just renders what it gets. No JavaScript logic runs on the client.

📈 SSR Flow Diagram

⚠
Key Trade-off

Every user action requires a full page reload. The browser flashes white, fetches a new page, and re-renders everything. This feels slow and clunky compared to modern SPAs, but it is simple to build and the server has full control over the state.

Flask app skeleton for Tic Tac Toe Python 
from flask import Flask, render_template, request, session

app = Flask(__name__)
app.secret_key = 'your-secret-key-here'


# Route 1: Show the board
@app.route('/')
def index():
    if 'board' not in session:
        session['board'] = [''] * 9
        session['turn'] = 'X'
    return render_template('board.html', board=session['board'],
                           turn=session['turn'])


# Route 2: Handle a move
@app.route('/move', methods=['POST'])
def move():
    cell = int(request.form['cell'])
    board = session['board']
    turn = session['turn']

    if board[cell] == '':
        board[cell] = turn
        turn = 'O' if turn == 'X' else 'X'

    session['board'] = board
    session['turn'] = turn
    return render_template('board.html', board=board, turn=turn)
💡
Why Session?

HTTP is stateless — Flask does not remember previous requests. The session object (backed by a signed cookie) lets us persist the board state across requests without a database. The browser sends the session cookie with every request; Flask decrypts it to recover the board.

Request-Response Cycle (Detailed)

Every time a player clicks a cell, a complete HTTP request-response cycle executes. Here is the exact sequence of events.

🔁 Complete Cycle for One Move

⏱ Timeline of a Single Move

StepWhat HappensTime Cost
1User clicks a cell<1 ms
2Browser serialises form data<1 ms
3HTTP POST request travels to server10-100 ms (network)
4Flask matches URL to route<1 ms
5View function executes (update board, check win)<5 ms
6Jinja2 renders HTML template1-5 ms
7HTTP response travels back10-100 ms (network)
8Browser parses HTML, re-renders10-50 ms

Total: roughly 30-250 ms per move. Most of the time is network latency, not server processing.

⚠
Why This Feels Slow

Every move requires a round-trip to the server. The browser shows a blank white page during reload. For a game with many rapid clicks, the delay and visual flash are annoying. This is why modern apps use client-side rendering instead.

HTTP Communication in Detail

Every interaction between the browser and Flask happens over HTTP. Understanding the request and response structure is essential.

📥 The HTTP Request (Browser → Server)

When a player clicks cell 3, the browser sends this:

HTTP POST requestHTTP
POST /move HTTP/1.1
Host: localhost:5000
Content-Type: application/x-www-form-urlencoded
Cookie: session=eyJib2FyZCI6WyIiLCIiLCIiLCIiLCIiLCIiLCIiLCIiLCIiXX0

cell=3
  • Method: POST — sends data to the server (changes state)
  • Path: /move — matches the Flask route
  • Headers: Content-Type tells Flask how to parse the body; Cookie carries the session.
  • Body: cell=3 — the clicked cell index.

📤 The HTTP Response (Server → Browser)

Flask sends back a full HTML document:

Content-Type: text/html; charset=utf-8

<html>

<body>

    <table class="board">
      <tr>
        <td><button type="submit" name="cell" value="0">X</button></td>
        <td><button type="submit" name="cell" value="1"> </button></td>
        <td><button type="submit" name="cell" value="2"> </button></td>
      </tr>
      ...
    </table>

  • Status: 200 OK — request succeeded.
  • Headers: Content-Type: text/html tells the browser to render as HTML.
  • Body: The completely rendered HTML page with the updated board.
  • Set-Cookie: The updated session cookie (with the new board state).
ℹ
Stateless Protocol Reminder

HTTP itself does not remember anything. Each request is independent. The session cookie is the bridge — it carries the board state to the server so Flask knows what happened in previous requests. Without it, every move would start a fresh game.

06

Why Every Move Reloads the Page

In the Flask Tic Tac Toe demo, clicking a cell causes a full page reload. This is not a bug — it is a direct consequence of the architecture.

📱 The Mechanism

The board is rendered as an HTML <form>. Each cell is a <button type="submit">.

HTML board template (Jinja2)HTML
<!-- board.html -->
<form method="POST" action="/move">
  <table class="board">
    {% for i in range(9) %}
      <td>
        <button type="submit"
                name="cell" value="{{ i }}">
          {{ board[i] or '&nbsp;' }}
        </button>
      </td>
    {% endfor %}
  </table>
</form>

When a user clicks any button inside the form, the browser:

  1. Serialises the form data (cell=N).
  2. Navigates away from the current page.
  3. Sends a POST request to /move.
  4. Replaces the entire page with the server's HTML response.
💡
Real-Life Analogy: A Letter

Imagine playing chess by mail. You mail a letter saying "I moved my pawn to e4". Your opponent receives it, updates the board on their end, and mails you back the new board state. You cannot see the board while waiting for the reply. The form submission is like mailing a letter — you have to wait for the reply to see the updated state.

07

Advantages & Disadvantages

✅ Advantages

  • Simple to build — just Python and HTML templates, no JavaScript needed.
  • State is secure — game logic and data live on the server; users cannot cheat by inspecting browser code.
  • Works without JavaScript — the game functions even if JS is disabled in the browser.
  • SEO-friendly — search engines get fully rendered HTML directly.
  • Great for learning — makes the HTTP request-response cycle visible and concrete.
  • No client-side framework needed — reduces bundle size and complexity.

❌ Disadvantages

  • Full page reload every move — causes visual flash and perceived slowness.
  • More bandwidth — sending the entire HTML page for every interaction is wasteful.
  • Server load — every click hits the server, even trivial UI updates.
  • Limited interactivity — no animations, hover effects, or instant feedback.
  • No offline support — the game requires a network connection for every move.
  • Scaling challenges — each user's state lives in the session, which can complicate horizontal scaling.

📦 Flask Session Internals

The Flask session is implemented as a signed cookie (not stored on the server).

How session worksPython
# Flask does this automatically:

# 1. When you set session['board'] = [...], Flask serialises it to JSON
# 2. Signs it with app.secret_key to prevent tampering
# 3. Sets a Set-Cookie header with the signed data

# On the next request:
# 1. Browser sends the cookie back
# 2. Flask verifies the signature
# 3. Decrypts it and populates session dict

# This is why secret_key must be strong!
app.secret_key = 'a1b2c3d4-e5f6-7890-very-secure-key'
⚠
Security Note

While the session is signed (tamper-proof), it is not encrypted by default. The user can decode the base64 content and see the board state. Do not put sensitive data (passwords, credit cards) in the default Flask session. Use server-side sessions for sensitive data.

Flask SSR vs JavaScript Approach

The same Tic Tac Toe game can be built with client-side JavaScript. Understanding the differences clarifies why frameworks like Vue.js exist.

🧰 Same Game, Two Architectures

📝 Flask Version (SSR)

app.pyPython
from flask import Flask, render_template, request, session

app = Flask(__name__)
app.secret_key = 'secret'

@app.route('/')
def index():
    return render_template('board.html',
        board=session.get('board', ['']*9),
        turn=session.get('turn', 'X'))

@app.route('/move', methods=['POST'])
def move():
    cell = int(request.form['cell'])
    board = session['board']
    if board[cell] == '':
        board[cell] = session['turn']
        session['turn'] = 'O' if session['turn'] == 'X' else 'X'
        session['board'] = board
    return render_template('board.html', board=board, turn=session['turn'])

📝 Vue.js Version (Client-Side)

App.vueVue
<template>
  <div>
    <div v-for="(cell, i) in board" :key="i"
         @click="makeMove(i)" class="cell">
      {{ cell }}
    </div>
  </div>
</template>

<script>
export default {
  data() {
    return {
      board: ['', '', '', '', '', '', '', '', ''],
      turn: 'X'
    }
  },
  methods: {
    makeMove(i) {
      if (this.board[i]) return;
      this.board[i] = this.turn;
      this.turn = this.turn === 'X' ? 'O' : 'X';
      // Optional: sync to server via API
    }
  }
}
</script>

Key difference: In Vue, clicking a cell updates the board instantly in the browser. No page reload, no HTTP round-trip. The server is only contacted if you need to persist or share the state.

Real-World Examples

The patterns in this demo appear everywhere in production web applications.

🏠 Where SSR is Used Today

  • Blogs & Content Sites — WordPress, Medium (HTML is generated server-side for SEO).
  • E-commerce Product Pages — Amazon, Flipkart render product pages on the server for fast initial load.
  • News Websites — BBC, CNN serve fully rendered HTML for SEO and performance.
  • Documentation — Read the Docs, Docusaurus use SSR for search-engine visibility.
  • Legacy Enterprise Apps — Many internal tools still use server-rendered forms.

🌐 Where Client-Side Shines

  • Dashboards — Real-time charts, filters, and tables need instant updates.
  • Social Media — Twitter, Facebook (infinite scroll, likes, comments without reload).
  • Collaborative Tools — Google Docs, Figma (multi-user real-time editing).
  • Games — Interactive games need instant feedback (the Tic Tac Toe example!).
  • Form-heavy Apps — Multi-step forms with validation feedback on each field.
ℹ
Modern Hybrid Approach

Most modern apps use both. The initial page is server-rendered (for SEO and fast first paint), and subsequent interactions are client-rendered (for smooth UX). Frameworks like Next.js (React) and Nuxt (Vue) make this pattern easy.

Interview & Exam Questions

Q1: What is server-side rendering (SSR) and how does it differ from client-side rendering? +

SSR: The server generates the full HTML page and sends it to the browser. The browser just displays it. CSR: The browser downloads a minimal HTML shell and JavaScript, then JavaScript generates the UI on the client.

Key difference: SSR sends ready-to-display HTML; CSR sends code that builds the HTML on the client. SSR gives faster first paint and better SEO; CSR gives smoother subsequent interactions.

Q2: Why does every move cause a full page reload in the Flask Tic Tac Toe demo? +

Because each cell is a <button type="submit"> inside an HTML <form>. Clicking a submit button triggers the browser's native form submission behaviour, which navigates to the form's action URL (/move) and loads the server's response as a new page. Since the server returns a complete HTML document (not just data), the entire page is replaced.

Q3: How does Flask persist the game board across HTTP requests? +

Flask uses the session object, which is backed by a signed cookie. The board state is serialised to JSON, signed with app.secret_key, and stored in the browser as a cookie. On each subsequent request, the browser sends the cookie back, Flask verifies the signature, deserialises it, and makes the data available via session['board'].

Q4: What are the disadvantages of storing game state in Flask's session? +

Disadvantages: (1) Size limit — cookies are limited to ~4 KB; (2) Sent on every request — increases bandwidth; (3) Not encrypted by default — the value is signed but base64-encoded, so users can decode it; (4) No server-side control — the user holds the session data; (5) Not suitable for sensitive or large data.

Q5: Describe the full request-response cycle when a player clicks a cell. +

1. User clicks a button inside the form. 2. Browser serialises the form data (cell=3). 3. Browser sends an HTTP POST request to /move with the session cookie. 4. Flask receives the request, matches the URL to the move() view function. 5. View function reads request.form['cell'], updates session['board'], checks for a win condition, toggles the turn. 6. Flask renders board.html template with the updated board. 7. Flask sends the HTML response back with an updated session cookie. 8. Browser parses the HTML and displays the new page.

Q6: What is the role of app.secret_key in Flask? +

app.secret_key is used to cryptographically sign the session cookie. It ensures that the session data has not been tampered with by the client. Without a secret key, Flask's session feature will not work (Flask raises an error). The key should be a long, random string kept secret in production.

Q7: Compare the Flask SSR approach with a Vue.js approach for Tic Tac Toe. +

Flask SSR: State on the server, full HTML sent per move, page reloads, simpler code, no JavaScript needed. Vue.js: State in browser memory, DOM updates in-place, smooth UX, more complex setup (build tools, components), needs API if persistence required. Flask is better for learning HTTP; Vue is better for real-world interactive apps.

Q8: What is WSGI and how does it relate to Flask? +

WSGI (Web Server Gateway Interface) is the Python standard for web servers to communicate with web applications. Flask includes Werkzeug, a WSGI toolkit, which handles the low-level HTTP details. When you run app.run(), Flask starts a WSGI server that listens for HTTP requests, parses them, and calls your view functions.

Q9: How would you modify the Flask demo to support multiple concurrent games? +

Several approaches: (1) Store game state in a database with a game ID, pass the game ID in the URL (/game/<id>/move); (2) Use server-side sessions (Redis, filesystem) instead of cookies; (3) Use query parameters to identify the game. The simplest approach is to use a dictionary in memory keyed by a game ID, though it would not survive a server restart.

Q10: What are Jinja2 templates and why are they used? +

Jinja2 is Flask's default template engine. It allows embedding Python-like expressions inside HTML files using {{ }} for expressions and {% %} for control flow (loops, conditionals). In Tic Tac Toe, Jinja2 iterates over the board array to generate the 9 cells, each pre-filled with the correct mark or blank. This is how dynamic HTML is generated on the server.

Common Mistakes Students Make

✖ Mistake 1: Forgetting to Set secret_key

Running the app without app.secret_key causes a runtime error when using sessions.

ErrorTraceback
RuntimeError: The session is unavailable because no secret
key was set.  Set the secret_key on the application to
something unique and secret.

Fix: Always set app.secret_key = 'your-strong-key' after creating the Flask app.

✖ Mistake 2: Using GET Instead of POST for State Changes

If the form uses method="GET", the cell index goes into the URL query string (/move?cell=3).

  • Problem: GET requests can be bookmarked, cached, and replayed accidentally. They are meant for reading data, not modifying it.
  • Fix: Use method="POST" for any action that changes state (like making a move). This follows HTTP semantics correctly.

✖ Mistake 3: Modifying the Session Without Marking it Modified

When you mutate a mutable object inside the session (like a list), Flask may not detect the change automatically.

WrongPython
board = session['board']
board[cell] = 'X'        # Mutates the list in-place
# session is NOT automatically saved!

Fix: Assign the modified object back to the session to mark it as changed:

CorrectPython
board = session['board']
board[cell] = 'X'
session['board'] = board  # Force session save

✖ Mistake 4: Not Validating the Move on the Server

Students often skip validation and let the server blindly update the board. A malicious user can send a POST request with any cell index, even one already occupied.

Always check: Is the cell index valid? Is the cell empty? Is it really this player's turn?

✖ Mistake 5: Confusing render_template with redirect

render_template() returns HTML directly. redirect() sends a 302 response telling the browser to navigate elsewhere. If you redirect after a move without passing the board state, the user sees the initial board again. Use render_template for this demo since every response includes the full board.

Key Takeaways

  • SSR means the server generates HTML. The browser is just a rendering engine — all logic lives on the server.
  • Every form submission causes a full page reload. This is browser-native behaviour for <form> elements.
  • HTTP is stateless. Flask uses signed cookies (sessions) to persist state across requests.
  • The request-response cycle is the fundamental unit of web communication. Every click = one full cycle.
  • Flask + Jinja2 gives you a powerful server-side rendering stack with minimal code.
  • SSR is simple but limited. It is great for learning and content-focused sites but poor for highly interactive apps.
  • Client-side frameworks (Vue, React) solve the page-reload problem by managing state and DOM in the browser.
  • Validation must happen on the server. Even in SSR, never trust client input.
  • Flask session has limitations: size (~4 KB), not encrypted, sent every request.
  • Understanding SSR first makes learning client-side frameworks easier — you appreciate why they exist.

Summary

📖 Lecture at a Glance

The Flask Tic Tac Toe demo is a server-side rendered web application. Every player click submits an HTML form, triggering a full HTTP request-response cycle. Flask receives the move, updates the board state stored in a signed session cookie, checks for a winner, re-renders the entire board using Jinja2 templates, and sends the complete HTML page back to the browser.

  • HTTP Communication — POST requests carry form data; responses carry full HTML documents.
  • State Management — Flask session (signed cookies) bridges HTTP's stateless nature.
  • Why Reloads Happen — HTML forms natively navigate to the action URL on submit.
  • Trade-offs — SSR is simple and SEO-friendly but causes page flashes and wastes bandwidth.
  • Comparison with JS — Client-side frameworks keep state in the browser for instant, smooth interactions.

Glossary

Client-Side Rendering The browser downloads JavaScript that builds the UI on the client. The server sends code, not ready HTML.
Flask A micro web framework for Python. Minimal, flexible, and includes Jinja2 templating and Werkzeug WSGI toolkit.
Form Submission The browser-native behaviour where a <form> element serialises its inputs and navigates to the action URL.
HTTP (Hypertext Transfer Protocol) The protocol used for communication between web browsers and servers. Request-response based and stateless.
Jinja2 Flask's default template engine. Allows embedding Python-like expressions in HTML using {{ }} and {% %} syntax.
POST An HTTP method used to send data to the server, typically causing a change in state (e.g., making a move in a game).
Route A URL pattern mapped to a view function using the @app.route() decorator in Flask.
Session A mechanism to persist data across HTTP requests. In Flask, implemented via signed cookies by default.
Server-Side Rendering (SSR) The server generates the complete HTML page and sends it to the browser. The browser renders it directly.
Signed Cookie A cookie whose contents are cryptographically signed to prevent tampering. Flask sessions use this.
View Function A Python function in Flask that handles a specific URL route and returns a response (HTML, JSON, redirect, etc.).
Werkzeug The WSGI toolkit that Flask uses internally to handle HTTP requests and responses at a low level.
WSGI Web Server Gateway Interface — the Python standard for connecting web servers to web applications.
