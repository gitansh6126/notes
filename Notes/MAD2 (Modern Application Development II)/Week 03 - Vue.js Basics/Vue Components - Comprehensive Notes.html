01

Overview

A component is a self-contained, reusable piece of UI that encapsulates its own template, script, and style. Components are the fundamental building blocks of any Vue application. Instead of one monolithic file, you split the UI into independent, composable pieces.

🧰 What is a Component?

Think of components like Lego bricks. Each brick has a specific shape and purpose. You can combine bricks to build anything from a small house to a castle. If you need a window, you use the window brick — you do not carve a window into a wall brick.

  • Encapsulated — each component owns its template, logic, and styles.
  • Reusable — write once, use anywhere in the app.
  • Composable — components contain other components (parent-child hierarchy).
  • Communicative — components talk via props (down) and events (up).
ℹ
From Pages to Components

In the Flask Tic Tac Toe demo, the entire UI was one HTML file. In Vanilla JS, we used the DOM directly. Vue components are the next level — they let you split the UI into logical, self-contained pieces that each manage their own slice of the UI.

Single File Components (SFC)

The .vue file is Vue's single-file component format. It packages template, script, and style into one file — the component in its entirety.

📄 Three Sections of a .vue File

📁 Folder Structure (Scalable App)

src/
  components/
    GameBoard.vue
    GameCell.vue
    GameStatus.vue
    AppHeader.vue
  views/
    HomePage.vue
    GamePage.vue
  assets/
    main.css
  App.vue // Root component
  main.js // Entry point
Full .vue component for a Tic Tac Toe cellVue (SFC)
<!-- GameCell.vue -->
<template>
  <button
    class="cell"
    :class="{ taken: value, x: value === 'X', o: value === 'O' }"
    @click="handleClick"
  >
    {{ value }}
  </button>
</template>

<script>
export default {
  name: 'GameCell',
  props: {
    value: { type: String, default: '' },
    index: { type: Number, required: true }
  },
  methods: {
    handleClick() {
      this.$emit('cell-clicked', this.index);
    }
  }
}
</script>

<style scoped>
.cell {
  width: 80px; height: 80px; font-size: 2rem;
  border: 2px solid #1e4f8a; border-radius: 8px;
  background: #fff; cursor: pointer;
}
.taken { cursor: default; background: #eef; }
.x { color: #a8482a; }
.o { color: #2c6b5d; }
</style>
💡
Scoped CSS

The scoped attribute on <style> tells Vue to add a unique data attribute to every element in this component and rewrite the CSS selectors to include that attribute. This means styles in GameCell.vue cannot leak out and affect other components.

04

Component Hierarchy

Vue applications are organised as a tree of components. There is a single root component (usually App.vue) that contains child components, which may contain their own children.

🌳 The Component Tree

🔗 Unidirectional Data Flow

Data flows in one direction only:

Parent Component
props ↓
↔
events (emit) ↑
Child Component
  • Parent → Child: Passes data via props.
  • Child → Parent: Sends messages via events ($emit).
  • Never mutate props in a child. They are read-only.
05

Props — Parent to Child Communication

Props are custom attributes you can register on a component. They allow a parent to pass data down to a child. Props are read-only in the child — the child must never mutate them.

📦 Declaring Props

Array syntax (simple)Vue
// Child component
export default {
  props: ['title', 'likes', 'published']
}
Object syntax (typed)Vue
// Object syntax with validation
export default {
  props: {
    title: { type: String, required: true },
    likes: { type: Number, default: 0 },
    published: { type: Boolean, default: false }
  }
}

📝 Passing Props from Parent

Parent passes props to childVue template
<!-- Static prop: passed as string literal -->
<GameCell value="X" />

<!-- Dynamic prop: bound with : (v-bind) -->
<GameCell :value="board[3]" :index="3" />

<!-- Using v-for -->
<GameCell
  v-for="(cell, i) in board"
  :key="i"
  :value="cell"
  :index="i"
  @cell-clicked="makeMove"
/>

⚠ Props Validation

Always validate your props. It catches bugs and serves as documentation:

Complete prop validationVue
props: {
  value: {
    type: String,
    default: '',
    validator(val) {
      return ['', 'X', 'O'].includes(val);
    }
  },
  index: {
    type: Number,
    required: true
  }
}
⚠
Props Are Read-Only!

Never assign to a prop inside a child component: this.value = 'X'. If you need to modify the data, emit an event and let the parent do it. Mutating props leads to unpredictable behaviour and Vue warns you in the console.

06

Events & $emit() — Child to Parent

When a child needs to communicate something to its parent (a button click, a completed action), it emits an event. The parent listens and responds.

📤 The emit Mechanism

📝 $emit in Action

Child emitsVue
// GameCell.vue (child)
export default {
  props: ['value', 'index'],
  methods: {
    handleClick() {
      // Emit event named 'cell-clicked'
      // with the index as payload
      this.$emit('cell-clicked', this.index);
    }
  }
}
Parent listensVue
// GameBoard.vue (parent)
// Template:
// <GameCell
//   v-for="..."
//   @cell-clicked="makeMove"
// />

export default {
  methods: {
    makeMove(index) {
      console.log('Cell clicked:', index);
      // Update board[index] = this.turn
    }
  }
}

📋 Event Payloads & Multiple Arguments

You can pass additional data with the event:

Emitting with payloadVue
// Child: emit with multiple payload values
this.$emit('move-made', index, turn);

// Parent: receives them as arguments
// Template: @move-made="handleMove"
handleMove(index, turn) {
  console.log(`${turn} played at cell ${index}`);
}
💡
Event Naming Convention

Use kebab-case for event names: $emit('cell-clicked') and listen as @cell-clicked. While Vue also handles camelCase, kebab-case is the HTML-standard convention and works consistently across template and script.

07

Props & Events Together: The Data Flow

Props and events form the complete communication system. Here is the entire cycle in a Tic Tac Toe component architecture.

🔄 Complete Communication Flow

User clicks a cell
↓
GameCell handles @click
↓
GameCell calls this.$emit('cell-clicked', index)
↓
GameBoard receives @cell-clicked="makeMove"
↓
GameBoard updates this.board[index] = this.turn
↓
Vue reactivity updates the prop "value" for that cell
↓
GameCell re-renders with new value
Parent template showing full wiringVue
<!-- GameBoard.vue -->
<template>
  <div class="board">
    <GameCell
      v-for="(cell, i) in board"
      :key="i"
      :value="cell"          <!-- prop down -->
      :index="i"            <!-- prop down -->
      @cell-clicked="makeMove"  <!-- event up -->
    />
  </div>
</template>

<script>
import GameCell from './GameCell.vue';

export default {
  components: { GameCell },
  data() {
    return { board: ['']*9, turn: 'X', gameOver: false }
  },
  methods: {
    makeMove(index) {
      if (this.gameOver || this.board[index]) return;
      this.board[index] = this.turn;
      // ... check win, toggle turn ...
    }
  }
}
</script>
ℹ
The Golden Rule

Props go down through the component tree. Events go up. Data in a child component is never directly modified by the child — instead, the child asks the parent to do it via an event. This pattern keeps data flow predictable and debuggable.

08

Reusability & Modular Architecture

Components are designed for reuse. A well-designed component can be used in many places without modification.

🔄 Making Components Reusable

A component is reusable when it is self-contained and communicates only through its public API:

Public APIDescriptionExample
PropsInput data from parent:value="cell"
EventsOutput messages to parent$emit('cell-clicked', i)
SlotsContent injection points<slot /> in layout components

A reusable GameCell knows nothing about the game rules, the board, or the turn logic. It just shows a value and emits a click. The parent (GameBoard) holds all the game logic.

📝 Reusability Example: Different Contexts

Same GameCell used in different gamesVue
<!-- Tic Tac Toe -->
<GameCell
  v-for="(c,i) in board"
  :value="c" :index="i"
  @cell-clicked="handleTicTacToeMove"
/>

<!-- Minesweeper (same component!) -->
<GameCell
  v-for="(c,i) in minefield"
  :value="c.display" :index="i"
  @cell-clicked="revealCell"
/>

🏗️ Benefits of Modular Architecture

  • Separation of concerns — each component does one thing.
  • Easier testing — test GameCell in isolation without the full game.
  • Parallel development — one developer builds GameCell, another builds GameBoard.
  • Readability — template shows the component tree clearly.
  • Maintainability — fix a bug in GameCell once; it is fixed everywhere.
  • Design system — build a library of reusable UI components (buttons, inputs, cards).
09

Best Practices

✅ Component Design Guidelines

  • One component, one concern — if a component does too much, split it.
  • Props as public API — design props carefully; they are the contract with the parent.
  • Validate props — always use object syntax with type, required, default.
  • Never mutate props — use $emit to request changes.
  • Kebab-case events — $emit('cell-clicked') not $emit('cellClicked').
  • Scoped styles — always use <style scoped> in SFCs.
  • Name your components — name: 'GameCell' helps debugging in Vue DevTools.
  • Flat props — avoid passing entire objects when only a few properties are needed.
  • Small components — if a .vue file exceeds 300 lines, consider splitting it.
  • Document prop contracts — use JSDoc comments or Storybook for shared components.

Common Mistakes

✖ Mistake 1: Mutating Props Directly

WrongVue
// GameCell.vue
props: ['value'],
methods: {
  handleClick() {
    this.value = 'X';  // ✖ BAD: props are read-only!
  }
}

Fix: Emit an event: this.$emit('cell-clicked', this.index) and let the parent update the data.

✖ Mistake 2: Missing components Registration

WrongVue
// Parent uses GameCell but forgot to register it
export default {
  // components: { GameCell }  // missing!
  data() { ... }
}

Fix: Always import and register child components in the components option.

✖ Mistake 3: Using CamelCase Events Without Translation

HTML attributes are case-insensitive. @cellClicked in a template will not match $emit('cellClicked'). Vue's template compiler handles this for in-DOM templates inconsistently. Stick to kebab-case for events: $emit('cell-clicked') and @cell-clicked.

✖ Mistake 4: Passing Too Many Props

If a component has 10+ props, it is probably doing too much. Consider splitting it. A component with many props becomes hard to use and test. Use composition (smaller components composed together) or provide shared state via a store.

✖ Mistake 5: Forgetting scoped in Styles

Without scoped, CSS in a .vue file is global and can affect other components. Always use <style scoped> unless you intentionally want global styles.

✖ Mistake 6: Creating Components When Not Needed

Not everything needs to be a component. If a piece of UI is used exactly once and is small, it may be simpler to keep it inline. Over-componentisation adds complexity. Use components when you need reusability or encapsulation.

Interview & Exam Questions

Q1: What is a Vue component and why use them?+

A Vue component is a reusable, self-contained unit of UI with its own template, script, and styles. Components are used to split the UI into smaller, manageable pieces. Benefits: reusability, encapsulation, easier testing, parallel development, and maintainability. Components communicate via props (down) and events (up).

Q2: How do props work in Vue? What is unidirectional data flow?+

Props are custom attributes that a parent component passes to a child. The child declares its expected props and receives them as properties on this. Unidirectional data flow means data moves in one direction: parent → child via props. Children cannot modify props directly; they emit events to request changes. This makes data flow predictable and easier to debug.

Q3: What is the difference between array syntax and object syntax for props?+

Array syntax (props: ['title', 'likes']) is simple but provides no validation. Object syntax (props: { title: { type: String, required: true } }) allows type checking, required flags, default values, and custom validators. Use object syntax for any component that others will use — it documents the expected API and catches bugs early.

Q4: How does $emit work? What is the event payload?+

$emit(eventName, ...args) fires a custom event on the component instance. The parent listens with @event-name="handler". The first argument is the event name (kebab-case convention), and subsequent arguments become the payload passed to the parent's handler. Example: this.$emit('cell-clicked', index) → parent receives index as the first argument to makeMove(index).

Q5: What are Single File Components (.vue)?+

An SFC is a file with the .vue extension that encapsulates a component's template (<template>), logic (<script>), and styles (<style scoped>) in one file. The build tool (Vite) compiles these into JavaScript. Scoped styles are automatically scoped to the component by adding a unique data attribute to each element.

Q6: Explain the component hierarchy in a Vue application.+

A Vue app has a single root component (usually App.vue) mounted to a DOM element. The root component contains child components, which may contain their own children, forming a component tree. Data flows down via props; events flow up via $emit. This hierarchy mirrors the UI structure and makes the application modular and maintainable.

Q7: What is scoped CSS and why is it important?+

Scoped CSS (<style scoped>) tells Vue to process the CSS so it only applies to the current component. Vue adds a unique data attribute (like data-v-abc123) to every element in the component and rewrites the CSS selectors to include that attribute. Without scoped CSS, styles leak globally and can unintentionally affect other components.

Q8: How would you design a reusable GameCell component?+

A reusable GameCell should: (1) Accept value prop (what to display) and index prop (its position). (2) Emit a cell-clicked event with the index. (3) Have scoped CSS for its visual appearance. (4) Know nothing about the game rules, whose turn it is, or how to check for a win. Those responsibilities belong to the parent component.

Q9: What is the components option and when is it needed?+

The components option registers child components so they can be used in the parent's template. Without registration, Vue does not know what <GameCell /> refers to. In Vue 3's Options API: export default { components: { GameCell } }. In the Composition API with <script setup>, imported components are automatically available in the template.

Q10: Compare building a Tic Tac Toe UI in a single Vue file vs using separate components.+

Single file: Everything in one Vue component. Simple for a small demo, but all logic and template are mixed. Harder to test, reuse, or scale. Separate components: GameBoard (logic + layout), GameCell (reusable cell), GameStatus (score display). Each piece is independently testable, reusable across projects, and maintainable by separate developers. The modular approach is essential for real applications.

Key Takeaways

  • Components are self-contained UI units. They are the building blocks of every Vue application.
  • Single File Components (.vue) package template, script, and style into one file for full encapsulation.
  • Props flow down from parent to child. They are read-only and form the component's public input API.
  • Events flow up via $emit(). The child requests changes; the parent decides whether to act.
  • Unidirectional data flow makes applications predictable and easy to debug. Data always moves in one direction.
  • Scoped CSS prevents style conflicts. Always use <style scoped> in SFCs.
  • Reusability is achieved by designing components with a clear public API (props + events) and no hidden dependencies.
  • Component hierarchy mirrors the UI tree. App.vue at the root, children branching down.
  • Validation on props catches bugs early. Use object syntax with type checking and defaults.
  • Small focused components beat large monolithic ones. Split when a component does more than one thing.

Summary

📖 Lecture at a Glance

Vue components are the fundamental unit of UI organisation. A Single File Component (.vue) encapsulates template (HTML), script (JS), and scoped styles (CSS) in one file. Components form a tree with a root (App.vue) and nested children.

  • Props carry data from parent to child. They are read-only and validated via type declarations.
  • Events ($emit) carry messages from child to parent. The parent listens with @event.
  • Unidirectional data flow — props down, events up — keeps data flow predictable.
  • Scoped CSS isolates styles to their component. No style leaks, no conflicts.
  • Reusability comes from a clean public API: props for input, events for output, slots for content.
  • Component hierarchy mirrors the UI. Each component owns one piece of the interface.
  • The same GameCell component can power Tic Tac Toe, Minesweeper, or any grid-based game.

Glossary

ComponentA reusable, self-contained unit of UI with its own template, data, methods, and styles.
SFC (Single File Component)A .vue file that contains template, script, and style for one component in a single file.
PropsCustom attributes passed from a parent component to a child. Read-only in the child.
$emitA method that fires a custom event on a component, used for child-to-parent communication.
Unidirectional Data FlowA pattern where data moves in one direction: parent to child via props, child to parent via events.

Component RegistrationThe act of declaring a child component in the components option so it can be used in the template.
Prop ValidationSpecifying type, required flag, default value, and custom validator for a prop using object syntax.

Event PayloadThe additional data passed from $emit to the parent's event handler after the event name.
