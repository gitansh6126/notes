Modules, NPM & Objects

How JavaScript code is packaged, shared, and organized — from ES6 modules and the npm ecosystem to object fundamentals and the this keyword.

Why This Topic Exists
Modern JavaScript applications are built from hundreds of files. Without modules, you'd have global namespace pollution and unmanageable dependency graphs. NPM connects your code to 2M+ packages. Objects are the backbone of JS data modeling. And this is the #1 source of confusion and bugs. Master these four pillars to write production-grade JavaScript.
The Problem This Solves
You're building a 50-file web app. Without modules:
1. Global scope pollution: Every script shares one global namespace — variable names collide silently.
2. No dependency management: You manually include script tags in order — one reorder breaks everything.
3. No code reuse: Copy-pasting utility functions across files creates maintenance nightmares.
4. this confusion: Arrow functions, class methods, callbacks — this changes meaning in each context, causing cryptic bugs.
How do you isolate code, manage dependencies, and keep this predictable?
Mental Model — The Office Building
Think of a JavaScript project as an office building:
Modules = Individual offices with doors. Each office has its own supplies (variables) that don't leak into the hallway (global scope). You open the door (export/import) to share specific documents. NPM = The building's supply chain — you order furniture (packages) instead of building it yourself. this = The person in the room — it always refers to whoever is currently executing the code, not where the code was written.
Module → Office with a door (encapsulation)
export → Putting documents on the shared printer
import → Picking up documents from the shared printer
NPM → Ordering pre-built furniture (don't reinvent the wheel)
this → "Who's in charge right now?" — depends on calling context

1 ES6 Modules — Export & Import

Named & Default Exports

Every file is its own module. You control exactly what is exposed to the outside world.
// ------ math.js ------
// Named export — multiple per module
export const PI = 3.14159;
export function add(a, b) { return a + b; }

// Default export — one per module
export default class Calculator { /* … */ }

// ------ app.js ------
// Named import — must use exact name
import { PI, add } from './math.js';

// Default import — any name you choose
import Calc from './math.js';

// Combined import
import Calc, { PI, add } from './math.js';

// Aliases — rename to avoid conflicts
import { add as sum } from './math.js';

// Namespace import — all exports on one object
import * as math from './math.js';
console.log(math.PI);     // 3.14159
console.log(math.add(2, 3)); // 5
Key Insight
Module scope is strict mode by default — no need for "use strict". All top-level declarations are private unless explicitly exported.
Real-World Analogy
A module is like a toolbox drawer with a label. export is hanging a tool on the pegboard outside the drawer; import is taking that specific tool from the pegboard into your own workspace. Default export is the "main" tool everyone knows the drawer for.
CommonJS (Node.js)
Synchronous require() / module.exports. Dynamic imports allowed anywhere. Not supported in browsers without bundling. Used in legacy Node.js code.
ES6 Modules (ESM)
Static import / export (analyzed at parse time). Async loading by default. Works in browsers with <script type="module">. The standard for modern JS.
2 NPM — The Package Ecosystem

package.json, Dependencies & Lockfiles

npm is the default package manager for Node.js. It manages project metadata, scripts, and dependencies.
// Initialize a new project
npm init -y    // creates package.json with defaults

// Install a dependency
npm install lodash    // adds to "dependencies"
npm install --save-dev jest  // adds to "devDependencies"

package.json — stores project name, version, scripts, and dependency lists.

node_modules — where installed packages live. Never commit to version control.

package-lock.json — locks exact version numbers for reproducible installs across machines.

Industry Best Practice
Always commit package-lock.json to version control. It ensures every developer and deployment pipeline installs the exact same dependency tree — preventing "works on my machine" bugs caused by minor version drift.
Common Mistake
Installing packages globally (npm install -g) when they should be local dev dependencies. Tools like ESLint, Prettier, and bundlers belong in devDependencies so every team member uses the same version.
3 Objects — Syntax & Patterns

Object Literals & Modern Shorthands

Objects are the fundamental data structure in JS. Modern syntax makes them more concise.
// Basic literal
const user = {
  name: 'Alice',
  age: 30
};

// Computed property keys
const key = 'dynamicField';
const obj = {
  [key]: 'value',         // evaluates key at definition time
  [`${key}2`]: 'another'
};

// Property value shorthand
const name = 'Bob';
const age = 25;
const person = { name, age }; // { name: 'Bob', age: 25 }

// Method shorthand
const greeter = {
  sayHi() {                 // instead of sayHi: function()
    return `Hi, I'm ${this.name}`;
  }
};

The this Keyword

this is determined by how a function is called, not where it's defined. This is a frequent source of confusion.
// 1. Method call — 'this' is the object the method is called on
user.sayHi();  // this → user

// 2. Standalone call — 'this' is undefined (strict) or global (non-strict)
const greet = user.sayHi;
greet();       // this → undefined (module scope = strict)

// 3. Arrow functions — inherit 'this' from surrounding scope
const obj = {
  name: 'Obj',
  regular() { return this.name; },    // "Obj"
  arrow: () => this.name              // undefined — inherits from outer scope
};
Real-World Analogy
this is like saying "my" — the meaning depends entirely on who says it. If Alice says "my car," "my" refers to Alice. If Bob borrows Alice's words and says "my car," it now refers to Bob. The same function borrowed by different objects changes what this means.
5 Execution Timeline — this Binding
How JavaScript Determines this
RULE 1 — Method call
obj.method() → this = obj. The object before the dot becomes this.
RULE 2 — Standalone function
func() → this = window (strict: undefined). No object = global context.
RULE 3 — Arrow function
() => {} → this = enclosing scope. Arrow functions inherit this from where they were defined, not called.
RULE 4 — Constructor
new Foo() → this = new object being created. Constructor creates a fresh context.
RULE 5 — Explicit binding
call/apply/bind → this = whatever you pass as the first argument. Manual override.

6 Limitations & Gotchas
Limitations
  • ES Modules are browser-only (without bundler): Node.js uses CommonJS (require). Browsers need <script type="module"> or a bundler like Webpack/Vite.
  • NPM has no built-in security audit: Run npm audit manually. Supply chain attacks (like event-stream) are real.
  • node_modules is massive: A typical project has 1000+ packages. Never commit it to git — use .gitignore.
  • this in callbacks loses context: Classic function callbacks lose the outer this. Fix with arrow functions or .bind(this).
  • Computed property names require brackets: { [expr]: value } — can't use dot notation to access them.
  • Object shorthand only works when key and variable name match: { name, age } is shorthand for { name: name, age: age }.
7 Debugging Checklist
Common Module / Object / this Issues
1
If this is undefined in a method, check if you're using an arrow function — arrows don't have their own this
2
If import fails, verify the file has .js extension in the path and type="module" on the script tag
3
If require is not defined, you're in ESM context — use import instead
4
If npm install breaks, delete node_modules and package-lock.json, then reinstall
5
If object destructuring gives undefined, verify the property name matches exactly (case-sensitive)
6
If console.log(this) shows window inside a method, you lost context — use arrow function or .bind()
8 Previous Year Questions (PYQs)
PYQs — Modules, NPM & Objects
Q: What is the difference between named export and default export?
A: Named exports: export const foo = ..., imported as import { foo } from '...'. Multiple allowed. Default exports: export default foo, imported as import foo from '...'. Only one per module. Named exports enforce exact name matching; default allows renaming on import.

Q: What is the value of this in an arrow function inside a class method?
A: this refers to the enclosing lexical scope, not the object that called the method. Arrow functions don't bind their own this — they inherit from the surrounding code (the class instance in this case).

Q: What is the difference between dependencies and devDependencies in package.json?
A: dependencies = packages needed in production (e.g., express, vue). devDependencies = packages needed only during development (e.g., webpack, eslint, jest). Install dev deps with npm install --save-dev.

10 Memory Tricks
Mnemonics to Remember
export = "EX-PORT" (Exit the building)
Export = Exit. Data leaves the module. Import = data comes in. Think of a port: ships leave (export) and arrive (import).
Arrow functions = "ARROW points to definition"
An arrow points at a target. Arrow functions point at where they were defined (not called) for this. Regular functions point at where they were called.
NPM = "Not my problem (I'll just install it)"
Before NPM, developers wrote everything from scratch. Now? "Not my problem" — someone already built it. npm install and move on.
Computed keys = "C Bracket Expression"
Computed keys use [brackets]. Static keys use .dots. { [expr]: val } vs obj.key.

5 Interview Q&A
Q: What's the difference between import { x } and import * as x?
A: import { x } imports only the specific binding x — tree-shakeable, explicit. import * as x imports all exported bindings as properties of a namespace object — never tree-shakeable, used when you need everything from a module.
Q: What does npm ci do and when should you use it?
A: npm ci installs dependencies exactly from package-lock.json — it fails if the lockfile is out of sync with package.json. Use it in CI/CD pipelines for deterministic, faster installs.
Q: How does this behave differently in arrow functions vs regular functions?
A: Regular functions get their own this based on invocation context (method call, standalone, etc.). Arrow functions have no own this — they capture it lexically from the enclosing scope at definition time.
