Asynchrony & the Event Loop

JavaScript is single-threaded but non-blocking. The event loop orchestrates deferred work — timers, network calls, and UI updates — without freezing the main thread.

Why This Topic Exists
JavaScript is single-threaded — it can only do one thing at a time. But modern apps need to fetch data, handle clicks, animate, and process responses simultaneously. The Event Loop is JavaScript's solution: it lets non-blocking async code run on a single thread by offloading operations and processing results via queues. Understanding this is the single most important concept for writing correct async JavaScript.
The Problem This Solves
Without async handling, a single slow API call would freeze the entire browser:
1. User clicks "Submit" → browser sends HTTP request → waits 3 seconds → nothing else works during the wait.
2. Multiple API calls → each blocks the next → total wait = sum of all waits (9 seconds for 3 calls).
3. UI freezes → animations stutter, clicks are lost, users think the app crashed.
How does a single-threaded language handle multiple concurrent operations without freezing?
Mental Model — The Restaurant Kitchen
Think of JavaScript as a restaurant with ONE chef (single thread):
The chef (call stack) handles one order at a time. When an order needs the oven (async API call), the chef puts it in the oven (offloads to browser API) and moves to the next order. When the oven dings (callback/microtask), the chef checks the priority tickets (microtask queue) first, then the regular tickets (macrotask queue). The Event Loop is the sous-chef who keeps checking if the chef is free and hands off the next ticket.
Call Stack → The chef (one person, one task at a time)
Web APIs → The oven (offloaded work, runs independently)
Microtask Queue → Priority tickets (Promises) — checked FIRST
Macrotask Queue → Regular tickets (setTimeout) — checked after microtasks
Event Loop → Sous-chef who hands off tickets when chef is free

1 Sync vs Async & the Call Stack

The Call Stack — One Thing at a Time

The call stack tracks function execution. Each function call adds a frame; each return removes it. Stack overflow = too many frames.
function a() { b(); }
function b() { c(); }
function c() { console.log('done'); }

a();
// Stack: global → a → b → c (then pops in reverse)

// Blocking — if c() took 5 seconds, nothing else runs
function block() {
  const start = Date.now();
  while (Date.now() - start < 5000) {} // blocks for 5s
  console.log('unblocked');
}
Real-World Analogy
The call stack is like a stack of plates at a buffet: you can only wash one plate at a time. If you stop to scrubs a single plate for 5 minutes, every other plate in the stack waits. That's blocking — and no new plates can even be placed on the stack.

Non-Blocking with setTimeout

setTimeout doesn't pause execution — it schedules a callback to run after at least N milliseconds, then continues immediately.
console.log(1);
setTimeout(() => console.log(2), 0); // 0ms delay
console.log(3);

// Output: 1, 3, 2 (NOT 1, 2, 3)
// Why even setTimeout(0) is deferred

setInterval(() => console.log('tick'), 1000);
// Runs callback every ~1000ms (not guaranteed exact)
Common Mistake
Assuming setTimeout(fn, 0) runs immediately. It only guarantees a minimum delay — the callback still waits for all pending code in the current execution context to finish and its turn in the task queue.
2 The Event Loop — Complete Picture

Event Loop with All Queues

The event loop continuously checks: is the call stack empty? If yes, process microtasks, then pick the next macrotask from the task queue.
Microtask Priority
The event loop processes the entire microtask queue before moving to the next macrotask. This is why Promise.resolve().then(fn) always runs before a scheduled setTimeout — even when both are ready at the same time.
Macrotask (Task Queue)
setTimeout, setInterval, I/O callbacks, UI rendering events. Processed one per event loop cycle. Lower priority than microtasks.
Microtask
Promise.then/catch/finally, queueMicrotask(), MutationObserver. Processed immediately after the current task finishes — entire queue is drained before next macrotask.
4 Execution Timeline — Classic Exam Trace
Trace: console.log(1); setTimeout(() => console.log(2), 0); Promise.resolve().then(() => console.log(3)); console.log(4);
STEP 1 — Call Stack
console.log(1) → prints 1. Stack empty.
STEP 2 — setTimeout
setTimeout(cb, 0) → offloaded to Web API. API starts timer (0ms). Callback registered to macrotask queue.
STEP 3 — Promise.then
Promise.resolve().then(cb) → callback registered to microtask queue (not executed yet).
STEP 4 — console.log(4)
console.log(4) → prints 4. Call stack empty.
STEP 5 — Microtask first!
Event loop checks microtask queue → finds Promise callback → prints 3. Microtask queue empty.
STEP 6 — Macrotask last
Event loop checks macrotask queue → finds setTimeout callback → prints 2.
FINAL OUTPUT
1 → 4 → 3 → 2 (not 1, 2, 3, 4! Promises before setTimeout)
3 Callback Pattern & Callback Hell

The Nested Callback Problem

Before Promises, async operations used callbacks. Nesting multiple callbacks created the infamous "callback hell" pyramid.
// Callback hell — deeply nested, hard to read, error-prone
getUser('alice', (err, user) => {
  if (err) { return console.error(err); }
  getPosts(user.id, (err, posts) => {
    if (err) { return console.error(err); }
    getComments(posts[0].id, (err, comments) => {
      if (err) { return console.error(err); }
      console.log(comments);
    });
  });
});

// Each level adds a new indentation — the "pyramid of doom"
// Solution: Promises + async/await flatten this to linear code
Industry Context
Modern codebases use async/await exclusively for async control flow. Callback patterns survive only in event emitters (on('data', cb)) and some Node.js legacy APIs. The util.promisify wrapper converts callback-based APIs to Promise-based ones.

requestAnimationFrame

Schedules code to run before the next browser repaint. The browser batches all rAF callbacks in a single frame — ideal for animations.
function animate(timestamp) {
  element.style.transform = `translateX(${Math.sin(timestamp / 1000) * 50}px)`;
  requestAnimationFrame(animate);  // schedule next frame
}
requestAnimationFrame(animate);

// Unlike setInterval(fn, 16) — rAF pauses when tab is inactive
// rAF fires ~60 times/sec (matches display refresh rate)

5 Limitations & Gotchas
Limitations
  • Single-threaded bottleneck: CPU-intensive work (sorting 1M items, image processing) blocks the main thread. Use Web Workers for heavy computation.
  • Microtask starvation: If microtasks keep adding more microtasks, macrotasks (including setTimeout, UI updates) never run. The browser may kill the tab.
  • setTimeout(fn, 0) is not instant: Minimum delay is ~4ms in browsers. It queues, not executes immediately.
  • Callback hell: Deep nesting of callbacks makes code unreadable and error-prone. Fix with async/await.
  • No error propagation in microtasks: Unhandled Promise rejections crash silently until they become unhandledrejection events.
  • requestAnimationFrame doesn't guarantee 60fps: It depends on monitor refresh rate and system load.

When Async Code Behaves Unexpectedly
1

2

3

4

5

6

7 Previous Year Questions (PYQs)
PYQs — Asynchrony & Event Loop
Q: Explain the Event Loop. What are microtasks and macrotasks?
A: The Event Loop continuously checks: if call stack is empty, process ALL microtasks first (Promise callbacks, queueMicrotask), then ONE macrotask (setTimeout, setInterval, I/O). Microtasks have higher priority. This ensures Promises resolve before setTimeout callbacks, even with 0ms delay.

Q: What is the output and why? setTimeout(() => console.log("a"), 0); console.log("b");
A: Output: "b" then "a". console.log("b") is synchronous and runs immediately. setTimeout offloads the callback to the Web API, which queues it as a macrotask. It runs after the current synchronous code finishes.
20233 marks
Q: What is callback hell and how do you avoid it?
A: Callback hell = deeply nested callbacks creating a pyramid shape, making code hard to read/debug. Avoid with: 1) Named functions, 2) Promises (.then chains), 3) async/await (cleanest). async/await makes async code look synchronous.

8 Memory Tricks
Mnemonics to Remember
"Micro before Macro"
Microtasks (Promises) always run before Macrotasks (setTimeout). Think: Micro < b Macro — the smaller one goes first.
"Single Thread = Single Chef"
One chef = one task at a time. The chef can hand off to the oven (Web API) but can't cook two dishes simultaneously. When the oven dings, the chef finishes the current task first.
Output order: 1 → 4 → 3 → 2
Sync code (1, 4) runs immediately. Microtasks (3) run before macrotasks (2). Remember: Sync → Micro → Macro = S → M → M.
setTimeout(0) ≠ instant
A 0ms timeout means "run as soon as possible after current code and microtasks." Minimum delay is ~4ms. It's never truly instant.

5 Interview Q&A
Q: Explain the difference between macrotasks and microtasks with an example.
A: console.log(1); setTimeout(() => console.log(2)); Promise.resolve().then(() => console.log(3)); console.log(4); — Output: 1, 4, 3, 2. Synchronous code runs first (1,4), then microtask queue drains (3), then the next macrotask runs (2).
Q: Is setTimeout(fn, 0) truly zero delay?
A: No. Browsers enforce a minimum delay of 4ms for nested timeouts (HTML spec). Even at 0ms, the callback must wait for the call stack to clear and its turn in the task queue. It's "as soon as possible, but not immediately."
Q: What happens to the event loop when an error is thrown in an async callback?
A: Uncaught errors in macrotask callbacks bubble to the global error event (or crash). Uncaught Promise rejections trigger unhandledrejection. The event loop itself continues running — the error only affects the current task.
