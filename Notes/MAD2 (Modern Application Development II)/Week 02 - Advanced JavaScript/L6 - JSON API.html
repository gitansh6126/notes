JSON & the Fetch API

Two pillars of modern web communication: JSON — the universal data interchange format — and the Fetch API — the native browser tool for HTTP requests.

Why This Topic Exists
Every modern app talks to a server. JSON is the universal data format for APIs — it's how frontend and backend communicate. The Fetch API is how JavaScript makes HTTP requests. Without understanding these, you can't build login systems, load data, submit forms, or integrate with any third-party service. This is the bridge between frontend and backend.
The Problem This Solves
Your Vue frontend needs user data from a Flask backend. Without JSON and Fetch:
1. Language mismatch: JavaScript objects can't be sent over HTTP directly — they need serialization.
2. No standard format: Without JSON, each API would use its own format (XML, YAML, custom), making integration chaotic.
3. Blocking requests: Old XMLHttpRequest was callback-based, verbose, and blocked the UI during requests.
4. No streaming: Basic XHR loaded entire responses into memory, crashing on large datasets.
How do you serialize data, send HTTP requests, and handle responses without blocking the UI?
Mental Model — The Courier Service
Think of JSON + Fetch as a courier service:
You have a gift (JavaScript object). You can't hand it to someone across the world directly. So you pack it in a box (JSON.stringify), the courier ships it (Fetch request), the recipient unpacks it (JSON.parse), and now they have the same gift (object). The box (JSON) is universal — anyone in the world can open it. The courier (Fetch) handles delivery asynchronously — you don't stand at the door waiting.
JS Object → The gift (data you want to send)
JSON.stringify → Packing into a box (serialization)
Fetch → The courier (HTTP request, async)
Response → The delivered box (raw text/binary)
JSON.parse → Unpacking (deserialization)

1 JSON Syntax Rules

What Makes Valid JSON

JSON is a subset of JavaScript object syntax — stricter, but language-agnostic. Every JSON document is a valid JS expression (but not vice versa).
// ✅ Valid JSON
{
  "name": "Alice",
  "age": 30,
  "isStudent": false,
  "hobbies": ["reading", "coding"],
  "address": null
}

// ❌ Invalid JSON — common mistakes
{
  name: "Alice",       // keys must be double-quoted
  "age": 30,           // trailing comma not allowed
  "comment": "...",   // no comments allowed
  "date": new Date()   // no Date type — only strings, numbers, objects, arrays, booleans, null
}
Real-World Analogy
JSON is like a shipping container standardized by ISO — every container has the same locking mechanism, dimensions, and labeling rules. It doesn't matter if the cargo came from a factory in China or a farm in Brazil; as long as it fits the container, any port in the world can handle it.
2 JSON.stringify & JSON.parse

Serializing & Deserializing

JSON.stringify converts JS values to JSON text. JSON.parse converts JSON text back to JS values. Both accept optional transform functions.
const data = {
  name: 'Alice',
  age: 30,
  born: new Date('1995-01-01'),
  score: undefined,      // dropped by stringify
  greet() { return 'hi'; } // functions are dropped
};

// Basic stringify
JSON.stringify(data);
// → {"name":"Alice","age":30,"born":"1995-01-01T00:00:00.000Z"}

// With replacer — control what gets included
JSON.stringify(data, ['name', 'age']);

// With space — pretty printing
JSON.stringify(data, null, 2);

// JSON.parse with reviver — transform values on the way in
const parsed = JSON.parse(jsonStr, (key, value) => {
  if (key === 'born') return new Date(value);
  return value;
});
Common Mistake
Assuming JSON.parse(JSON.stringify(obj)) creates a deep clone. It doesn't handle undefined, functions, Symbols, Dates (converted to strings), circular references, or special objects like Map/Set. Use a structured clone algorithm for true deep cloning.
3 The Fetch API — GET

Fetching Data from a Server

The Fetch API provides a Promise-based interface for HTTP requests. Always check response.ok before consuming the body.
// Basic GET request
const response = await fetch('https://api.example.com/users');

// Always check for HTTP errors
if (!response.ok) {
  throw new Error(`HTTP ${response.status}`);
}

// Parse JSON body
const users = await response.json();

// Alternative: parse as text
const text = await response.text();

// Chaining with .then() instead of await
fetch('https://api.example.com/users')
  .then(res => {
    if (!res.ok) throw new Error(`Error ${res.status}`);
    return res.json();
  })
  .then(data => console.log(data))
  .catch(err => console.error(err));

// Checking specific status codes
if (response.status === 404) { /* not found */ }
if (response.status === 401) { /* unauthorized */ }
Important
fetch() only rejects on network errors (DNS failure, no connection). HTTP 4xx/5xx responses are not rejections — you must check response.ok or response.status explicitly.
4 Execution Timeline — Fetch Request Lifecycle
Trace: fetch('/api/users').then(r => r.json()).then(data => console.log(data))
STEP 1 — Request initiated
fetch() creates a Request, sends it via HTTP. Returns a Promise that resolves to a Response object. No data yet — only headers received.
STEP 2 — Response headers arrive
.then(r => r.json()) — the Response object is available. r.json() reads the body stream and parses JSON. Returns another Promise.
STEP 3 — Body parsed
.then(data => console.log(data)) — the parsed JavaScript object is now available. Use it in your app.
STEP 4 — Error handling
.catch(err => ...) handles network errors, DNS failures, or JSON parse errors. Always add this.
5 POST with JSON Body

Sending Data to a Server

POST requests send JSON in the request body. Set the Content-Type header so the server knows how to parse it.
const newUser = {
  name: 'Bob',
  email: 'bob@example.com'
};

const response = await fetch('https://api.example.com/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify(newUser)
});

if (!response.ok) {
  const errorBody = await response.text();
  throw new Error(`POST failed: ${errorBody}`);
}

const created = await response.json();
REST API Best Practices
Use POST for creating resources, GET for reading, PUT for full replacement, PATCH for partial updates, and DELETE for removal. Always match the HTTP method to the operation's semantics.
XMLHttpRequest (Legacy)
Event-based (onload, onerror), verbose boilerplate, no Promise support. Requires manual JSON.parse. Still exists in browsers but not used in modern code.
Fetch API (Modern)
Promise-based, clean API, native streaming support, built-in JSON parsing (.json()). Rejects only on network errors — must check response.ok for HTTP errors.

7 Limitations & Gotchas
Limitations
  • No request timeout built-in: fetch() has no timeout option. Use AbortController or Promise.race() with a timer.
  • No automatic retry: If a request fails, you must implement retry logic yourself (exponential backoff).
  • response.ok is NOT auto-checked: fetch() doesn't reject on 404/500 — it only rejects on network errors. Always check response.ok.
  • No progress tracking: Unlike XHR, basic fetch() doesn't support upload/download progress. Use XHR or ReadableStream for large files.
  • CORS restrictions: Cross-origin requests are blocked by browsers unless the server sends Access-Control-Allow-Origin headers.
  • JSON.parse is synchronous: Parsing a 10MB JSON response blocks the main thread. Consider streaming for large payloads.
8 Interview Q&A
Q: Why does JSON.stringify drop functions and undefined values?
A: JSON is a data-interchange format, not a serialization format for executable code. Functions have no equivalent in other languages, and undefined is not a valid JSON value. This ensures the output can be parsed by any language, not just JavaScript.
Q: How would you handle a fetch request that might return either JSON or an error text?
A: Check response.ok first. If the request failed, call response.text() to read the error body (which may contain a human-readable message). If successful, call response.json() to parse the data. You cannot call both .json() and .text() on the same response — the body can only be read once.
Q: What's the difference between PUT and PATCH?
A: PUT replaces the entire resource with the provided representation (idempotent). PATCH applies a partial update — it only changes the fields you send. PUT requires the full resource; PATCH can send just the diff.
9 Debugging Checklist
Common Fetch/JSON Issues
1
If fetch rejects with "Failed to fetch", check: CORS headers, network URL, server running, HTTPS mixed content
2
If response.json() throws, the server returned invalid JSON — check Content-Type header and server logs
3
If response seems empty, check if you forgot to await the r.json() call — it returns a Promise, not data
4
If POST fails, verify Content-Type header is application/json and body is JSON.stringify(data)
5
If response.ok is false but no error thrown, remember: fetch only rejects on network errors, not HTTP errors
6
If data arrives but is empty array, check server-side query — the API works but returned no matching records

Q: What is JSON? What are its rules?
A: JSON (JavaScript Object Notation) is a lightweight data interchange format. Rules: 1) Keys must be in double quotes. 2) No trailing commas. 3) No comments. 4) Values: string (double-quoted), number, boolean, null, array, object. 5) Strings use \" for escaping.

Q: What is the difference between JSON.stringify and JSON.parse?
A: JSON.stringify(obj) converts a JavaScript object to a JSON string (serialization). JSON.parse(str) converts a JSON string back to a JavaScript object (deserialization). Round-trip: JSON.parse(JSON.stringify(obj)) creates a deep copy.

Q: Does fetch() reject on HTTP 404 or 500?
A: No. fetch() only rejects on network failures (DNS error, no internet). It resolves with response.ok = false for 4xx/5xx. Always check if (!response.ok) throw ... manually.

Q: Compare XHR and Fetch API.
A: XHR: callback-based, verbose, supports progress events, older browsers. Fetch: Promise-based, cleaner syntax, no progress events, modern browsers. Fetch is preferred for new code. Both handle the same HTTP operations.

8 Memory Tricks
Mnemonics to Remember
stringify = "String-ify" (to string)
String-ify = make it a string. JSON.stringify() turns objects into strings. JSON.parse() turns strings back into objects. String = send, Parse = receive.
fetch() resolves, doesn't reject on 404
fetch = Fetches whatever the server sends, even errors. You must check response.ok yourself. It's like a courier who delivers broken packages too.
JSON rules = "No trailing commas, double quotes only"
JSON is strict: double quotes for keys/strings (no single quotes), no trailing commas, no comments, no undefined. Think: JSON is the minimal, strict subset of JS object notation.
Response must be consumed once
Like opening a letter — you can only read the body once. Call .json(), .text(), or .blob() once. Cache the result if needed.
