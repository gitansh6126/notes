Collections & Arrays

JavaScript provides three built-in collections — Array, Map, and Set — each designed for different data access patterns. Master when to use which.

Why This Topic Exists
Every real application needs to store, query, transform, and deduplicate data. Plain objects fall short: they coerce keys to strings, lack iteration order, and have no built-in dedup. Arrays, Maps, and Sets solve these gaps — they're the foundation of React state management, Redux reducers, caching layers, and real-time data pipelines. Without mastering collections, you can't write efficient JavaScript beyond toy examples.
The Problem This Solves
Imagine you're building a real-time trading app with 500K+ stock symbols. You need to:
1. Look up a stock by its ticker symbol — but plain objects coerce keys to strings, risking collisions with prototype properties like constructor or toString.
2. Maintain insertion order so the UI shows stocks in the order they were added — plain objects don't guarantee this.
3. Deduplicate a stream of incoming trades — checking arr.includes() is O(n), making it too slow for high-frequency data.
4. Use DOM elements as keys for click tracking — impossible with plain objects since they only accept strings.
How do you store heterogeneous keys, guarantee order, and check membership in O(1)?
1 Arrays — The Workhorse Collection
Mental Model — The Three Warehouses
Think of collections as three types of warehouses:
Array = A numbered shelf row. Items sit at fixed positions (0, 1, 2…). You can add to the end (push), remove from the end (pop), or jump to any position by index. Fast for ordered access, slow for searching by value.
Array → Numbered shelves, fast index access, slow search
Map → Labeled filing cabinet, any label type, O(1) lookup
Set → VIP guest list, no duplicates, instant membership check
Concept Map — When to Use What

Array Literal & Indexing

Arrays are ordered, integer-indexed collections that can hold any type. Created with square brackets.
// Array literal syntax
const fruits = ['apple', 'banana', 'cherry'];
const mixed = [42, 'hello', true, null, { a: 1 }];

// Indexing — zero based
console.log(fruits[0]);        // "apple"
console.log(fruits[fruits.length - 1]); // "cherry"

// Length property — automatically tracked
console.log(fruits.length); // 3

// Setting length truncates or extends
fruits.length = 2;
console.log(fruits); // ["apple", "banana"]
Real-World Analogy
An array is like a numbered locker row — each locker has an index (0, 1, 2…), you can put anything inside any locker, and the sign at the end tells you how many lockers exist.

Mutating Arrays — Stack & Queue Operations

JavaScript arrays double as stacks (LIFO) and queues (FIFO) through built-in methods.
const arr = [1, 2, 3];

// Push — add to end (stack push)
arr.push(4);           // [1, 2, 3, 4]

// Pop — remove from end (stack pop)
arr.pop();              // [1, 2, 3], returns 4

// Unshift — add to front
arr.unshift(0);        // [0, 1, 2, 3]

// Shift — remove from front (queue dequeue)
arr.shift();            // [1, 2, 3], returns 0

// Splice — general purpose insert/remove/replace
arr.splice(1, 1);        // remove 1 item at index 1 → [1, 3]
arr.splice(1, 0, 99);    // insert 99 at index 1 → [1, 99, 3]
arr.splice(1, 1, 42);    // replace index 1 with 42 → [1, 42, 3]
2 Iteration Methods

Transforming & Querying Arrays

Modern JS favors declarative iteration helpers over manual for loops. Each method returns a predictable result.
const nums = [1, 2, 3, 4, 5];

// forEach — side effects, no return value
nums.forEach(n => console.log(n));

// map — transform every element, returns new array
const doubled = nums.map(n => n * 2);
// doubled → [2, 4, 6, 8, 10]

// filter — keep elements passing a test
const evens = nums.filter(n => n % 2 === 0);
// evens → [2, 4]

// reduce — accumulate a single value
const sum = nums.reduce((acc, n) => acc + n, 0);
// sum → 15

// find — first element matching predicate
const found = nums.find(n => n > 3);
// found → 4

// findIndex — index of first match
const idx = nums.findIndex(n => n === 3);
// idx → 2

// some — any element passes the test?
nums.some(n => n > 4);   // true

// every — all elements pass?
nums.every(n => n > 0);   // true
Industry Usage
React's useState + map/filter is the standard pattern for rendering lists. Redux reducers rely heavily on .reduce() and the spread operator for immutable updates. In production code, for loops are rarely seen — these declarative methods are the norm.
3 Spread & Destructuring

Spreading Arrays

The spread operator (...) expands an array into individual elements — essential for immutability.
const a = [1, 2];
const b = [3, 4];

// Concatenate (immutable)
const merged = [...a, ...b];  // [1, 2, 3, 4]

// Copy (shallow)
const copy = [...a];

// Prepend / insert
const withHead = [0, ...a];    // [0, 1, 2]

// Pass to function arguments
Math.max(...a);  // 2

Destructuring Arrays

Unpack values from arrays into distinct variables in a single expression.
const rgb = [255, 128, 64];

// Basic destructuring
const [red, green, blue] = rgb;
console.log(red);   // 255

// Skipping with holes
const [first, , third] = rgb;
console.log(third); // 64

// Rest pattern
const [head, ...tail] = rgb;
// head → 255, tail → [128, 64]

// Default values
const [r = 0, g = 0, b = 0] = [];

// Nested destructuring
const matrix = [[1, 2], [3, 4]];
const [[a11, a12], [a21, a22]] = matrix;
4 Map — Keyed Collections

Map: Beyond Plain Objects

Map holds key-value pairs where keys can be any type (objects, functions, primitives). Maintains insertion order.
const userMap = new Map();

// Set — add or update a key-value pair
userMap.set('alice', { age: 30 });
userMap.set('bob', { age: 25 });

// Get — retrieve by key
userMap.get('alice');  // { age: 30 }

// Has — check for key existence
userMap.has('charlie'); // false

// Delete — remove a key
userMap.delete('bob');

// Size — count entries (not available on objects)
userMap.size;  // 1

// Clear — remove all entries
userMap.clear();

// Iteration — keys(), values(), entries()
for (const [k, v] of userMap) { /* … */ }

// Objects as keys — impossible with plain objects
const btn = document.getElementById('submit');
clickMap.set(btn, { count: 0 });
Common Mistake
Using for...in on a Map — for...in iterates enumerable properties, not Map entries. Always use for...of or the .forEach() method on Maps.
Object as Map
Keys coerced to strings; no .size; no iteration order guarantee; prototype pollution risk; object keys not supported.
Map
Keys any type; .size property; insertion order preserved; no prototype keys; built-in iteration with for...of.
5 Set — Unique Values

Set: Automatic Uniqueness

Set stores unique values of any type. Duplicates are silently ignored. Perfect for deduplication.
const tags = new Set();

// Add — duplicates are ignored
tags.add('js');
tags.add('react');
tags.add('js');        // ignored — already present

// Has — fast membership check
tags.has('react');     // true

// Delete — remove a value
tags.delete('react');

// Size
console.log(tags.size); // 1

// Clear
tags.clear();

// Deduplicate an array
const dupes = [1, 2, 2, 3, 3, 3];
const unique = [...new Set(dupes)];
// unique → [1, 2, 3]

// Iteration
for (const val of tags) { /* … */ }
tags.forEach(v => console.log(v));
Real-World Analogy
A Set is like a VIP guest list: you can try to add a name a hundred times, but the list only ever holds one entry per person. Checking if someone is on the list is instant.

7 Execution Timeline — How reduce Works Step-by-Step
Execution Trace: [1,2,3,4].reduce((acc, n) => acc + n, 0)
STEP 0 — Initialize
acc = 0 (initial value), array = [1, 2, 3, 4]
STEP 1 — Iteration 0
acc = 0, n = 1 → returns 0 + 1 = 1. New accumulator = 1
STEP 2 — Iteration 1
acc = 1, n = 2 → returns 1 + 2 = 3. New accumulator = 3
STEP 3 — Iteration 2
acc = 3, n = 3 → returns 3 + 3 = 6. New accumulator = 6
STEP 4 — Iteration 3
acc = 6, n = 4 → returns 6 + 4 = 10. New accumulator = 10
RESULT
Final accumulator = 10 → returned by reduce
8 Limitations & Gotchas
Limitations
  • Array is not truly associative: reduce depends on the order of elements — rearranging gives different results (unlike mathematical addition).
  • Map/Set are not JSON-serializable: JSON.stringify(new Map()) returns "{}". You must convert to arrays first: JSON.stringify([...myMap]).
  • Map memory overhead: For simple string-keyed data, a plain object is more memory-efficient. Map adds per-entry overhead for tracking insertion order.
  • Set uses SameValueZero comparison: NaN === NaN is false, but Set.has(NaN) returns true. However, +0 and -0 are treated as equal.
  • No built-in sorting for Map/Set: Maps maintain insertion order, not sorted order. For sorted access, convert to array and sort.
  • Spread creates shallow copies: [...nestedArr] copies references, not deep values. Mutating a nested object affects both copies.
9 Debugging Checklist
When Collections Behave Unexpectedly
1
Check if you're using for...in on a Map/Set — use for...of instead
2
Verify map.get(key) returns undefined vs null — get returns undefined for missing keys
3
Check if Set.has() is comparing by reference, not value — objects are compared by identity
4
If reduce throws "cannot read property of undefined", verify you provided an initialValue
5
If array mutations don't trigger React re-renders, check you're using splice/push (mutating) vs spread (immutable)
6
If JSON.stringify(map) gives {}, convert to array first: [...map]
10 Code Patterns — Do This, Not That
Anti-pattern Mutating array in a React component 
// BAD — mutates state, won't trigger re-render
const addItem = () => {
  items.push(newItem);   // mutates!
  setItems(items);       // same reference — no re-render
};
Correct Immutable update with spread 
// GOOD — new array reference triggers re-render
const addItem = () => {
  setItems([...items, newItem]);  // new reference
};
Anti-pattern Using object for non-string keys 
// BAD — keys coerced to strings, prototype pollution risk
const cache = {};
cache[domElement] = "data";  // key becomes "[object HTMLDivElement]"
Correct Using Map for object keys 
// GOOD — preserves object identity as key
const cache = new Map();
cache.set(domElement, "data");  // O(1), no coercion
11 Memory Tricks
Mnemonics to Remember
MAP — "MAny Properties"
M-A-P: Many key types, Anything goes, Preserves order. If you need non-string keys or guaranteed insertion order, use Map.
SET — "Simply Eliminate Tweaks (duplicates)"
A Set is a duplicate killer. Feed it duplicates, it strips them out. Think of it as filter + unique in one operation.
reduce = "Remember: Every Value Eventually Collapses"
Remember: Every Value Eventually Collapses → many items collapse into one. That's what reduce does.
spread vs destructuring
Spread = Pour OUT (expand inline). Destructuring = Pour INTO variables (extract). Same ... syntax, opposite directions.
12 Previous Year Questions (PYQs)
PYQs — Collections & Arrays
Q: Explain the difference between Array, Map, and Set. When would you use each?
A: Array = ordered indexed collection for sequential data. Map = key-value pairs with any-type keys and O(1) lookup. Set = unique values with O(1) membership check. Use Array for ordered lists, Map for caches/registries, Set for deduplication.
20244 marks
Q: Write a program to deduplicate an array using Set. What is the time complexity?
A: [...new Set(arr)] — O(n) time because Set.add() is O(1) and spread is O(n). Alternative: arr.filter((v, i) => arr.indexOf(v) === i) which is O(n²).
20233 marks
Q: What is the output of [1,2,3].map(parseInt)? Explain why.
A: [1, NaN, NaN]. map passes (element, index) to parseInt. So: parseInt(1, 0) → 1, parseInt(2, 1) → NaN (base 1 is invalid), parseInt(3, 2) → NaN (3 is not valid in base 2).
20243 marks
Q: How does reduce work? Write a function to flatten a nested array using reduce.
A: reduce iterates, passing accumulator + current element. Flatten: arr.reduce((acc, val) => Array.isArray(val) ? acc.concat(flatten(val)) : acc.concat(val), []). This recursively flattens nested arrays into a single-level array.

14 Interview Q&A
Q: When would you choose a Map over a plain object?
A: When you need keys that aren't strings (e.g., DOM elements as keys), when you need reliable iteration order, when you need .size, or when you want to avoid prototype pollution risks. Plain objects are fine for simple string-keyed dictionaries with static shape.
Q: What's the difference between map and forEach?
A: map returns a new array of the same length (transformation). forEach returns undefined and is used solely for side effects. If you need the result, use map — it's declarative and chainable.
Q: How does reduce work and when is it appropriate?
A: reduce(fn, initialValue) iterates through the array, passing an accumulated value and current element to the callback. It's appropriate whenever you need to derive a single value (sum, product, grouped object, flattened array) from a collection.
15 One-Page Revision Summary
Collections & Arrays — Complete Cheat Sheet
Array
Ordered, index-based. [1,2,3]. Fast index access O(1), slow search O(n). Mutable methods: push/pop/shift/unshift/splice.
Map
Key-value pairs, any-type keys, O(1) get/set/has. .size, insertion order. Use for caches, DOM refs, registries.
Set
Unique values only. O(1) has/add/delete. Dedup: [...new Set(arr)]. Uses SameValueZero (NaN === NaN is true).
Iteration Methods
map=transform, filter=keep, reduce=accumulate, find=first match, some/every=boolean query.
Spread (...)
Expand array inline. [...a, ...b] for concat, [item, ...arr] for prepend. Shallow copy only.
Destructuring
Unpack into variables: const [a, b] = arr. Skip: [a, , c]. Rest: [head, ...tail]. Default: [a = 0].
Time Complexity
Array index: O(1). Array search: O(n). Map get/has: O(1). Set has: O(1). Array.includes: O(n).
Gotchas
map(parseInt) → [1, NaN, NaN]. Map not JSON-safe. Spread is shallow. Set uses SameValueZero.
