Modularity in JavaScript

Early JS

No modules

All scripts dumped into one page. Like one giant kitchen with no departments.

Node.js era

CommonJS

Sync loading — fine for servers, blocks browsers.

Browser fix

AMD

Async loading for browsers. Solved blocking, but ugly syntax.

Modern standard

ES6 modules

Async, clean, works everywhere. Use this.

Modules — the "kitchen department" model

Bundle related code into files; share only what's needed

AnalogyA restaurant kitchen has separate departments: pastry, grill, salad. Each does its job independently. The head chef (main file) calls each station. No station messes with another's tools directly.
Real use — RMSSeparate modules: students.js (data), assessments.js (logic), dashboard.js (UI). Each imports only what it needs. Changing assessment logic doesn't break the UI file.

Export / import — the public counter

Only what you export is visible outside the file

AnalogyA pharmacy keeps most medicines in the back (private). Only approved items go on the public counter (exported). The customer (importing file) can only access what's on the counter.
Real use — CubedekhoExport calculateDiscount() from pricing.js. The cart page imports just that function — it can't accidentally read or modify the raw price table sitting inside that file.
export const calculateDiscount = (price, pct) => price - (price * pct / 100);
import { calculateDiscount } from './pricing.js';

Default export — the signature product

One primary thing per module; importers can name it anything

AnalogyA bakery's "default" is the one cake they're known for. You walk in and ask for "the special" — you don't need to know its full official name. The importer picks the nickname.
Real use — DMA lead-capture form component exported as default. Every page that imports it can call it whatever variable name makes sense locally.
export default function LeadForm() { ... }
import MyForm from './LeadForm.js';  // named anything
Imported variables are read-only. To change a module's value, call a function that the module itself exports — don't try to reassign the import directly.

NPM + bundlers — the supply chain

NPM = warehouse; Webpack/Rollup = delivery truck

AnalogyNPM is the Amazon warehouse — millions of packages. Your project orders what it needs. Webpack/Rollup is the delivery truck that bundles everything into one parcel the browser can open.
Real useCubedekho uses a WordPress/PHP stack now, but if you ever build a standalone JS frontend, Vite (modern bundler) would be your delivery truck — fast, zero-config.

Object orientation in JavaScript

Object literals — the student profile card

Key-value pairs bundled into one named thing

AnalogyAn RMS student's ID card: name, age, batch, score — all bundled into one physical card. The card is the object. Each field is a property.
Real use — RMSStore each student as an object with methods. student.getGrade() calculates and returns the grade — the logic lives inside the object, not scattered across your codebase.
const student = { name: 'Arjun', score: 88, getGrade() { return this.score >= 75 ? 'A' : 'B'; } };

The this keyword — "current object" pronoun

Refers to whichever object is calling the method

AnalogyIn a group of students, saying "I scored 88" means something different depending on who says it. this is that "I" — it refers to whoever is currently running the method.
Real useIn an RMS batch object, this.students inside a method refers to that batch's own student list — not some other batch's list. Without this, JavaScript doesn't know which object's data to read.

Prototypical inheritance — DNA chain

Objects inherit behaviour from a parent object, not a class blueprint

AnalogyYou inherit traits from your father who inherited them from your grandfather. There's no "human blueprint document" — the traits live on the actual person and pass down the chain. JS objects work the same way: one object is set as the prototype of another.
Real use — RMSA PremiumStudent object's prototype is Student. It automatically has all base student methods (like getGrade()), and adds its own (getCoachingSlot()). No need to rewrite shared logic.
The class keyword is just cleaner syntax on top of this prototype system — it doesn't change how JS actually works underneath.

Object copying — the "copy trap"

Assignment shares a pointer; spread makes a shallow copy

Analogy — referenceGiving a colleague your Google Doc link. You both see the same document. If they edit it, your version changes too. That's y = x — one object, two names pointing at it.
Analogy — shallow copyPrinting a photocopy of the document. Top-level text is independent. But if the doc has a link to another file, that link still points to the same place. That's spread: { ...x }.
Shallow copy danger: if an object has nested objects (e.g. a student object with an address sub-object), spread still shares that nested reference. Changing copy.address.city also changes the original. Use a deep-copy utility for nested structures.
const original = { name: 'Arjun', scores: [88, 92] };
const copy = { ...original };  // scores array still shared!
copy.name = 'Priya';  // safe — top-level
copy.scores.push(100);  // MUTATES original.scores too
