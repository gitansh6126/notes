Prototypes & Classes

JavaScript's inheritance model is prototype-based — objects inherit from other objects. The class keyword provides familiar syntax over this same mechanism.

Why This Topic Exists
JavaScript doesn't have classes like Java or C++ — it has prototypes. Every object inherits from another object through the prototype chain. The class keyword (ES6) is syntactic sugar over this prototype system. Understanding both is essential because: frameworks use classes extensively (Express, Mongoose, Jest), inheritance patterns power component hierarchies, and private fields enable true encapsulation.
The Problem This Solves
You need to model a hierarchy: Animal → Dog → GuideDog. Each level adds behavior. Without prototypes:
1. Code duplication: You'd copy the breathe() method into every subclass.
2. No dynamic inheritance: You can't add a method to a "parent" and have it appear on all "children" automatically.
3. No encapsulation: All properties are public — anyone can modify internal state.
4. new keyword confusion: What happens when you call new Dog()? The constructor chain is invisible.
How does JavaScript share behavior between objects without classes?
Mental Model — The Family Recipe
Prototypes are like family recipes passed down through generations:
Your grandmother has a secret sauce recipe (prototype). Your mother learned it and added her own twist (instance). You inherited both versions — you can use the original sauce, override it with mom's version, or add your own. The class keyword is like writing the recipe in a cookbook — it looks organized, but the underlying tradition (prototype chain) is what actually passes knowledge down.

1 The Prototype Chain

How Property Lookup Works

Every JS object has an internal link to another object called its prototype. When a property isn't found on the object itself, JS walks up the chain.
const parent = { greet: () => 'Hello' };
const child = Object.create(parent);
child.name = 'Child';

console.log(child.name);   // "Child" — own property
console.log(child.greet);  // "Hello" — found on prototype
console.log(child.toString); // found on Object.prototype

// __proto__ — the actual prototype reference (deprecated but widely used)
console.log(child.__proto__ === parent); // true

// .prototype — exists only on constructor functions (not instances)
console.log(Array.prototype); // the object that all arrays inherit from
Real-World Analogy
The prototype chain is like a junior developer asking a senior, who asks a principal — each level has more knowledge. If the junior doesn't know a method, the question automatically goes up to the next level without the code needing to do anything explicit.

Constructor Functions & new

Before class syntax, constructor functions created objects with a shared prototype.
// Constructor function
function Animal(name) {
  this.name = name;  // 'this' is the new instance
}

// Methods go on .prototype — shared across all instances
Animal.prototype.speak = function() {
  return `${this.name} makes a sound`;
};

const dog = new Animal('Rex');
console.log(dog.speak());   // "Rex makes a sound"

// instanceof checks prototype chain
console.log(dog instanceof Animal);  // true
console.log(dog instanceof Object); // true
Common Mistake
Defining methods inside the constructor body (this.speak = function…) instead of on .prototype. This creates a new function copy per instance, wasting memory. Shared methods always go on the prototype.
4 Execution Timeline — What new Actually Does
Trace: const dog = new Dog("Rex")
STEP 1 — Create empty object
{} — a blank object is created in memory
STEP 2 — Set prototype
dog.__proto__ = Dog.prototype — link the object to the class's prototype chain
STEP 3 — Execute constructor
this = the new object. this.name = "Rex" assigns the property. Constructor returns this implicitly.
RESULT
dog = the new object with name: "Rex" and access to all Dog.prototype methods
2 Class Syntax

Class Declaration, Constructor & Methods

The class keyword provides a cleaner syntax over the prototype pattern — but under the hood, it's still prototypes.
class Animal {
  // Constructor — runs when 'new' is called
  constructor(name) {
    this.name = name;
  }

  // Instance method — goes on Animal.prototype
  speak() {
    return `${this.name} makes a sound`;
  }

  // Static method — on the class itself, not instances
  static classInfo() {
    return 'This is the Animal class';
  }

  // Getter — computed property
  get uppercaseName() {
    return this.name.toUpperCase();
  }

  // Setter — reactive property assignment
  set uppercaseName(val) {
    this.name = val.toLowerCase();
  }
}

console.log(Animal.classInfo()); // "This is the Animal class"
const a = new Animal('Leo');
console.log(a.speak());          // "Leo makes a sound"

Private Fields (#) & Static Members

Private fields use # prefix — truly private, enforced at the language level.
class BankAccount {
  // Private field — cannot be accessed outside the class
  #balance = 0;

  constructor(owner) {
    this.owner = owner;
  }

  deposit(amount) {
    if (amount > 0) this.#balance += amount;
  }

  getBalance() {
    return this.#balance;
  }
}

const acct = new BankAccount('Alice');
acct.deposit(100);
console.log(acct.getBalance());  // 100
console.log(acct.#balance);      // SyntaxError — truly private
3 Inheritance — extends & super

Subclassing & Method Overriding

Inheritance creates an is-a relationship. super calls the parent constructor or method.
class Dog extends Animal {
  constructor(name, breed) {
    super(name);         // must call parent constructor first
    this.breed = breed;
  }

  // Override parent method
  speak() {
    return `${super.speak()} — WOOF!`;
  }
}

const rex = new Dog('Rex', 'Husky');
console.log(rex.speak()); // "Rex makes a sound — WOOF!"
console.log(rex instanceof Dog);    // true
console.log(rex instanceof Animal); // true
Key Rule
In a subclass constructor, super() must be called before accessing this. The parent constructor is responsible for creating the instance — the child then extends it.
Prototype Pattern
Function constructors + manual .prototype assignment. Object.create() for inheritance. More verbose but shows the underlying mechanism clearly. Full control over property descriptors.
Class Syntax
class + extends + super. Cleaner, self-documenting. Supports static methods, getters/setters, private fields. Still prototype-based under the hood — no different performance characteristics.

5 Limitations & Gotchas
Limitations
  • No multiple inheritance: A class can only extends one parent. Use mixins for multiple behavior sources.
  • #private fields need transpilation: Older browsers don't support # syntax. Babel/Buble transpiles it.
  • Prototype chain is slow for deep hierarchies: Each property lookup walks up the chain. Keep hierarchies shallow.
  • super() must be called before accessing this: Calling this before super() in a subclass constructor throws a ReferenceError.
  • Static methods aren't inherited by instances: Dog.bark() works, but dog.bark() doesn't if bark is static.
  • Prototype pollution is a security risk: Modifying Object.prototype affects ALL objects. Never do this in production.
6 Debugging Checklist
Common Prototype/Class Issues
1
If super() is not called before this, you'll get "Must call super constructor" — always call super() first in subclass constructors
2
If a method shows "is not a function", check if the method is on the prototype, not the instance
3
If instanceof returns false, check the prototype chain — the object might be from a different version of the class
4
If private field #foo throws "not defined", ensure the class that defines it is the same class accessing it — private fields don't cross class boundaries
5
If this is wrong inside a callback, use an arrow function to capture the enclosing this
7 Previous Year Questions (PYQs)
PYQs — Prototypes & Classes
Q: Explain the prototype chain in JavaScript. How does property lookup work?
A: Every object has an internal [[Prototype]] link. When you access a property, JS checks the object first, then walks up the chain: obj → obj.__proto__ → obj.__proto__.__proto__ → ... → null. This is why inherited methods work — they're found higher in the chain.
20244 marks
Q: What is the difference between __proto__ and prototype?
A: __proto__ is the actual link from an instance to its parent's prototype. prototype is a property of constructor functions/classes that becomes the __proto__ of instances created with new. Object.getPrototypeOf(obj) is the proper way to access the prototype.
20233 marks
Q: What happens when you call new? List all steps.
A: 1) Create empty object. 2) Set __proto__ to constructor's prototype. 3) Execute constructor with this = new object. 4) Return the object (or custom return if constructor returns an object).
20243 marks
Q: What are private fields (#) and why use them?
A: Private fields (#field) are only accessible within the class that defines them. They enforce encapsulation — external code can't read or modify them. Unlike _convention (which is just a naming convention), # is enforced by the runtime.

5 Interview Q&A
Q: What's the difference between __proto__ and .prototype?
A: __proto__ is the actual prototype link on every object (what JS follows during property lookup). .prototype is a property that exists only on constructor functions — it's the object that will become the __proto__ of instances created with new.
Q: Can you create an object without a prototype?
A: Yes — Object.create(null) creates an object with no prototype chain. This is useful for pure dictionaries (like hash maps) where you don't want any inherited properties like toString or hasOwnProperty.
Q: When should you use prototypal inheritance vs composition?
A: Prefer composition ("has-a") over inheritance ("is-a") for code reuse — use mixins, factory functions, or dependency injection. Reserve extends for true taxonomic relationships (e.g., Dog extends Animal). Deep inheritance hierarchies are fragile.

10 One-Page Revision Summary
Prototypes & Classes — Complete Cheat Sheet
Prototype Chain
obj → obj.__proto__ → ... → null. Property lookup walks up the chain. Found = used, not found = undefined.
new keyword
4 steps: create obj, set prototype, run constructor (this=new obj), return obj.
class syntax
Sugar over prototypes. class Dog { constructor() {} bark() {} }. Clean, readable, familiar to OOP devs.
extends + super
class B extends A. Call super() before this in constructor. super.method() calls parent's version.
#private fields
#secret — only accessible inside the class. True encapsulation, not just a naming convention.
static methods
static create() — called on the class, not instances. Dog.create() works, dog.create() doesn't.
this in methods
Method call → obj. Arrow → enclosing scope. Constructor → new object. Call/apply → manual.
Key gotcha
super() must come before this in subclass constructors. Private fields don't cross class boundaries.
