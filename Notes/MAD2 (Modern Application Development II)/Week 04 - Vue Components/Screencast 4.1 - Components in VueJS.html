AppDev2 · Vue.js

Components in Vue.js

Refactoring a monolithic app into small, self-contained, reusable pieces — each with its own data, customized from outside via props, and talking back via events.

The mental model — one HQ, many independent centers
1 COMPONENTIZATION

Breaking the monolith into pieces

A component is a self-contained, reusable UI block, registered with Vue.component(name, options). Its data must be a function (not a plain object), and its template needs exactly one root element.

🏢
Analogy: RMS today runs like one giant classroom script handling every center manually. Componentizing is turning that into a "Center" blueprint — build it once, stamp out as many centers as you open, each running independently.
🎯
Real use: On Cubedekho, a ProductCard component is the reusable unit — same layout, different cube data per instance. On RMS's future franchise dashboard, a CenterCard component would work the same way.
// Registering a reusable "Center" component
Vue.component('rms-center', {
  // data MUST be a function → every center gets its OWN state,
  // not one shared object mutated by all instances
  data: function() {
    return {
      students: 0
    }
  },
  // exactly ONE root element wraps the template
  template: `<div class="center-card">
              <h3>{{ students }} students</h3>
            </div>`
})
2 PROPS — passing data IN

Parent customizes each instance

Props are declared as an array of strings on the component, then bound like a normal HTML attribute in the parent's template — data flows one direction: down.

📦
Analogy: Head Office ships the same curriculum kit to every center, but stamps each box with the city name — Sirsa, Panipat, Rohtak. Same component, different label passed in.
🎯
Real use: A rms-center component reused 3 times on one page, each customized only by the city prop — no copy-pasted markup.
// Component declares which props it accepts
Vue.component('rms-center', {
  props: ['city'],  // ← whitelist of allowed inputs
  data: function() { return { students: 0 } },
  template: `<div>{{ city }} — {{ students }} students</div>`
})
<!-- Parent template: same component, 3 different props -->
<rms-center city="Sirsa"></rms-center>
<rms-center city="Panipat"></rms-center>
<rms-center city="Rohtak"></rms-center>
3 EVENTS — communicating OUT

Child reports back to parent

A component fires this.$emit('event-name'), and the parent listens with v-on / @ — this is how a local action (inside one center) can update global state (Head Office's total count).

📞
Analogy: When Sirsa center enrolls a new student, it doesn't edit Head Office's records directly — it "calls in" the update. HQ is listening for that call and updates the master dashboard.
🎯
Real use: A future RMS parent-engagement dashboard where each center component emits enrollment-added, and HQ's total-students counter increments live — no page reload, no manual sync.
// Inside the child component's method
methods: {
  enrollStudent: function() {
    this.students++
    // tell the parent something happened — don't touch parent state directly
    this.$emit('enrollment-added')
  }
}
<!-- Parent listens with v-on (or @ shorthand) -->
<rms-center
  city="Sirsa"
  @enrollment-added="totalStudents++">
</rms-center>
