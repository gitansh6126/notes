Components in VueJS

Building self-contained, reusable UI blocks — following the DRY (Don't Repeat Yourself) principle to refactor large codebases into maintainable pieces.

1 WHAT ARE COMPONENTS?
🎯 Definition

A component is a self-contained, reusable UI block with its own data, template, and behavior. Components are registered with Vue.component(name, options) or defined as .vue single-file components. They allow you to split the UI into independent, reusable pieces that can be composed together.

🧠 Why Components?

When building complex interfaces, UI elements often repeat the exact same layout structures while displaying different data. Components let you:

  • DRY (Don't Repeat Yourself): Define a component once, reuse it many times.
  • Encapsulation: Each component manages its own state, template, and styling.
  • Composability: Build complex UIs by nesting simple components.
  • Maintainability: Change one component, and every instance updates across the app.
  • Testability: Isolated components are easier to unit test.
2 COMPONENT STRUCTURE
🏛 Three Core Pillars
1
Props
Custom options passed down from parent to child. Flow downward, read-only for the child.
2
Data
Isolated local state managed within the component instance. Must be a function.
3
Template
HTML structure with {{ }} text interpolation. One root element required.
💻 Code Example
// Register a global component
Vue.component('product-card', {
  // 1. Props: data passed IN from parent
  props: ['product', 'currency'],

  // 2. Data: local state (MUST be a function for isolation)
  data() {
    return {
      isFavorite: false,
      quantity: 1
    }
  },

  // 3. Template: exactly ONE root element
  template: `
    <div class="product-card">
      <img :src="product.image" :alt="product.name">
      <h3>{{ product.name }}</h3>
      <p>{{ currency }}{{ product.price }}</p>
      <button @click="isFavorite = !isFavorite">
        {{ isFavorite ? '❤️' : '🤍' }}
      </button>
    </div>
  `
});

<!-- Usage in parent template -->
<product-card
  :product="laptop"
  currency="$">
</product-card>
📩 Props — Parent to Child

Props are custom attributes registered on a component that allow the parent to pass data down to the child. Props flow one direction: downward. The child should never mutate a prop directly.

Parent
Owns the data
⬇ props
Child
Receives & displays
⬆ events
Parent
Handles the event
// Component declares props
Vue.component('user-profile', {
  props: {
    name: { type: String, required: true },
    age:  { type: Number, default: 0 },
    role: { type: String, default: 'viewer' }
  },
  template: '<div>{{ name }} ({{ role }})</div>'
});

// Parent passes data via attributes
<user-profile
  :name="currentUser.name"
  :age="currentUser.age"
  role="admin">   // static string (no colon)
</user-profile>
📤 Events — Child to Parent

A child component fires a custom event with this.$emit('event-name', payload), and the parent listens with @event-name="handler". This is how a local action (inside a child) can communicate back up to the parent.

// Inside child component
methods: {
  addToCart() {
    this.$emit('add-to-cart', this.product.id);
  }
}

// Parent listens
<product-card
  :product="item"
  @add-to-cart="handleAddToCart">
</product-card>
Best Practice
Use kebab-case for event names in templates (@add-to-cart) and camelCase or kebab-case inside $emit. Vue normalizes them automatically.
🎭 Slots — Content Distribution

Slots provide a flexible content distribution mechanism that allows components to act as structural templates. Instead of passing simple data via props, the parent can pass complex markup inside the child's tags.

<!-- Child template with slots -->
<div class="modal">
  <div class="modal-header">
    <slot name="header">Default Title</slot>
  </div>
  <div class="modal-body">
    <slot />  <!-- default unnamed slot -->
  </div>
  <div class="modal-footer">
    <slot name="footer"></slot>
  </div>
</div>

<!-- Parent uses the component with slot content -->
<modal>
  <template v-slot:header>
    <h2>Confirm Delete</h2>
  </template>
  <p>Are you sure you want to delete this item?</p>
  <template v-slot:footer>
    <button @click="confirm">Yes</button>
    <button @click="cancel">No</button>
  </template>
</modal>
3 REAL-WORLD CONTEXT
🏠 Real-Life Analogy — Restaurant Kitchen
Analogy — Restaurant
Component = Kitchen prep station. Each station (salad, grill, dessert) is a self-contained unit with its own ingredients (data), tools (methods), and output (template). The head chef (parent component) sends orders (props) to each station. The stations send back completed dishes (events). If you need a new menu item, you don't rebuild the kitchen — you just add a new station (component).
💼 Industry Example — Amazon Product Card
Industry Example — Amazon
Amazon.com displays thousands of products daily. Each product card is a component with:

• Props: product.name, product.price, product.image, product.rating
• Data: isInCart, isWishlisted, quantity
• Events: @add-to-cart, @add-to-wishlist
• Slots: Badge content (e.g., "Best Seller", "Amazon's Choice") can be slotted in dynamically

Amazon engineers never write 1000 product card HTML blocks. They write one <product-card> component and reuse it 1000 times with different product data passed via props. When they update the card design (e.g., adding a "Climate Pledge Friendly" badge), every instance updates instantly.

5 COMMON MISTAKES & BEST PRACTICES
⚠ Common Mistakes
Common Mistake
Mutating props directly. Props are read-only. If a child modifies a prop, Vue warns you. Use the prop as initial value for local data or emit an event to let the parent change it.
Common Mistake
Using a plain object for data instead of a function. In components, data must be a function that returns an object. Otherwise, all instances share the same data object.
Common Mistake
Forgetting the single root element. A component's template must have exactly one root element (one outermost tag). Use a <div> wrapper or a fragment in Vue 3.
💡 Best Practices
Best Practice 1
Validate props with types and requirements. Always specify type, required, and default for props. Catches bugs early and documents the component's API.
Best Practice 2
Keep components small and focused. Each component should do one thing well. If a component grows too large, break it into smaller sub-components.
Best Practice 3
Use kebab-case for components in templates. <my-component> not <MyComponent> (though Vue supports both, kebab-case is HTML-standard).
6 REVISION CHEAT SHEET
📋 Key Takeaways
  • Component = reusable self-contained UI block with props, data, and template.
  • Props flow down (parent → child). Events flow up (child → parent via $emit).
  • Data in components must be a function (returns new object per instance).
  • Slots allow content distribution — pass markup instead of just data.
  • Single root element required in Vue 2 templates.
  • DRY principle: Build once, reuse everywhere.
  • Lifecycle hooks: created (fetch data), mounted (DOM ready), destroyed (cleanup).
❓ Interview Questions
Q: What is the difference between props and data in a Vue component?
A: Props are data passed into a component from its parent — they are read-only and should not be mutated by the child. Data is the component's own local state — it is private to the component and can be mutated freely. Props are declared in the props array/object, data is declared in the data() function (which must return an object).
Q: Why must component data be a function?
A: If data were a plain object, all instances of the same component would share the same data object — mutating one instance would affect all others. By using a function, each instance gets its own fresh copy of the data object, ensuring proper isolation between component instances.
Q: How do you pass data from a child component to a parent?
A: The child calls this.$emit('event-name', payload) and the parent listens with @event-name="handler" on the child's tag in the parent template. Events are the standard mechanism for child-to-parent communication in Vue.
