🔍 Topic Deep Dives
Comprehensive coverage of every concept with exam-relevant details and PYQ notes.
🔥 Flask
Routes: @app.route() decorator, methods parameter (GET, POST, etc.), dynamic segments (<int:id>, <string:name>).
Blueprints: Blueprint('name', __name__) for modular organization. Register with app.register_blueprint(bp, url_prefix='/api').
App Factory: create_app() function returns configured Flask instance. Enables testing, multiple instances, extension lazy-loading.
Request/Response: request.args, request.json, request.form. Response via jsonify(), make_response(), or returning tuples (data, status, headers).
Error Handling: @app.errorhandler(404) for custom error pages. abort(status_code) to trigger handlers.
Context: Application context (current_app, g), Request context (request, session). Pushed automatically during requests.
PYQ: Flask route matching (order matters — Flask uses first-match), blueprint registration with URL prefix, app factory vs single module pattern.
⚡ Cache
Why cache? Reduce database/API calls, lower latency, improve throughput.
Cache types: In-memory (local), Redis (distributed), filesystem, Memcached.
Flask-Caching: @cache.cached(timeout=50) decorator. cache.set(key, value, timeout) and cache.get(key) for manual control.
Cache key: Generated from request path, query args, view function name. Customizable via cache_key_prefix.
Invalidation: cache.delete(key), cache.clear(), or use versioning. TTL-based expiry is the simplest strategy.
PYQ: Cache behavior on expiry (cache miss → recompute), TTL interpretation questions, cache key generation collisions.
🔒 JWT (JSON Web Tokens)
Structure: header.payload.signature — three base64url-encoded segments separated by dots.
Auth Flow: Login → server validates credentials → returns JWT → client stores token → sends in Authorization: Bearer <token> header.
Access vs Refresh: Access token (short-lived, ~15 min), Refresh token (long-lived, ~7 days). Refresh token used to get new access tokens without re-login.
flask-jwt-extended: create_access_token(identity=user_id), @jwt_required(), get_jwt_identity().
Claims: sub (subject/user ID), exp (expiration), iat (issued at), nbf (not before). Custom claims can be added.
PYQ: JWT verification failure scenarios (expired, tampered signature, wrong secret), token expiry detection, missing token returns 401.
🍪 Cookies
Setting: response.set_cookie('key', 'value', max_age=3600, httponly=True, secure=True, samesite='Lax').
Attributes: domain, path, expires/max_age, httponly (prevents JS access), secure (HTTPS only), samesite (Strict/Lax/None).
Sessions: Flask sessions use signed cookies (itsdangerous). session['user'] = 'value'. Secret key required.
CSRF: Cross-Site Request Forgery — use CSRF tokens, Samesite cookies, or custom headers.
PYQ: Cookie attribute effects (httpOnly prevents XSS, secure prevents MITM), session cookie vs persistent cookie (with/without max_age).
🌐 CORS (Cross-Origin Resource Sharing)
Same-Origin Policy: Browser blocks JS from making requests to a different origin (scheme + host + port).
Simple vs Preflight: Simple requests (GET, HEAD, POST with certain content types) skip preflight. Non-simple requests trigger OPTIONS preflight.
Headers: Access-Control-Allow-Origin: *, Access-Control-Allow-Methods, Allow-Headers, Allow-Credentials: true.
flask-cors: CORS(app) enables globally. @cross_origin() on specific routes.
Credentials: With credentials: 'include', Access-Control-Allow-Origin cannot be * — must specify exact origin.
PYQ: CORS headers analysis, preflight trigger conditions (custom headers, non-standard content types), credentialed requests restrictions.
📡 SSE (Server-Sent Events)
Direction: One-way communication channel — server pushes data to the client.
Client API: new EventSource('/stream') — automatically reconnects on connection loss.
Content Type: text/event-stream. Data format: data: {...}\n\n. Optional event:, id:, retry: fields.
vs WebSocket: SSE is one-way (server → client), WebSocket is bidirectional. SSE uses HTTP, simpler, auto-reconnect. WebSocket requires upgrade handshake.
Use cases: Live notifications, stock tickers, feed updates, log streaming.
PYQ: SSE event flow (connection → data stream → close/reconnect), reconnection behavior (last event ID), content-type requirement.
🔔 Webhook
Pattern: Event-driven callback — Server A sends HTTP POST to Server B's registered URL when an event occurs.
Verification: Payload signature using HMAC-SHA256 (shared secret). Server B verifies signature before processing.
Retry: Exponential backoff on failure. Dead letter after max retries.
vs Polling vs SSE: Webhook is push-based (server → server), Polling is pull-based, SSE is push (server → browser).
Use cases: Payment notifications (Stripe/Razorpay), CI/CD triggers, GitHub event hooks.
PYQ: Webhook flow (event → register → POST → verify → process), signature verification logic, replay attacks (mitigated by nonce/timestamp).
⏳ Long Polling
Mechanism: Client sends request → server holds connection open until data is available or timeout → sends response → client immediately sends new request.
vs SSE vs WebSocket: Long polling works over standard HTTP, no special protocol. Higher latency than SSE. Good fallback when SSE/WebSocket unavailable.
Headers: If-Modified-Since, Last-Modified for conditional requests.
Timeout: Server sends empty response (304 or 204) after timeout period. Client reconnects.
Use cases: Chat applications (legacy browsers), real-time updates where SSE/WS blocked.
PYQ: Long polling request/response cycle, timeout behavior, empty response handling, comparison with SSE.
🧊 Celery
Async Task Queue: Offloads long-running tasks from the request-response cycle.
Components: Celery app, Worker (executes tasks), Broker (RabbitMQ/Redis — message queue), Backend (stores results).
Task Definition: @celery.task decorator. task.delay(args) (simple) vs task.apply_async(args, countdown=10, queue='high') (fine control).
Task States: PENDING → STARTED → SUCCESS/FAILURE/RETRY. Tracked via AsyncResult.
Periodic Tasks: Celery Beat — scheduler that sends tasks at regular intervals (@celery.on_after_configure.connect or beat_schedule).
Chaining/Groups: chain(task1.s() | task2.s()), group(task1.s(), task2.s()), chord (group with callback).
PYQ: Celery task states lifecycle, delay() vs apply_async() differences, broker purpose (not Celery itself — common trap).
📨 RabbitMQ
Message Broker: Receives messages from producers and routes them to consumers via exchanges and queues.
Flow: Producer → Exchange → (binding) → Queue → Consumer.
Exchange Types: direct (routing key match), topic (pattern match), fanout (broadcast to all), headers (header-based routing).
ACK/NACK: Consumer acknowledges (basic_ack) after processing. basic_nack / basic_reject on failure.
Persistence: Durable queues + persistent messages survive broker restart.
Dead Letter Queue: Unprocessed/rejected messages are routed to DLQ for later inspection.
vs Redis: RabbitMQ is a full-featured message broker with routing. Redis is simpler, used for pub/sub + cache, less reliable for message delivery.
PYQ: Exchange types and their routing behavior, message flow through broker, ACK mechanism purpose.
📠 Redis
In-Memory Store: Key-value store with optional persistence. Sub-millisecond latency.
Use Cases: Cache (SET/GET with TTL), session store, pub/sub, rate limiter, Celery broker/backend.
Data Types: String, List, Set, Sorted Set (ZSET), Hash. Each has specialized commands.
Pub/Sub: PUBLISH channel message (producer), SUBSCRIBE channel (consumer). No message persistence — if no subscriber, message is lost.
TTL/Expiry: SET key value EX 60 or EXPIRE key 60. Keys auto-deleted after expiry.
PYQ: Redis commands and their effects, Cache behavior (LRU eviction, TTL), Pub/Sub message delivery (no persistence — important!).
💡 Lighthouse
Performance Metrics: FCP (First Contentful Paint), LCP (Largest Contentful Paint), TTI (Time to Interactive), TBT (Total Blocking Time), CLS (Cumulative Layout Shift).
Opportunities: Eliminate render-blocking resources, defer unused CSS/JS, optimize images, enable compression.
Categories: Performance, Accessibility, Best Practices, SEO, Progressive Web App.
PYQ: Lighthouse metrics meaning (LCP measures loading, CLS measures visual stability), identifying optimization opportunities from audit output.

⚠️ Tricky Questions — High-Yield Traps
These edge cases and nuance questions consistently appear in exams. Master them to avoid losing marks.
1. JWT expiry vs refresh: An access token is expired but the refresh token is still valid. What happens? → Server returns 401 for access token; client uses refresh token to obtain a new access token.
2. CORS preflight conditions: When does the browser send a preflight OPTIONS request? → When the request is non-simple: custom headers, non-standard content types (e.g. application/json), non-GET/POST methods.
3. Celery task retry loop: A task with infinite retries keeps failing. → Without max_retries or proper error handling, it can loop forever. Always set max_retries.
4. Cache staleness: Stale data is served from cache after TTL expiry. → Cache miss triggers recomputation, but concurrent requests might all miss and overload DB (thundering herd).
5. SSE reconnection: Client reconnects after network drop — duplicate events. → Use lastEventId to resume from the correct position and deduplicate on client.
6. Webhook replay attacks: Same webhook payload delivered multiple times. → Without nonce/timestamp in signature, attacker can replay. Use HMAC with timestamp + nonce + idempotency key.
7. Redis pub/sub message loss: Publisher sends but no subscriber is active. → Redis Pub/Sub has no persistence — messages are lost if no subscriber. Use Redis Streams or a queue for reliable delivery.
8. Flask blueprints order: Two blueprints with overlapping routes. → First-registered blueprint wins. Route registration order matters in Flask.
9. Long polling timeout: Server reaches timeout with no data. → Returns empty response (304/204). Client immediately sends a new long-poll request.
10. Cookie stealing: How to prevent? → httpOnly (prevents JS access), secure (HTTPS only), samesite (prevents CSRF), __Host- prefix for path binding.

🚧 Common Traps — Don't Fall For These
These are the most common mistakes students make. Each trap has been seen in multiple exam cycles.
❌ TRAP
"JWT stored in localStorage is safe because it's encrypted."
✅ TRUTH
JWT is encoded (base64url), not encrypted. Storing in localStorage makes it vulnerable to XSS attacks. Use httpOnly secure cookies instead.
❌ TRAP
"CORS prevents CSRF attacks."
✅ TRUTH
CORS is about relaxing same-origin policy for cross-origin requests. CSRF is about cookie theft via forged requests. They are different concerns — CORS does not prevent CSRF.
❌ TRAP
"Celery is a message broker."
✅ TRUTH
Celery is an async task queue. It requires a message broker like RabbitMQ or Redis to function. Calling Celery a broker is a common exam trap.
❌ TRAP
"Always cache everything — it always improves performance."
✅ TRUTH
Cache has trade-offs: stale data, memory overhead, invalidation complexity, thundering herd. Cache strategically, not universally.
❌ TRAP
"SSE is always better than WebSocket for real-time apps."
✅ TRUTH
SSE is one-way (server → client), auto-reconnects, simpler. WebSocket is bidirectional, lower latency, full-duplex. Choose based on need, not preference.

⚡ 5 Caching Patterns
1. Cache-Aside: App checks cache first; on miss, fetches from DB and populates cache. cache.get() → miss → DB → cache.set().
2. Write-Through: Every write goes to cache first, then DB. Ensures cache is always fresh.
3. Write-Behind: Write to cache immediately, asynchronously write to DB later. Fast writes, risk of data loss.
4. TTL-Based Expiry: @cache.cached(timeout=60). Key auto-deletes after timeout. Simple but may serve stale data.
5. Cache Invalidation: cache.delete(key) on update. Use versioned keys (user_123_v2) for fine-grained control.
🔒 5 Security Questions (JWT, CORS, Cookies)
1. JWT signature prevents tampering — verify with decode(token, secret, algorithms=['HS256']).
2. Never store JWT in localStorage — use httpOnly cookies to prevent XSS theft.
3. CORS Access-Control-Allow-Origin: * cannot be used with credentials (withCredentials: true).
4. CSRF vs CORS — CSRF attacks use cookies (automatic), CORS is about origin policy. Use CSRF tokens + SameSite cookies.
5. Cookie flags: HttpOnly (no JS), Secure (HTTPS only), SameSite=Lax (prevents cross-site sending).
📡 5 Real-Time Patterns
1. SSE: Server → Client, one-way. text/event-stream, auto-reconnect. Use for notifications, live feeds.
2. Webhook: Server → Server, event-driven. HTTP POST to registered URL. Verify with HMAC signature.
3. Long Polling: Client holds connection, server responds when data available. Fallback when SSE/WS unavailable.
4. WebSocket: Bidirectional, full-duplex. Requires upgrade handshake (ws://). Lower latency than SSE.
5. Polling: Client repeatedly sends HTTP requests at intervals. Simplest but wasteful. Use only for non-real-time data.
🧊 5 Celery & Redis Patterns
1. Celery: @celery.task(bind=True, max_retries=3) + self.retry(countdown=60) for retries.
2. task.delay(args) = task.apply_async(args=args). Use apply_async for countdown, queue, priority.
3. Redis as Celery broker: simple setup, but lacks RabbitMQ's routing (exchanges, bindings).
4. Redis Pub/Sub: PUBLISH channel msg + SUBSCRIBE channel. No persistence — messages lost if no subscriber.
5. Celery Beat: schedule=crontab(minute='*/5') for periodic tasks. Define in beat_schedule dict.
