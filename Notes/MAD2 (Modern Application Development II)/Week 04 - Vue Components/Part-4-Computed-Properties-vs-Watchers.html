Computed Properties vs. Watchers

Two distinct strategies for handling data changes — and why choosing the wrong tool for the job is one of the most common anti-patterns in Vue development.

1 COMPUTED PROPERTIES
🎯 Definition

Computed properties calculate and expose derived data based on original, reactive dependencies. Whenever a source dependency changes, the computed property automatically re-evaluates and updates its value. They always return a value used directly in the template.

🧠 Intuition

Computed properties are like formulas in a spreadsheet. Cell C1 has =A1+B1. When A1 or B1 changes, C1 recalculates automatically. You never call C1 — it's always up-to-date. Computed properties work the same way: they track their dependencies and cache the result until those dependencies change.

🏠 Real-Life Analogy — Bank Balance
Analogy — Bank Account Balance
Your bank account balance is never typed manually. It is computed from deposits − withdrawals. Whenever a transaction occurs (dependency changes), the balance updates automatically. The bank doesn't re-compute the balance 100 times if 100 people view it — it caches the result and only re-calculates when a new transaction happens. That's a computed property: cached, derived, auto-updating.
💻 Syntax — Getter & Setter
// Read-only computed (default — getter only)
computed: {
  fullName() {
    return this.firstName + ' ' + this.lastName;
  }
}

// Computed with getter AND setter (rare, but useful)
computed: {
  fullName: {
    get() {
      return this.firstName + ' ' + this.lastName;
    },
    set(newValue) {
      const parts = newValue.split(' ');
      this.firstName = parts[0];
      this.lastName = parts[1];
    }
  }
}
💼 Real-Life Applications
Application 1 — Shopping Cart Total
E-commerce cart: The total price is computed from cartItems.reduce((sum, item) => sum + item.price * item.qty, 0). Every time the user adds, removes, or changes quantities, the total auto-updates. No manual recalculation needed. This is the most common computed property example in industry.
Application 2 — GPA Calculation
Student portal: gpa() { return this.grades.reduce((sum, g) => sum + g.gradePoint * g.credits, 0) / this.totalCredits }. When a new grade is added or an existing one changes, the GPA recalculates automatically. Caching ensures viewing the GPA on multiple pages doesn't re-run the calculation.
Application 3 — Dashboard Statistics
Analytics dashboard: conversionRate() { return this.totalConversions / this.totalVisitors * 100 }. Every time new visitor or conversion data arrives, the rate updates. Computed properties keep dashboard widgets synchronized without manual refresh logic.
⚡ Computed vs Method

Caching✅ Cached until dependencies change❌ Re-executes on every render

Best Practice
If the template accesses a value multiple times, use a computed property to avoid redundant re-execution. Methods re-run on every re-render, even if their inputs haven't changed. Computed properties only re-run when dependencies change.
⚠ Common Mistakes
Common Mistake
Using a computed property for side-effects. Computed properties must only return a value, never trigger API calls, mutate data, or cause side-effects. Vue expects them to be pure functions of their dependencies.
Common Mistake
Mutating computed property dependencies inside the computed function. This creates infinite loops. If fullName sets this.firstName inside its getter, it triggers a re-evaluation, which sets it again — infinite loop.
2 WATCHERS (watch)
🎯 Definition

Watchers explicitly monitor reactive properties and execute callback functions whenever that tracked property changes. They are used primarily to trigger custom imperative code or execute complex asynchronous actions — like making an API request, saving state to local storage, or triggering an animation.

🧠 Intuition

A watcher is like a security camera pointed at a specific door. When the door state changes (opens/closes), the camera triggers a recording. It doesn't produce a value — it performs an action. Same with watchers: they don't return values for the template, they execute side-effect functions.

💻 Syntax — Basic, Deep, Immediate
// Basic watcher
watch: {
  searchQuery(newVal, oldVal) {
    console.log(`Query changed from ${oldVal} to ${newVal}`);
  }
}

// Deep watcher — watches nested object changes
watch: {
  user: {
    handler(newVal) {
      console.log('User object changed', newVal);
    },
    deep: true  // watches nested properties
  }
}

// Immediate watcher — runs ONCE on creation, then on changes
watch: {
  userId: {
    handler(newId) {
      this.fetchUserData(newId);
    },
    immediate: true  // runs on component mount too
  }
}
🏠 Real-Life Analogy
Analogy — Amazon Search Bar
When you type on Amazon's search bar:

l → "l" → API call for suggestions
la → "la" → API call for suggestions
lap → "lap" → API call for suggestions
laptop → "laptop" → API call for suggestions

Each keystroke changes the search query, and that triggers an API call. The API call is a side-effect, not a derived value. This is exactly what watchers are for — watch the search query, and when it changes, make an API request.
💼 Industry Applications
Industry Example 1 — Search-as-you-type
Google Search / Amazon / YouTube: The search input is watched. When the query changes, a debounced API call fetches suggestions. Uses watch with debouncing to avoid flooding the API on every keystroke.
Industry Example 2 — Form Autosave
Google Docs / Notion: A watch on the document content triggers localStorage.setItem('draft', newContent) or an API autosave endpoint. Changes are persisted without the user clicking "Save."
Industry Example 3 — Form Validation
Any signup form: Watch the password field to validate strength in real-time. Watch email to check format. Watch repeat_password to compare with password. Each watcher triggers validation logic and updates error messages.
⚠ Common Mistakes
Common Mistake
Using a watcher when a computed property would do. If you're watching firstName and lastName to set fullName, that's a computed property job. Watchers should be for side-effects, not derived state.
Common Mistake
Forgetting deep: true for nested objects. If you watch: { user: handler } and the user object's nested property changes (user.name), the watcher won't fire unless you add deep: true.
Common Mistake
Not cleaning up watchers. If a watcher sets up a timer, interval, or subscription, the component's beforeDestroy hook must clean it up to prevent memory leaks.

🔄 Decision Tree
Do you need to derive a value from existing data? → Use Computed
Do you need to react to a change and do something (API call, save, validate)? → Use Watcher
Do you need reusable logic called from events or multiple places? → Use Method
Does the value need to be re-evaluated on every render (no caching)? → Use Method
⚠ Common Mistakes — Both
Common Mistake
Using a watcher to compute fullName from firstName + lastName. A computed property does this declaratively with caching. A watcher is imperative, doesn't cache, and requires manual management of the derived state.
Common Mistake
Calling a method from template for derived state. {{ getFullName() }} in the template re-runs on EVERY re-render, even if firstName/lastName haven't changed. Use a computed property for caching.
💡 Best Practices
Best Practice 1
Prefer computed over watcher whenever possible. Computed properties are declarative, cached, and easier to reason about. Watchers should be your last resort for side-effects only.
Best Practice 2
Keep computed properties pure. No API calls, no data mutation, no DOM access. They should only read dependencies and return a value.
Best Practice 3
Use immediate + deep watchers deliberately. immediate: true runs the handler on component creation — useful for fetching initial data. deep: true adds overhead for nested objects — use sparingly.

5 EXECUTION TIMELINE
🔄 Execution Timeline — Computed Caching

Step-by-step walkthrough of how Vue evaluates a computed property and when it decides to re-run vs. use the cache.

1 Template renders and accesses {{ fullName }} for the first time.
2 Vue detects no cached value — executes the computed getter.
3 Getter reads this.firstName and this.lastName — Vue records these as dependencies.
4 Result is cached. Template receives the cached value.
5 Later, this.firstName changes — Vue marks the computed as dirty.
6 Next template access re-runs the getter, records new dependencies, caches new result.
7 If the template accesses the same computed 100 times without a dependency change, it returns the cached value all 100 times — zero re-computation.
Key Insight
Computed properties follow a lazy evaluation pattern — they only recalculate when a dependency actually changes. This is fundamentally different from methods, which re-execute on every render regardless of input changes.
6 DEBUGGING CHECKLIST
🔍 Debugging Checklist

Use this checklist when computed properties or watchers aren't behaving as expected.

Check 1 — Value Not Updating
Computed value shows stale data? Verify that you're reading the property in the template correctly ({{ fullName }}, not {{ fullName() }}). If using a method call, you've lost caching — switch to a computed property.
Check 2 — Infinite Loop
Computed getter causes infinite re-render? You're likely mutating a dependency inside the getter. Remove any this.someDep = ... assignments from computed getters — they must be pure read-only functions.
Check 3 — Watcher Not Firing
Watcher callback never runs? If watching an object, add deep: true. Also confirm the property name matches exactly — typos like user vs userProfile silently fail. Check that the property is reactive (declared in data()).
Check 4 — Watcher Fires Too Often
Watcher runs on every render? You may be watching a non-reactive value or the entire component instance. Narrow your watch target to a specific property, and avoid deep: true on large objects — it triggers on every nested change.
Check 5 — Memory Leak
Component destroyed but watcher side-effects continue? If your watcher starts a timer, interval, or subscription, clean it up in beforeUnmount (Vue 3) or beforeDestroy (Vue 2). Use the watcher's return cleanup function or explicit hook calls.
7 EXAM PATTERNS
📝 Exam Patterns

Common exam-style questions and what examiners look for in answers.

Pattern 1: "When would you use a watcher instead of a computed property?"
Expected answer: When you need to perform side-effects — such as API calls, DOM manipulation, writing to localStorage, or triggering animations. Computed properties are for deriving values; watchers are for reacting to changes with imperative actions. Mention that watchers support async code (e.g., fetch) while computed getters cannot be async.
Pattern 2: "Trace the execution of this code — what happens when firstName changes?"
Expected answer: Identify which computed properties depend on firstName. Explain that Vue marks those computeds as dirty. On next template access, the getter re-runs, new dependencies are recorded, and the cached result updates. Also check if any watcher monitors firstName — its callback fires immediately after the reactive update.
Pattern 3: "What is wrong with this code?" (given a watcher that computes a value)
Expected answer: A watcher that sets a derived property (e.g., watch: { first() { this.full = this.first + ' ' + this.last } }) is an anti-pattern. The fix is to replace it with a computed property — it provides caching, automatic dependency tracking, and is declarative. The watcher version is imperative, doesn't cache, and duplicates what computed does natively.
8 MEMORY TRICKS
🧠 Memory Tricks
Trick 1 — Computed = Calculator
Same inputs always give the same output. Just like a calculator — press 2 + 2 and you always get 4. A computed property takes reactive inputs and returns a consistent derived value. No side effects, no surprises.
Trick 2 — Watcher = Observer
Watches for changes, then reacts. Think of a security guard watching a door — nothing happens until someone walks through, then the guard takes action (sounds an alarm, logs the entry). A watcher sits quietly until its watched property changes, then it fires a callback to perform an action.
Trick 3 — Computed is Lazy
Only recalculates when dependencies change. Like a grocery list that only updates when you add or remove an item — it doesn't re-print itself every time you look at it. Computed properties are evaluated lazily and cached, so repeated accesses cost nothing until a dependency mutates.

10 REVISION CHEAT SHEET
📋 Key Takeaways
  • Computed: Derives + caches a value. Auto-tracks dependencies. Pure function.
  • Watcher: Triggers side-effects on change. Imperative. Good for API calls, save, validation.
  • Method: Reusable logic. No caching. Good for event handlers.
  • Rule of thumb: if it returns a value for the template, use computed. If it triggers an action, use watcher.
  • Computed properties are cached; methods re-run on every render.
  • Watchers have deep and immediate options for advanced scenarios.
  • Never mutate dependencies inside a computed getter — infinite loop risk.
❓ Interview Questions
Q: What is the difference between a computed property and a watcher?
A: A computed property derives a value from reactive dependencies, caches the result, and only re-evaluates when dependencies change — it's declarative and must return a value. A watcher is imperative, executes a callback when a watched property changes, is used for side-effects (API calls, localStorage, etc.), and does not return a value. Use computed when you need a derived value; use watcher when you need to react to a change.
Q: Why would you use a computed property instead of a method?
A: Computed properties are cached — they only re-evaluate when their dependencies change. Methods re-execute on every render, even if their inputs haven't changed. If a value appears multiple times in a template or is expensive to compute, a computed property avoids redundant work.
Q: Can a computed property have a setter?
A: Yes. By default, computed properties are getter-only (read-only). But you can provide both a get and set function. The setter allows you to run logic when the computed value is assigned — typically splitting the assigned value back into its source properties. This is useful for v-model with computed properties.
