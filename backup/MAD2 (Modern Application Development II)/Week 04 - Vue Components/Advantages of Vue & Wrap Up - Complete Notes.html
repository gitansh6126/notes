Why Vue Exists & Problems Solved

Vue.js was created by Evan You in 2014 to address the growing complexity of building interactive web interfaces. It fills a specific niche: approachable enough for beginners, yet powerful enough for large-scale applications. Understanding why Vue exists is the foundation of appreciating every feature it offers.

🏷 The Problem Landscape Before Vue

ProblemTraditional ApproachWhy It Was Painful
DOM manipulation jQuery / document.querySelector Every state change required manual DOM updates. One missed line = bugs.
State & UI sync Manual bookkeeping If user.name changed, you had to update every place it appeared on screen.
Code organisation Scripts scattered in HTML + JS files No clear separation of concerns. Spaghetti code was the default.
Reusability Copy-paste HTML blocks No standard way to package UI widgets with their logic and styles.
Large app performance Full page re-renders Every interaction could trigger expensive reflows; no granular updates.
Learning curve AngularJS / React (early) Angular had a steep barrier; React introduced JSX and a new mental model.

✅ Vue's Solution

  • Reactivity — State changes automatically propagate to the DOM.
  • Declarative templates — You describe what the UI should look like, not how to update it.
  • Component-based — Encapsulated, reusable building blocks.
  • Virtual DOM — Batched, minimal DOM updates for performance.
  • Progressive — Use as much or as little as you need; CDN to full SFC toolchain.
  • Gentle learning curve — Standard HTML, CSS, JS patterns — no JSX required.

💡 Vue's Design Philosophy

  • Approachable — Anyone who knows HTML, CSS, JS can start building.
  • Versatile — Works for simple widgets or full SPA/SSR applications.
  • Performant — Optimised rendering with minimal overhead.
  • Maintainable — Single-File Components keep related code together.
  • Community-driven — MIT licensed, backed by a large ecosystem.
ℹ
Key Insight

Vue doesn't compete with jQuery — it solves a different class of problem. jQuery makes DOM manipulation easier; Vue makes state-driven UI possible. The two can even coexist.

02

Virtual DOM

The Virtual DOM (VDOM) is a lightweight JavaScript representation of the real DOM. It is the core mechanism that enables Vue to update the UI efficiently without touching the browser's expensive DOM API on every change.

🧩 How the Virtual DOM Works

When state changes, Vue does not immediately manipulate the real DOM. Instead:

  1. Render function produces a new VDOM tree based on the current state.
  2. Diff algorithm (patch algorithm) compares the new VDOM tree with the previous one.
  3. Patch — only the differences are applied to the real DOM in a single batch.
Virtual DOM pipeline: state change → new VDOM tree → diff → patch → real DOM update

📊 Real DOM vs Virtual DOM

AspectReal DOMVirtual DOM (Vue)
TypeBrowser's actual HTML node treeLightweight JS object tree
Manipulation costHigh — triggers layout, paint, compositingLow — just creating plain objects
Update strategyEach change applied immediatelyChanges batched, diffed, applied as one patch
Read/Write cycleMixed → layout thrashingSeparated → no thrashing
API surfaceHundreds of properties/methodsMinimal JS object shape (tag, props, children)
💡
Did You Know?

Vue 3's VDOM implementation is compiler-optimized. During compilation, Vue detects static content and hoists it outside the render function — it never needs to be re-created on every re-render. This gives Vue a significant performance advantage over libraries that do pure runtime VDOM.

Reactivity System

Vue's reactivity system is the engine that automatically tracks state dependencies and re-renders only what changed. It is the single biggest advantage Vue offers over imperative DOM manipulation.

Vue 3 Reactivity System: Proxy-based getter/setter tracking with automatic dependency management

⚙ Vanilla JS vs Vue Reactivity

ScenarioVanilla JSVue.js
Update a counter counter++ then el.textContent = counter Just this.counter++ — Vue updates the DOM automatically
Toggle a class el.classList.toggle('active') :class="{ active: isActive }" — reactive binding
Filter a list Loop, create new elements, append, remove old ones v-for with computed filter — auto re-renders

Component Architecture

Components are the fundamental building blocks of a Vue application. Each component encapsulates its own template, logic, and styles into a self-contained unit. This architecture is what makes Vue applications maintainable, testable, and scalable.

🧰 Component Hierarchy & Data Flow

📦 Single-File Component (SFC) Anatomy

MyComponent.vue — the standard SFC formatVue SFC
<template>
  <div class="my-component">
    <h3>{{ title }}</h3>
    <p v-if="description">{{ description }}</p>
    <button @click="handleClick">
      Clicked {{ count }} times
    </button>
  </div>
</template>

<script>
export default {
  name: 'MyComponent',
  props: {
    title: String,
    description: String
  },
  data() {
    return { count: 0 }
  },
  methods: {
    handleClick() {
      this.count++
      this.$emit('count-changed', this.count)
    }
  }
}
</script>

<style scoped>
.my-component { border: 1px solid #ddd; padding: 1rem; }
h3 { color: #1e4f8a; }
</style>

⬆ Props (Parent → Child)

  • Read-only — child must never mutate a prop directly.
  • Typed — validate with String, Number, Array, custom validators.
  • Required / default — mark props as required or provide defaults.
  • Reactive — changes in parent propagate automatically.

⬇ Events (Child → Parent)

  • $emit — child fires event with data payload.
  • @event-name — parent listens with v-on.
  • Custom events — well-defined API contract between components.
  • emit("update:prop") — enables v-model on custom components.

Maintainability

Vue's design choices directly contribute to long-term code maintainability — arguably the most important quality metric for real-world applications that evolve over years.

📝 How Vue Promotes Maintainability

Separation of Concerns SFC: <template>, <script>, <style> per file Each component is a self-contained unit. Find, fix, test in one place.
Declarative over Imperative Templates, directives, data binding Less code to write and read. Intent is clear.
Component Isolation Props down, events up, scoped styles Change one component without fear of breaking others.

⚠
Common Maintainability Pitfall

Putting too much logic in a single component. If a component has more than ~200 lines, consider splitting it. Use the "one responsibility per component" rule — if a component does two unrelated things, refactor.

Performance

Vue is designed for performance out of the box. Its architecture makes deliberate trade-offs to deliver fast initial render, efficient updates, and small bundle size.

Relative rendering performance: Vue 3 VDOM with compiler optimisation dramatically reduces DOM update time

⚡ Key Performance Features

  • Tree-shaking — unused features removed at build time.
  • Static hoisting — static template parts rendered once.
  • Patch flags — compiler marks dynamic nodes for fast diff.
  • Cache handlers — inline event handlers cached.
  • Props stability — stable props skip child re-render.
  • v-once / v-memo — explicit caching directives.
  • Async component — lazy load via defineAsyncComponent.
  • Suspense — coordinate async dependencies.

State Management Overview

As applications grow, managing shared state across many components becomes challenging. Vue provides a tiered approach: from simple component-local state to full centralised stores.

State management maturity model: from simple to enterprise-grade

📊 State Management Comparison

ApproachWhen to UseProsCons
Local state Form input, toggle, counter Simple, encapsulated Can't share across siblings
Props + $emit Parent-child communication Explicit data flow Prop drilling in deep trees
Provide / Inject Themes, i18n, user context No drilling, reactive Source unclear, debugging harder
Pinia Store Auth state, cart, global prefs DevTools, plugins, modular Boilerplate for small apps
Pinia Store ExampleJS
// stores/counter.js
import { defineStore } from 'pinia'

export const useCounterStore = defineStore('counter', {
  state: () => ({ count: 0 }),
  getters: {
    double: (state) => state.count * 2
  },
  actions: {
    increment() {
      this.count++
    }
  }
})

// In a component:
const store = useCounterStore()
store.increment()
console.log(store.count)      // 1
console.log(store.double)      // 2

Common Misconceptions

Misunderstandings about Vue are widespread. Here are the most common ones, corrected.

✖ "Vue is just a library, not a framework"
✔ Truth: Vue is a progressive framework. It can be used as a library (CDN, single widget) or as a full framework (CLI, SFCs, router, store, SSR). It grows with your needs.
✖ "Vue's reactivity is magic / unpredictable"
✔ Truth: Vue's reactivity is entirely deterministic. Proxy traps track getter/setter access. You can predict exactly when updates occur. Vue 3's Proxy-based system has zero caveats.
✖ "Vue is only for small projects"
✔ Truth: Vue powers production applications at Alibaba, Xiaomi, GitLab, Nintendo, and thousands of enterprise systems. Nuxt + Pinia + TypeScript scale to 1000+ components.
✖ "Virtual DOM means Vue never touches the real DOM"
✔ Truth: Vue does touch the real DOM — but only to apply the minimal set of changes computed by the diff algorithm. The VDOM is an optimisation layer, not a replacement.
✖ "Vue components are like Web Components"
✔ Truth: Different standards. Vue SFCs are compiled components with scoped CSS via attribute selectors. Web Components use Shadow DOM. Vue can compile to Web Components.
✖ "You must use the CLI/build tools to use Vue"
✔ Truth: Vue works from a single HTML file with a CDN script tag. The build toolchain is optional. Start with CDN, add build tools when your project grows.

Interview Questions

Common Vue interview questions — click to reveal answers.

Q1: Explain the Virtual DOM and how Vue uses it. +

The Virtual DOM is a lightweight JavaScript object tree that mirrors the real DOM structure. When state changes, Vue:

  1. Creates a new VDOM tree from the render function.
  2. Diffs it against the previous tree using an optimized O(n) algorithm.
  3. Computes the minimal set of DOM operations needed.
  4. Applies them in a single batched update.

This avoids expensive layout thrashing and makes updates predictable. Vue 3's compiler further optimises this by hoisting static content and using patch flags.

Q2: How does Vue's reactivity differ from React's setState? +

Vue reactivity: Proxies intercept get/set at the property level. When you set this.count = 5, Vue immediately knows which effects depend on count and queues them. No manual trigger needed.

React setState: You call setState({ count: 5 }) to tell React that something changed. React then re-renders the entire component tree (with virtual DOM diffing). React relies on explicit state update calls.

Key difference: Vue is push-based (reactivity pushes updates); React is pull-based (reconciliation pulls changes).

Q3: Explain the component lifecycle in Vue. +

Vue components go through these lifecycle hooks (in order):

  1. beforeCreate — component initialized, data not reactive yet
  2. created — data reactive, but DOM not mounted
  3. beforeMount — template compiled, about to mount
  4. mounted — component added to DOM — safe to access DOM
  5. beforeUpdate — state changed, DOM not yet patched
  6. updated — DOM patched and re-rendered
  7. beforeUnmount — about to destroy
  8. unmounted — component destroyed, cleanup done

In Vue 3 Composition API, use onMounted(), onUpdated(), etc.

Q4: What is the difference between v-if and v-show? +
v-ifv-show
Conditionally creates/destroys the DOM elementAlways renders, toggles display: none
Lazy — does not render unless condition is trueEager — always renders regardless of visibility
Higher toggle cost (create/destroy cycle)Lower toggle cost (just CSS change)
Use for rare, conditional sectionsUse for frequently toggled elements
Q5: How does the Composition API differ from the Options API? +

Options API: Organises code by option type: data(), methods, computed, watch, lifecycle hooks. Best for small-to-medium components. Simple and familiar.

Composition API: Organises code by feature using setup() or <script setup>. Groups related logic together. Better for complex components, custom composables, and TypeScript inference.

Both are fully supported in Vue 3. Use Options for simplicity, Composition for flexibility and reuse.

Q6: Explain unidirectional data flow in Vue. +

Data flows down from parent to child via props. Child components never mutate props directly. Instead, they send events up via $emit for the parent to handle.

This creates a predictable cycle: parent owns state → passes it as props → child displays/interacts → child emits event → parent updates state → new props flow down.

Benefits: debugging clarity (you know where state changes originate), no unexpected mutations, easy testing (components are pure functions of props).

Q7: What is Pinia and why was it preferred over Vuex? +

Pinia is the official state management library for Vue 3. It replaced Vuex because:

  • Full TypeScript support without extra declarations.
  • No mutations — just state + actions (simpler mental model).
  • Multiple stores (no single root store limitation).
  • DevTools support, HMR, and plugin system.
  • Composition API native — works seamlessly with <script setup>.

Exam Questions

Practice questions for the MAD2 exam. Try answering before revealing the solution.

Exam Q1: Describe the difference between reactive() and ref() in Vue 3. +

ref() wraps a value in a reactive object with a .value property. Works for primitives (strings, numbers, booleans) and objects. Template auto-unwraps (no .value needed in template).

reactive() takes an object and returns a deep reactive proxy. No .value needed. Cannot hold primitives directly. Limited to objects, arrays, Maps, Sets.

Rule of thumb: Use ref() for primitives and when you need reassignment; use reactive() for object collections. Many developers now use ref() exclusively for consistency.

Exam Q2: Explain what v-model does and how it works internally. +

v-model is a directive for two-way data binding. It's syntactic sugar for:

<!-- This: -->
<input v-model="name">

<!-- Is equivalent to: -->
<input
  :value="name"
  @input="name = $event.target.value"
>

On custom components, v-model defaults to a modelValue prop and update:modelValue event.

Exam Q3: What is the purpose of the :key attribute in v-for? +

The :key attribute gives Vue a hint to identify which items have changed, been added, or removed during re-rendering. Without it, Vue uses an in-place patch strategy that can cause bugs with:

  • Component state (wrong component instance reused)
  • List animations (items animate incorrectly)
  • Input values (input state preserved on wrong row)

Always use a unique, stable identifier (like :key="item.id"), never the array index unless the list is static and never reordered.

Exam Q4: How would you share state between two sibling components? +

There are three common approaches:

  1. Lift state up — move the shared state to the nearest common ancestor, pass it down as props, and emit events up. Best for simple cases.
  2. Provide / Inject — the ancestor provides the state using provide(), descendants inject it with inject(). Useful for deep trees but less explicit.
  3. Pinia store — centralise the state in a store. Both components import the same store. Best for complex state shared across many components.
Exam Q5: What is the difference between computed and watch? +

computed: Derives a value from other reactive state. It caches the result and only recomputes when dependencies change. Used in templates as a value. Must return something.

watch: Performs a side effect when a reactive value changes. Used for things like: logging, API calls, syncing with localStorage, triggering animations. Does not return a value.

Rule: Can you derive it from existing state? Use computed. Do you need to react with a side effect? Use watch.

Exam Q6: Explain nextTick() and when you would use it. +

nextTick() is a utility that waits for the next DOM update cycle to complete. Vue batches state changes asynchronously. If you change reactive state and immediately try to read the DOM, you'll get the old value.

await nextTick()
// DOM is now updated

Use cases: reading DOM measurements after a change, integrating with non-Vue libraries that need the DOM, testing assertions after state updates.

Final Cheat Sheet

Quick-reference cards for the most important Vue patterns, directives, and APIs.

🔡 Template Interpolation

{{ expression }}
Renders the value of any JS expression in the template.

🔗 Attribute Binding

:attr="value"
Shorthand for v-bind:attr. Dynamically binds an attribute.

💡 Event Binding

@event="handler"
Shorthand for v-on:event. Attaches event listeners.

🔄 Two-Way Binding

v-model="state"
Syntactic sugar for :value + @input. Works on inputs, selects, textareas, and custom components.

📃 Conditional Rendering

v-if="cond" / v-else-if / v-else
v-show="cond"
v-if creates/destroys; v-show toggles display.

📋 List Rendering

v-for="item in items" :key="item.id"
Renders a list. Always provide a unique :key.

🎨 Dynamic Classes

:class="{ active: isActive }"
Object syntax; keys are classes, values are booleans.

🎨 Dynamic Styles

:style="{ color: activeColor, fontSize: size + 'px' }"
CamelCase properties; object or array syntax.

📦 Props Declaration

props: { title: String, count: { type: Number, required: true } }
Typed, validated, with required/default support.

📤 Custom Events

this.$emit('event-name', payload)
Child fires; parent listens with @event-name.

📊 Computed Property

computed: { double() { return this.count * 2 } }
Cached derived value. Recomputes only when dependencies change.

🔍 Watcher

watch: { count(newVal, oldVal) { ... } }
Reacts to state changes with side effects.

📍 Template Refs

<div ref="myRef">
Access DOM element or child component instance via this.$refs.myRef.

🔄 Lifecycle (Composition)

onMounted(() => { ... })
Lifecycle hooks prefixed with on in Composition API.

🎯 Vue 3 ref()

const count = ref(0) → count.value
Reactive primitive wrapper. Auto-unwrapped in templates.

🎯 Vue 3 reactive()

const state = reactive({ count: 0 })
Deep reactive proxy for objects. No .value needed.

💳 Pinia Store

defineStore('id', { state, getters, actions })
Official state management. Modular, Typed, DevTools-enabled.

🌊 Provide / Inject

provide('key', value) / inject('key')
Share state across deep component trees without prop drilling.

⏳ nextTick

await nextTick()
Wait for DOM update after reactive change.

🚀 Async Component

defineAsyncComponent(() => import('./Comp.vue'))
Lazy-load components for code splitting.

Glossary

Virtual DOMA lightweight JavaScript object tree that mirrors the real DOM, used to compute minimal updates.
ReactivityThe ability of a framework to automatically track state dependencies and update the UI when state changes.
ComponentA self-contained, reusable piece of UI with its own template, logic, and styles.
SFC (Single-File Component)A .vue file that encapsulates template, script, and style in one file.
PropsCustom attributes passed from parent to child component. Read-only in the child.
$emitA method that fires a custom event from child to parent, optionally with a payload.
DirectiveSpecial HTML attributes prefixed with v- that apply reactive behaviour (e.g., v-if, v-for).
v-modelA directive for two-way data binding, shorthand for :value + @input.
Computed PropertyA derived value that caches its result and only recomputes when dependencies change.
WatcherA function that performs side effects when a reactive dependency changes.
PiniaThe official Vue 3 state management library with modular stores, TypeScript support, and DevTools.
Composition APIA set of functions (ref, reactive, computed, lifecycle hooks) for organising component logic by feature.
Options APIThe traditional Vue API that organises component logic by option blocks: data, methods, computed, etc.
Provide / InjectA mechanism for a parent component to provide data to all descendants regardless of depth.
Tree-shakingA build-time optimisation that removes unused code from the final bundle.
SuspenseA built-in component that coordinates async dependencies and shows fallback content while loading.

Module Summary & Takeaways

A concise wrap-up of everything covered in the Vue.js module (Weeks 03 – 04).

📚 What You Have Learned

  • Why Vue exists: To solve the pain of manual DOM manipulation with a reactive, declarative, component-based architecture.
  • Virtual DOM: An optimisation layer that diffs JS object trees and applies minimal, batched real-DOM updates.
  • Reactivity: Vue 3's Proxy-based system automatically tracks dependencies and triggers updates — no manual state management needed.
  • Components: Encapsulated SFCs with props (down) and events (up) create a predictable, maintainable architecture.
  • State Management: From local data() to Pinia stores — choose the right tool for your scope.
  • Performance: Compiler-optimised VDOM, tree-shaking, async components, and lazy loading keep apps fast.
  • Scalability: Composition API, TypeScript, monorepo patterns, and SSR via Nuxt scale to enterprise applications.
  • Maintainability: SFCs, scoped styles, unidirectional data flow, and DevTools make code easy to reason about and refactor.

✅ Do This

  • Use v-for with a unique :key
  • Keep components small and focused
  • Use computed for derived values
  • Use v-show for frequent toggles
  • Validate props with types and defaults
  • Start with Options API, graduate to Composition
  • Use Pinia only for truly shared state

✖ Avoid This

  • Mutating props in child components
  • Putting all state in a store
  • Using v-if + v-for on same element
  • Overusing watchers when computed suffices
  • Ignoring :key in lists
  • Mixing Vue 2 and Vue 3 APIs
  • Writing business logic in templates
