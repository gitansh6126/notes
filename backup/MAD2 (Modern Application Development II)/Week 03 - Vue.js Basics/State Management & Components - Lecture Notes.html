Introduction to Web Frontend

The web frontend is everything the user sees and interacts with in a browser. It is the client side of a web application — the HTML, CSS, and JavaScript that render the user interface.

📡 What is a Web Frontend?

When you visit a website, your browser downloads HTML, CSS, and JavaScript files from a server. The browser then interprets these files to render a page you can see and interact with. That rendered interface is the frontend.

  • HTML — Structure (headings, paragraphs, buttons)
  • CSS — Style (colours, layout, animations)
  • JavaScript — Behaviour (click handlers, API calls, DOM updates)
ℹ
Key Insight

The frontend runs on the client's machine (browser), not on the server. This is why modern frameworks like Vue.js exist — they help manage complexity on the client side.

📜 The Three-Layer Architecture

Every web frontend can be understood as three layers:

LayerTechnologyRoleAnalogy
Structure HTML Defines elements on the page Bones of a body
Presentation CSS Controls appearance & layout Skin & clothing
Behaviour JavaScript Handles logic, events, data Nervous system

<div id="app">
  <p>Count: <span id="count">0</span></p>
  <button onclick="increment()">+1</button>

</script>
💡
Real-Life Analogy: A Restaurant

HTML = The menu (structure of items). CSS = The plating & decor (presentation). JavaScript = The waiter who takes your order and brings food (behaviour). The server = The kitchen where food is prepared.

02

Web Accessibility & Aesthetics

Accessibility (a11y) ensures that people with disabilities can perceive, understand, navigate, and interact with the web. Aesthetics ensures the experience is visually pleasing and usable for everyone.

♿ Why Accessibility Matters

  • Legal — Many countries mandate WCAG compliance (e.g., ADA, EU Web Accessibility Directive)
  • Ethical — The web should be available to everyone regardless of ability
  • Business — 15% of the world's population has some form of disability; accessibility also improves SEO
  • Technical — Accessible sites tend to have cleaner, more semantic code

📋 WCAG Guidelines (Quick Reference)

WCAG 2.1 is organised around four principles (POUR):

PrincipleMeaningExample
Perceivable Users must be able to perceive the content Alt text on images, captions on videos
Operable Users must be able to operate the interface Keyboard navigation, enough time to read
Understandable Content and UI must be understandable Clear labels, predictable behaviour
Robust Content must work with current & future tools Semantic HTML, ARIA attributes

🎨 Aesthetics & User Experience (UX)

Aesthetics is not just about "looking pretty" — it directly affects usability:

  • Visual Hierarchy — Important elements should stand out (size, colour, spacing)
  • Consistency — Similar elements should look and behave the same way
  • Feedback — Users should know when an action succeeded or failed
  • Affordance — Buttons should look clickable; links should look like links
  • Colour Contrast — Text must have sufficient contrast against its background (WCAG AA: 4.5:1)
⚠
Common Accessibility Mistake

Using <div onclick="..."> instead of a <button>. Screen readers don't know a <div> is interactive. Always use semantic HTML elements.

💡
Pro Tip

Use the <label> element with for attribute on every form input. It makes the form accessible to screen readers and also increases the clickable area — a win-win.

Accessible form example HTML 
<!-- Good: accessible -->
<label for="email">Email address</label>
<input type="email" id="email" name="email" required>

<!-- Bad: inaccessible -->
<span>Email:</span>
<div contenteditable></div>

🔗 Analogy: Accessibility is like Curb Cuts

Curb cuts (sloped kerbs at crosswalks) were designed for wheelchair users but are now used by parents with strollers, cyclists, delivery workers, and everyone else. Similarly, accessibility features help everyone — captions help in noisy environments, high contrast helps in bright sunlight, keyboard navigation helps power users.

Problem with Stateful Protocol

HTTP is a stateless protocol — each request is independent and carries no memory of previous requests. But modern web applications need to remember who you are, what's in your cart, and your preferences. This creates the state management problem.

🔄 Stateful vs Stateless

CharacteristicStatelessStateful
DefinitionEach request is independentServer remembers past requests
ExampleHTTP GET a web pageLogged-in session on a bank site
ScalabilityEasy (any server can handle any request)Harder (need sticky sessions or shared state)
ResilienceHigh (server crash = no lost data)Lower (server crash can lose session)
ComplexityLowMedium to High

⚠ The Core Problem

HTTP was designed for documents (stateless), but modern web apps are applications (stateful). The mismatch causes several problems:

  • Authentication — How does the server know who you are on each request?
  • Session Data — Cart items, form progress, user preferences must be stored somewhere
  • Consistency — What happens when state changes on the server vs the client?
  • Performance — Sending all state on every request wastes bandwidth

🔧 Solutions to the State Problem

SolutionHow it WorksProsCons
Cookies Small data stored in browser, sent with every request Simple, widely supported Limited size (4 KB), security risks (XSS, CSRF)
Sessions Session ID in cookie; data stored server-side Secure (data not exposed to client) Server memory usage, sticky sessions needed
JWT Signed token containing user data Stateless, scalable Token size, revocation is hard
LocalStorage Key-value storage in the browser Persistent, 5-10 MB space Not sent automatically, no server-side access
Client-side State Mgmt Vuex/Pinia (Vue), Redux (React) Reactive, predictable, devtools Boilerplate, learning curve
ℹ
Why this matters for Vue.js

Vue.js applications run entirely in the browser. Managing state across different components is one of the hardest challenges in frontend development. This is why Vue offers tools like data(), computed, Vuex, and Pinia.

💡
Real-Life Analogy: A Bank Visit

Stateless = Walking into a bank where every teller asks "Who are you?" even if you just spoke to the person next to them. Stateful = The bank gives you a token number and any teller can look up your details using that token. The token is like a session cookie.

Saving state with localStorage (vanilla JS) JavaScript 
// Save state
const cart = ["item1", "item2"];
localStorage.setItem('cart', JSON.stringify(cart));

// Retrieve state later
const savedCart = JSON.parse(
  localStorage.getItem('cart') || '[]'
);
console.log(savedCart); // ["item1", "item2"]

Introduction to Vue Components

A component is a reusable, self-contained piece of UI that encapsulates HTML, CSS, and JavaScript. Components are the building blocks of any Vue application.

🧰 What is a Component?

Think of components like Lego bricks. Each brick has its own shape, colour, and purpose. You combine them to build complex structures. Each component:

  • Has its own template (HTML)
  • Has its own logic (JavaScript)
  • Has its own style (CSS) — optionally scoped
  • Can receive data from parent (props)
  • Can send events to parent (emit)

📦 Props: Passing Data Down

Props are custom attributes you can register on a component. They allow a parent component to pass data down to a child component.

Parent component passing props Vue 
<!-- Parent.vue -->
<template>
  <ArticleCard
    title="Vue.js Basics"
    :likes="42"
    :published="true"
  />
</template>

<script>
import ArticleCard from './ArticleCard.vue';
export default {
  components: { ArticleCard }
}
</script>
Child component receiving props Vue 
<!-- ArticleCard.vue -->
<template>
  <div class="card">
    <h3>{{ title }}</h3>
    <p>Likes: {{ likes }}</p>
    <p v-if="published">✓ Published</p>
  </div>
</template>

<script>
export default {
  props: {
    title: String,
    likes: Number,
    published: Boolean
  }
}
</script>

📤 Events: Communicating Up

When a child component needs to tell its parent about something (a button click, a form submit), it emits an event.

Child emits an event Vue 
<!-- ChildButton.vue -->
<template>
  <button @click="handleClick">
    {{ label }}
  </button>
</template>

<script>
export default {
  props: { label: String },
  methods: {
    handleClick() {
      this.$emit('clicked', 'Button was pressed!');
    }
  }
}
</script>

<!-- Parent listens -->
<ChildButton
  label="Click me"
  @clicked="handleChildClick"
/>

📊 Components vs. A Traditional Page

AspectTraditional (jQuery)Component-based (Vue)
Code organisationOne big JS fileSmall, reusable files
Data flowGlobal variablesProps down, events up
ReusabilityCopy-pasteImport & use
StylingGlobal CSS (cascade issues)Scoped CSS per component
TestingHard (monolithic)Easy (isolated units)
State managementDOM as state (bad)Reactive data layer
Full component example — TodoItem Vue 
<!-- TodoItem.vue -->
<template>
  <div class="todo" :class="{ completed: done }">
    <input type="checkbox" :checked="done" @change="toggle" />
    <span>{{ text }}</span>
    <button @click="remove">✖</button>
  </div>
</template>

<script>
export default {
  props: {
    text: { type: String, required: true },
    done: { type: Boolean, default: false }
  },
  methods: {
    toggle() { this.$emit('toggle'); },
    remove() { this.$emit('remove'); }
  }
}
</script>

<style scoped>
.todo { display: flex; align-items: center; gap: 0.5rem; padding: 0.5rem; }
.completed span { text-decoration: line-through; opacity: 0.6; }
</style>
ℹ
Unidirectional Data Flow

Data in Vue flows in one direction: parent to child via props. Children cannot directly modify props. When a child needs to change something, it emits an event, and the parent handles the change. This makes applications easier to debug and reason about.

Common Mistakes Students Make

✖ Mistake 1: Mutating Props Directly

Wrong: Trying to change a prop's value inside the child component.

WrongVue
// Child component
props: ['count'],
methods: {
  increment() {
    this.count++; // ✖ BAD: props are read-only!
  }
}

Fix: Emit an event and let the parent update the value.

✖ Mistake 2: Forgetting v-bind: (or :)

Without v-bind:, Vue treats the value as a string literal, not a JavaScript expression.

ConfusionVue
<!-- Passes STRING "42", not NUMBER 42 -->
<ArticleCard likes="42" />

<!-- Passes NUMBER 42 (correct) -->
<ArticleCard :likes="42" />

✖ Mistake 3: Overusing v-if vs v-show

  • v-if — Removes/adds the element from the DOM (costly for toggling)
  • v-show — Always renders, just toggles display: none (cheaper for frequent toggling)

Common error: Using v-if inside a frequent toggle (e.g., a dropdown) which causes performance issues.

✖ Mistake 4: Not Using :key in v-for

Vue uses :key to track list items for efficient re-rendering. Without it, Vue may reuse DOM elements incorrectly, causing bugs.

FixVue
<!-- Always provide a unique key -->
<li v-for="item in items" :key="item.id">
  {{ item.text }}
</li>

✖ Mistake 5: Putting Async Logic in computed Properties

Computed properties must be synchronous and have no side effects. Students often try to fetch data inside a computed property, which won't work.

  • Use computed for synchronous derived data (e.g., fullName from firstName + lastName)
  • Use watchers or lifecycle hooks for async operations (e.g., API calls in mounted())
06

Interview & Exam Questions

Common questions with concise answers to help you prepare for interviews and exams.

Q1: What is the difference between stateful and stateless protocols? +

Stateless protocols (like HTTP) treat each request independently — the server does not remember previous requests. Stateful protocols (like WebSockets or SSH) maintain session state across requests. HTTP solves this with cookies, sessions, and JWTs to simulate statefulness on top of a stateless protocol.

Q2: What is WCAG and what are its four principles? +

WCAG (Web Content Accessibility Guidelines) is the international standard for web accessibility. The four principles (POUR) are:

  • Perceivable — Content must be presentable to at least one sense
  • Operable — UI components must be usable
  • Understandable — Information and UI must be clear
  • Robust — Content must work across assistive technologies
Q3: What are Vue components and why use them? +

A Vue component is a reusable, self-contained unit of UI with its own template, script, and style. Benefits:

  • Reusability — Write once, use anywhere
  • Encapsulation — Scoped CSS prevents style leaks
  • Maintainability — Each component is a small, focused piece
  • Testability — Isolated components are easy to unit test
  • Team work — Multiple developers can work on different components
Q4: Explain props and events in Vue. +

Props are how a parent passes data down to a child. Props are read-only in the child. Events are how a child communicates up to a parent using $emit. This creates a unidirectional data flow — data flows down, events flow up — making the application predictable and easier to debug.

Q5: What is the difference between v-if and v-show? +

v-if conditionally renders the element (adds/removes from DOM). v-show always renders the element but toggles display: none. Use v-if for rare/conditional toggles; use v-show for frequent toggling where the element is often visible.

Q6: Why is accessibility important in web development? +

Accessibility is important because: (1) Legal compliance — many countries mandate WCAG; (2) Ethical responsibility — the web should be usable by everyone; (3) Business reach — ~15% of people have a disability; (4) SEO benefits — semantic HTML helps search engines; (5) Better UX for all — captions, keyboard nav, and contrast help everyone.

Q7: How does Vue manage component state? +

Vue manages state through its reactive data system. Each component has a data() function that returns an object; Vue wraps each property in a reactive proxy. When data changes, Vue automatically re-renders only the parts of the DOM that depend on that data. For complex cross-component state, Vue offers Vuex (Vue 2) or Pinia (Vue 3) as centralized state stores.

Q8: What are the three layers of a web frontend? +

The three layers are: (1) HTML — structure and content (like a building's frame); (2) CSS — presentation and layout (like paint and furniture); (3) JavaScript — behaviour and interactivity (like electricity and plumbing). Together they create the complete user experience.

07

Key Takeaways

  • Web Frontend = HTML (structure) + CSS (presentation) + JS (behaviour), all running in the browser.
  • Accessibility (a11y) follows the POUR principles: Perceivable, Operable, Understandable, Robust. Benefits everyone, not just users with disabilities.
  • HTTP is stateless — the server forgets you after every request. State is simulated via cookies, sessions, JWTs, or client-side storage.
  • Vue components are reusable self-contained UI blocks with a template, script, and scoped style.
  • Props flow down (parent to child); events flow up (child to parent via $emit). This is called unidirectional data flow.
  • Never mutate props directly in a child component. Emit an event and let the parent handle the change.
  • Use :key in v-for loops for efficient re-rendering.
  • Accessible code is better code — semantic HTML, proper labels, keyboard navigation, and good contrast ratios improve the experience for everyone.

Summary

📖 Lecture at a Glance

This lecture covers the foundational concepts you need before diving into Vue.js components:

  • The web frontend is made of HTML, CSS, and JavaScript — the three-layer architecture.
  • Accessibility (WCAG POUR principles) and aesthetics (visual hierarchy, consistency, contrast) are essential to good frontend design.
  • HTTP's stateless nature creates a state management problem that every web application must solve.
  • Cookies, sessions, JWTs, and client-side storage are common solutions to the state problem.
  • Vue.js solves frontend complexity with components — reusable, encapsulated pieces of UI.
  • Components communicate via props (down) and events (up), enforcing a unidirectional data flow.
  • This component model makes applications scalable, testable, and maintainable.

Glossary

Accessibility (a11y) The practice of making websites usable by people of all abilities, including those with visual, hearing, motor, or cognitive impairments.
Component A reusable, self-contained unit of UI in Vue.js that encapsulates template (HTML), script (JS), and style (CSS).
Cookie A small piece of data (max ~4 KB) stored in the browser and sent with every HTTP request to the same domain.
Emit A Vue mechanism that allows a child component to send a custom event (with optional payload) to its parent.
JWT (JSON Web Token) A digitally signed token containing user data, used for stateless authentication. The server verifies the signature rather than storing session data.
Prop A custom attribute registered on a Vue component that allows a parent to pass data down to a child. Props are read-only in the child.
Reactivity Vue's system that automatically tracks data dependencies and updates the DOM when data changes.
Session A server-side store of user state, identified by a session ID typically stored in a cookie.
Stateful Protocol A communication protocol where the server retains information (state) about previous interactions with the same client.
Stateless Protocol A communication protocol where each request is treated independently, with no memory of previous requests (e.g., HTTP).
Unidirectional Data Flow A design pattern where data moves in a single direction — from parent to child via props — and changes flow back up via events.
WCAG Web Content Accessibility Guidelines — the international standard for web accessibility, organised around the POUR principles.
v-if / v-show Vue directives for conditional rendering. v-if adds/removes from DOM; v-show toggles CSS display.
