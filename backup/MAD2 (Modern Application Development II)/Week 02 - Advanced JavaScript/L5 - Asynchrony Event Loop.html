Asynchrony & the Event Loop

JavaScript is single-threaded but non-blocking. The event loop orchestrates deferred work — timers, network calls, and UI updates — without freezing the main thread.

1 Sync vs Async & the Call Stack

The Call Stack — One Thing at a Time

The call stack tracks function execution. Each function call adds a frame; each return removes it. Stack overflow = too many frames.
function a() { b(); }
function b() { c(); }
function c() { console.log('done'); }

a();
// Stack: global → a → b → c (then pops in reverse)

// Blocking — if c() took 5 seconds, nothing else runs
function block() {
  const start = Date.now();
  while (Date.now() - start < 5000) {} // blocks for 5s
  console.log('unblocked');
}
Real-World Analogy
The call stack is like a stack of plates at a buffet: you can only wash one plate at a time. If you stop to scrubs a single plate for 5 minutes, every other plate in the stack waits. That's blocking — and no new plates can even be placed on the stack.

Non-Blocking with setTimeout

setTimeout doesn't pause execution — it schedules a callback to run after at least N milliseconds, then continues immediately.
console.log(1);
setTimeout(() => console.log(2), 0); // 0ms delay
console.log(3);

// Output: 1, 3, 2 (NOT 1, 2, 3)
// Why even setTimeout(0) is deferred

setInterval(() => console.log('tick'), 1000);
// Runs callback every ~1000ms (not guaranteed exact)
Common Mistake
Assuming setTimeout(fn, 0) runs immediately. It only guarantees a minimum delay — the callback still waits for all pending code in the current execution context to finish and its turn in the task queue.
2 The Event Loop — Complete Picture

Event Loop with All Queues

The event loop continuously checks: is the call stack empty? If yes, process microtasks, then pick the next macrotask from the task queue.
Microtask Priority
The event loop processes the entire microtask queue before moving to the next macrotask. This is why Promise.resolve().then(fn) always runs before a scheduled setTimeout — even when both are ready at the same time.
Macrotask (Task Queue)
setTimeout, setInterval, I/O callbacks, UI rendering events. Processed one per event loop cycle. Lower priority than microtasks.
Microtask
Promise.then/catch/finally, queueMicrotask(), MutationObserver. Processed immediately after the current task finishes — entire queue is drained before next macrotask.
3 Callback Pattern & Callback Hell

The Nested Callback Problem

Before Promises, async operations used callbacks. Nesting multiple callbacks created the infamous "callback hell" pyramid.
// Callback hell — deeply nested, hard to read, error-prone
getUser('alice', (err, user) => {
  if (err) { return console.error(err); }
  getPosts(user.id, (err, posts) => {
    if (err) { return console.error(err); }
    getComments(posts[0].id, (err, comments) => {
      if (err) { return console.error(err); }
      console.log(comments);
    });
  });
});

// Each level adds a new indentation — the "pyramid of doom"
// Solution: Promises + async/await flatten this to linear code
Industry Context
Modern codebases use async/await exclusively for async control flow. Callback patterns survive only in event emitters (on('data', cb)) and some Node.js legacy APIs. The util.promisify wrapper converts callback-based APIs to Promise-based ones.

requestAnimationFrame

Schedules code to run before the next browser repaint. The browser batches all rAF callbacks in a single frame — ideal for animations.
function animate(timestamp) {
  element.style.transform = `translateX(${Math.sin(timestamp / 1000) * 50}px)`;
  requestAnimationFrame(animate);  // schedule next frame
}
requestAnimationFrame(animate);

// Unlike setInterval(fn, 16) — rAF pauses when tab is inactive
// rAF fires ~60 times/sec (matches display refresh rate)
4 Interview Q&A
Q: Explain the difference between macrotasks and microtasks with an example.
A: console.log(1); setTimeout(() => console.log(2)); Promise.resolve().then(() => console.log(3)); console.log(4); — Output: 1, 4, 3, 2. Synchronous code runs first (1,4), then microtask queue drains (3), then the next macrotask runs (2).
Q: Is setTimeout(fn, 0) truly zero delay?
A: No. Browsers enforce a minimum delay of 4ms for nested timeouts (HTML spec). Even at 0ms, the callback must wait for the call stack to clear and its turn in the task queue. It's "as soon as possible, but not immediately."
Q: What happens to the event loop when an error is thrown in an async callback?
A: Uncaught errors in macrotask callbacks bubble to the global error event (or crash). Uncaught Promise rejections trigger unhandledrejection. The event loop itself continues running — the error only affects the current task.
