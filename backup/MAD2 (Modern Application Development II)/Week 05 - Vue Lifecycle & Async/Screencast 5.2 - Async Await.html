Async / Await

Modern JavaScript's syntax for writing asynchronous code that reads like synchronous code. Built on Promises, async/await eliminates callback hell and makes error handling straightforward with try/catch.

1 THE async KEYWORD

Making a Function Return a Promise

Adding the async keyword before a function declaration makes it always return a Promise. If the function returns a value, that value is wrapped in a resolved promise automatically.

// Without async — returns a string directly
function greet(name) {
  return `Hello, ${name}!`
}
console.log(greet('Alice'))  // "Hello, Alice!"

// With async — always returns a Promise
async function asyncGreet(name) {
  return `Hello, ${name}!`
}
console.log(asyncGreet('Alice'))  // Promise {<fulfilled>: "Hello, Alice!"}
asyncGreet('Alice').then(console.log)  // "Hello, Alice!"
Real-World Analogy
An async function is like ordering a coffee at a busy café. The barista gives you a receipt (a Promise) immediately. That receipt guarantees you'll get coffee eventually, but you don't stand there waiting — you can chat with friends while they make it. When the coffee's ready, the receipt is "redeemed" for the actual coffee.
// Async arrow function
const fetchData = async () => {
  return { id: 1, name: 'Item' }
}

// Async as object method
const api = {
  async get(url) {
    return fetch(url)
  }
}
Key Insight
The async keyword doesn't make the function asynchronous internally — it enables the use of await inside it and wraps the return value in a Promise. The function body still runs synchronously until it hits an await.
2 THE await KEYWORD

Pausing Execution

await can only be used inside an async function. It pauses the execution of the function until the Promise resolves, then returns the resolved value. The rest of the program continues running — only the async function is "paused."

async function getUser(id) {
  console.log('Fetching user...')
  const response = await fetch(`/api/users/${id}`)
  // Execution pauses here until fetch resolves
  const user = await response.json()
  console.log('User fetched:', user.name)
  return user
}

// Output:
// "Fetching user..."  (immediately)
// (pause — network request in flight)
// "User fetched: Alice"  (when resolved)

3 ERROR HANDLING WITH TRY/CATCH

Catching Failures Gracefully

The try/catch block catches any error that occurs inside the try block — including rejected Promises. This is much cleaner than chaining .catch() on every Promise.

async function getUserSafely(id) {
  try {
    const res = await fetch(`/api/users/${id}`)
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}: ${res.statusText}`)
    }
    return await res.json()
  } catch (err) {
    console.error('Failed to fetch user:', err.message)
    return { id, name: 'Unknown' }  // fallback
  }
}

// .then() equivalent — harder to read with more nesting
function getUserWithThen(id) {
  return fetch(`/api/users/${id}`)
    .then(res => {
      if (!res.ok) throw new Error(`HTTP ${res.status}`)
      return res.json()
    })
    .catch(err => {
      console.error(err.message)
      return { id, name: 'Unknown' }
    })
}
Real-World Analogy
try/catch is like planning a picnic: you bring a blanket and food (the try block), but you also pack an umbrella just in case it rains (the catch block). If it's sunny, the umbrella stays in the bag. If it rains, you grab the umbrella and the picnic goes on. Without it, rain would ruin everything and you'd have no backup plan.
Common Mistake
Forgetting to await inside try/catch — if you forget await on a Promise inside try, the function won't throw synchronously. The Promise rejection becomes an unhandled rejection, and your catch block never fires. Always await Promises you expect to reject.
4 SEQUENTIAL VS PARALLEL

Order Matters

Using multiple await expressions sequentially waits for each Promise to resolve before starting the next. Use Promise.all when the operations are independent and can run in parallel.

Sequential (Slow)

async function loadDashboardSequential() {
  // Each fetch waits for the previous one — 300ms + 300ms + 300ms = 900ms
  const users = await fetch('/api/users').then(r => r.json())
  const posts = await fetch('/api/posts').then(r => r.json())
  const comments = await fetch('/api/comments').then(r => r.json())
  return { users, posts, comments }
}

Parallel (Fast)

async function loadDashboardParallel() {
  // All three fetch at the same time — max(300, 300, 300) = 300ms
  const [users, posts, comments] = await Promise.all([
    fetch('/api/users').then(r => r.json()),
    fetch('/api/posts').then(r => r.json()),
    fetch('/api/comments').then(r => r.json())
  ])
  return { users, posts, comments }
}
Performance Tip
Use Promise.allSettled when you need all results even if some fail — it never rejects, returning an array of {status, value/reason} objects. Use Promise.race when you only need the fastest result (e.g., timeout logic).

Promise.allSettled Example

async function loadWithPartialFailure() {
  const results = await Promise.allSettled([
    fetch('/api/users'),
    fetch('/api/invalid-endpoint'),
    fetch('/api/posts')
  ])
  results.forEach((r, i) => {
    if (r.status === 'fulfilled') {
      console.log(`Request ${i} succeeded`)
    } else {
      console.log(`Request ${i} failed:`, r.reason)
    }
  })
}
5 COMPARISON: THEN VS ASYNC/AWAIT
.then() Chains
async/await
Linear, top-to-bottom reading. Single try/catch for all errors. Easy to mix with sync logic. Debugging works naturally (step over works).
// .then() — multiple sequential fetches get nested
fetch('/api/user')
  .then(res => res.json())
  .then(user => {
    return fetch(`/api/orders/${user.id}`)
  })
  .then(res => res.json())
  .then(orders => console.log(orders))
  .catch(err => console.error(err))

// async/await — linear and clear
async function getUserOrders() {
  try {
    const user = await fetch('/api/user').then(r => r.json())
    const orders = await fetch(`/api/orders/${user.id}`).then(r => r.json())
    console.log(orders)
  } catch (err) {
    console.error(err)
  }
}
6 COMMON MISTAKES
Mistake 1: await in a loop
Using await inside a for or forEach loop makes it sequential when it should be parallel. forEach doesn't respect await at all — it fires all Promises concurrently without waiting.
// ❌ BAD — forEach doesn't await
async function badLoop(ids) {
  ids.forEach(async id => {
    const data = await fetch(`/api/${id}`)
    console.log(data)  // fires immediately, logs undefined
  })
  console.log('Done')  // prints BEFORE any fetch completes
}

// ✅ GOOD — for...of is sequential (use when order matters)
async function goodSequentialLoop(ids) {
  for (const id of ids) {
    const data = await fetch(`/api/${id}`)
    console.log(data)  // one at a time, in order
  }
}

// ✅ BEST — Promise.all for independent parallel requests
async function bestParallel(ids) {
  const results = await Promise.all(
    ids.map(id => fetch(`/api/${id}`).then(r => r.json()))
  )
  console.log(results)  // all at once, fastest total time
}
Mistake 2: Forgetting await
Calling an async function without await gives you a Promise instead of the actual value. This is especially common in Vue lifecycle hooks and event handlers.
// ❌ BAD — forgetting await
mounted() {
  this.user = fetchUser('123')   // this.user = Promise, not the user object!
}

// ✅ GOOD — await the async function
async function mounted() {
  this.user = await fetchUser('123')  // this.user = actual user object ✅
}
7 TOP-LEVEL AWAIT

await Outside async Functions

In modern JavaScript (ES2022+), you can use await at the top level of modules without wrapping it in an async function. This works in browsers (with <script type="module">), Node.js ESM, and Deno.

// top-level-await.js — ES module
import { fetch } from 'node:http'

console.log('Loading config...')
const config = await fetch('https://api.example.com/config')
console.log('Config loaded:', config)

export default config  // modules that import this will wait
Warning
Top-level await blocks module execution — any module that imports from a module with top-level await will wait for it to resolve. Use it sparingly for initialization code (config loading, DB connections) and never in a hot-path import chain.

9 INTERVIEW Q&A
Q: What happens when you use await on a non-Promise value?
A: It works fine — await wraps non-Promise values in Promise.resolve(). await 42 becomes await Promise.resolve(42), which resolves to 42. It's a no-op for non-Promise values.
Q: How does async/await relate to the event loop?
A: When execution hits an await, the async function yields control back to the event loop. The microtask queue handles the Promise resolution, and when resolved, the function is placed back on the call stack at the await point. This is why async/await doesn't block the main thread.
Q: Can you use await inside a map() or filter()?
A: No — await inside map() returns an array of Promises, not values. Use Promise.all(array.map(async ...)) instead. Similarly, filter won't work directly — filter the resolved values afterward.
10 QUICK REVISION
async — makes a function return a Promise, enables await
await — pauses execution until Promise resolves, returns value
try/catch — catches rejected Promises and thrown errors
Promise.all — parallel execution, fails fast on any rejection
Promise.allSettled — parallel, waits for all to complete regardless
Top-level await — await in module scope, blocks importing modules
Never use — await in forEach(), forgetting await on calls
Debugging — async/await works with breakpoints, step-over works naturally
