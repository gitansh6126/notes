Control Flow & Functions

Master how JavaScript controls execution through conditionals and loops, then dive deep into the most important concept: functions — declarations, expressions, arrows, parameters, callbacks, and scoping.

1Conditional Statements

Making Decisions in Code

if / else if / else

const score = 85;

if (score >= 90) {
  console.log("A");
} else if (score >= 80) {
  console.log("B");
} else {
  console.log("C or lower");
}

Ternary Operator

const age = 20;
const status = age >= 18 ? "Adult" : "Minor";

switch Statement

const day = "Monday";
switch (day) {
  case "Saturday":
  case "Sunday":
    console.log("Weekend!");
    break;
  default:
    console.log("Weekday");
}
Best Practice
Prefer if/else for ranges and complex conditions. Use ternary for simple binary choices. Use switch for many discrete values. Always include break to avoid fall-through.
2Loops

Iteration in JavaScript

for (classic)
Index-based. for (let i = 0; i < arr.length; i++). Most control, most verbose.
for...of (ES6)
Value-based. for (const val of arr). Clean, no index. Works on any iterable.
const fruits = ["apple", "banana", "cherry"];

// for loop
for (let i = 0; i < fruits.length; i++) {
  console.log(fruits[i]);
}

// for...of (recommended for arrays)
for (const fruit of fruits) {
  console.log(fruit);
}

// for...in (for object keys, NOT arrays)
const user = { name: "Alice", age: 30 };
for (const key in user) {
  console.log(`${key}: ${user[key]}`);
}

// while / do-while
let count = 0;
while (count < 3) { console.log(count); count++; }

let n = 0;
do { console.log(n); n++; } while (n < 3);
Common Mistake
Using for...in for arrays. for...in iterates over keys (property names), not values. For arrays, keys are "0", "1", "2" (strings!). It also iterates over inherited enumerable properties. Always use for...of or .forEach() for arrays.
3Function Declarations vs Expressions

Three Ways to Define a Function

Declaration
Hoisted (usable before definition). Named. function add(a, b) { return a + b; }
Expression
NOT hoisted. Can be anonymous or named. const add = function(a, b) { return a + b; };
Arrow Function (ES6)
NOT hoisted. Shorter syntax. Lexical this binding. const add = (a, b) => a + b; If single expression, implicit return.
// Declaration — hoisted
sayHi("Alice");            // works! (hoisted)
function sayHi(name) {
  return `Hi, ${name}!`;
}

// Expression — NOT hoisted
const greet = function(name) {
  return `Hello, ${name}!`;
};

// Arrow function
const square = x => x * x;
const add = (a, b) => a + b;
Real-World Analogy
A function declaration is like a recipe in a cookbook — the entire book is pre-loaded before you start cooking. A function expression is like a chef writing a recipe on the fly — you cannot use it until written. An arrow function is like a quick shorthand recipe card — great for simple tasks but does not have its own context.
4Parameters & Arguments

Parameters, Defaults, Rest

Default Parameters (ES6)

function greet(name = "Guest", greeting = "Hello") {
  return `${greeting}, ${name}!`;
}
greet();               // "Hello, Guest!"
greet("Bob");          // "Hello, Bob!"

Rest Parameters (ES6)

function sum(...numbers) {
  return numbers.reduce((acc, n) => acc + n, 0);
}
sum(1, 2, 3);         // 6
sum(1, 2, 3, 4, 5); // 15

Parameters vs Arguments

  • Parameters — the variables listed in function definition (formal parameters)
  • Arguments — the actual values passed when calling the function
// params: a, b     arguments: 3, 7
function multiply(a, b) { return a * b; }
multiply(3, 7);  // 21
5IIFE & Callbacks

Immediately Invoked & First-Class

IIFE — Immediately Invoked Function Expression

// Classic IIFE (wrapped in parens, then invoked)
(function() {
  const msg = "Runs immediately!";
  console.log(msg);
})();

// Arrow IIFE
(const result = (const x = 5; x * x) => x)();

// Modern equivalent: block scope
{
  const msg = "Block-scoped!";
  console.log(msg);
}

Callbacks (Functions as First-Class Citizens)

// Passing a function as argument
function processUser(userId, callback) {
  const user = { id: userId, name: "Alice" };
  callback(user);
}

processUser(1, function(user) {
  console.log(`User: ${user.name}`);
});

// With arrow function
processUser(2, user => console.log(user.name));
Industry Insight
The callback pattern was the first async pattern in JavaScript. However, deeply nested callbacks ("callback hell") led to Promises (ES6) and async/await (ES2017). In 2024, callbacks are still widely used in event handlers (.addEventListener) and Node.js APIs (fs.readFile), but most modern async code uses async/await. Understanding callbacks is essential because Promises and async/await are built on top of them.
6Scoping

Global, Function, Block Scope

Scope determines where variables are accessible. Functions create scope, and blocks ({}) create scope for let/const.
const globalVar = "global";

function outer() {
  const outerVar = "outer";

  function inner() {
    const innerVar = "inner";
    console.log(globalVar);  // "global"
    console.log(outerVar);   // "outer"
  }

  // console.log(innerVar); // ReferenceError!
}
Best Practice
Keep functions small and focused. Avoid deeply nested scopes. Use const by default. Prefer pure functions (same input always produces same output, no side effects).

8Interview Q&A
Q: What is the difference between a function declaration and a function expression?
A: A function declaration is hoisted — usable before its definition. A function expression is not hoisted — it behaves like a variable assignment. Declarations must have a name; expressions can be anonymous.
Q: What is an IIFE and why was it used?
A: IIFE (Immediately Invoked Function Expression) runs as soon as it is defined. Before ES6 modules and let/const block scoping, IIFEs were the primary way to create private scopes and avoid polluting the global namespace.
Q: What is "callback hell" and how do you avoid it?
A: Callback hell is deeply nested callbacks that become unreadable. Solutions: break into named functions, use Promises with .then() chains, or use async/await for sequential async operations.
Q: How do arrow functions differ from regular functions?
A: Arrow functions: (1) shorter syntax, (2) implicit return for single-expression bodies, (3) no this binding (lexical this from surrounding scope), (4) cannot be used as constructors, (5) no arguments object.
