JavaScript Origins & Overview

A deep dive into the history of JavaScript, its standardization through ECMAScript, the engines that power it, and the process that shapes its future.

1The Birth of JavaScript

From 10 Days to a World Language

In May 1995, Brendan Eich was hired by Netscape to embed the Scheme programming language into their browser. Instead, he created a new language in just 10 days — initially called Mocha, then LiveScript, and finally JavaScript.

Timeline

  • 1995 — Brendan Eich creates Mocha/LiveScript at Netscape
  • 1996 — Netscape ships JavaScript in Netscape Navigator 2.0
  • 1996 — Microsoft reverse-engineers JS as JScript for IE3
  • 1997 — ECMA International standardizes as ECMAScript (ES1)
  • 1998 — ES2, minor updates for ISO conformance
  • 1999 — ES3: regex, try/catch, do-while, switch improvements
  • 2009 — ES5: strict mode, JSON, Array methods (forEach, map, filter)
  • 2015 — ES6/ES2015: let/const, arrow functions, classes, modules, promises, template literals, destructuring
  • 2016+ — Yearly release cycle: ES2016, ES2017 (async/await), ES2018, ES2019, ES2020 (optional chaining, nullish coalescing), ES2021, ES2022, ES2023 — incremental improvements every year
2JavaScript Engines

The Engines That Run the Web

A JavaScript engine is a program that executes JavaScript code. Modern engines use JIT (Just-In-Time) compilation to achieve near-native performance.
Engine Browser Written In Other Runtimes
V8 Chrome, Edge, Opera C++ Node.js, Deno, Electron
SpiderMonkey Firefox C++, Rust GNOME Shell
JavaScriptCore Safari C++ Bun, React Native

How a JS Engine Works

  1. Parsing — source code → Abstract Syntax Tree (AST)
  2. Interpretation — baseline compiler produces bytecode quickly
  3. Profiling — monitor hot functions (frequently executed)
  4. Optimization — JIT recompiles hot paths to optimized machine code
  5. Deoptimization — if assumptions fail, fall back to interpreted bytecode
Real-World Analogy
A JS engine is like a simultaneous translator at the UN. The interpreter translates on the fly (slow but immediately usable). The JIT compiler is like the translator who learns the speaker's patterns and pre-writes the translation for common phrases — much faster, but needs to backtrack if the speaker says something unexpected.
3The TC39 Process

How JavaScript Gets New Features

TC39 (Technical Committee 39) is the committee that evolves the ECMAScript specification. It includes representatives from all major browser vendors, companies, and the community.

The 5-stage Proposal Process

  • Stage 0 — Strawman: Any idea submitted by a TC39 member. No requirements.
  • Stage 1 — Proposal: A champion identified, problem described, API shape sketched. Polyfills and demos expected.
  • Stage 2 — Draft: Formal spec text. Committee has committed to reviewing and eventually including (if it progresses).
  • Stage 3 — Candidate: Spec complete, implementations needed. Browser vendors begin experimental implementation.
  • Stage 4 — Finished: At least two independent implementations pass test suites. Feature is included in the next ECMAScript release.
Industry Insight
The TC39 process is one of the most open and transparent language specification processes. Anyone can follow proposals on GitHub (tc39/proposals). Features like optional chaining (obj?.prop) and nullish coalescing (a ?? b) went from Stage 0 to Stage 4 over ~2 years. Records & Tuples (immutable data structures) are currently at Stage 2 as of 2024.
4Strict Mode

"use strict" — Cleaner, Safer JS

Introduced in ES5, strict mode opts into a restricted variant of JavaScript that eliminates silent errors and prevents dangerous patterns. 
// Global strict mode (file-level)
"use strict";
console.log("This file runs in strict mode");

// Function-level strict mode
function strictFn() {
  "use strict";
  // only this function is strict
}

// What changes in strict mode:
x = 5;           // ❌ ReferenceError (undeclared var)
delete Array;     // ❌ TypeError (can't delete non-configurable)
arguments.callee; // ❌ TypeError (deprecated)
Best Practice
Always use strict mode in modern JavaScript. ES6 modules are strict by default — no need for explicit "use strict" when using modules.
5Browser vs Node.js APIs

Two Runtimes, Two API Sets

Browser APIs
DOM (document, element, events), window, fetch, localStorage, console, setTimeout, Canvas, WebSockets, Web Workers, History API, Geolocation
Node.js APIs
fs, path, http, process, os, crypto, stream, buffer, child_process, cluster, readline

Shared APIs (Both Environments)

  • console — log, error, warn, table, time
  • setTimeout / setInterval — timer functions
  • fetch — HTTP requests (Node 18+)
  • TextEncoder / TextDecoder — encoding utilities
  • URL / URLSearchParams — URL handling
Common Mistake
Using browser APIs in Node.js. document, window, localStorage are not available in Node.js. Similarly, require('fs') does not exist in the browser. Always check the runtime environment before using platform-specific APIs.
Quick Tip
To check if code is running in a browser: typeof window !== "undefined". For Node.js: typeof process !== "undefined" && process.release.name === "node".

7Interview Q&A
Q: Why is JavaScript called JavaScript if it's not related to Java?
A: Marketing. Netscape wanted to capitalize on Java's popularity. The language was originally named Mocha, then LiveScript. Just before release, they rebranded to JavaScript. The official standard name is ECMAScript.
Q: What is the difference between V8 and Node.js?
A: V8 is the JavaScript engine that compiles and executes JS code. Node.js is a runtime built on top of V8 that adds APIs like fs, http, and path. Think of V8 as the car engine and Node.js as the entire car.
Q: What is the "TC39" and what does it do?
A: TC39 (Technical Committee 39) is the committee under ECMA International that maintains and evolves the ECMAScript specification. They review proposals, provide feedback, and vote on feature inclusion through a 5-stage process (Stage 0 → Stage 4).
Q: What does JIT compilation mean for JavaScript?
A: JIT (Just-In-Time) compilation means the engine compiles hot functions to native machine code at runtime, achieving near-native performance. This is why JavaScript can be 50-100x faster today than in 1995, enabling complex applications like Figma, VS Code, and Google Docs.
